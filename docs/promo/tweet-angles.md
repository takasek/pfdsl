# 告知ツイート 切り口集

同じ記事への導線を、異なる痛点から何度でも試みるためのストック。
1回で全部使わず、反応を見ながら数日おきに別の切り口を投下する運用を想定。
文面はそのまま使える長さ（全角140字目安）に収めてある。添付欄の GIF は gif-storyboard.md の1本、または該当シーンの切り出し。

トーン方針: 煽りの疑問形・大げさな形容は使わず、事実と仕組みを平叙文で述べる。驚かせるのは文面ではなく添付のデモ。

## 1. 王道デモ（初回告知向け）

```
計画をテキストで書く DSL を作っている。
8行書くと工程図がレンダリングされ、依存の循環や出力のないプロセスは検査で見つかり、「今着手できる作業」は状態から計算される。
（記事URL）
```

添付: フル GIF。
ねらい: 機能を正面から。初回はこれ。

## 2. AI のプランは書き捨て問題

```
AI に立てさせた計画が方向転換で書き捨てになるのは、計画が「検証できない文章」だからという面が大きいと思う。
pfdsl は計画を依存グラフとして持つ。書き換えたら check が矛盾を検出し、ready が着手可能な作業を計算し直す。
```

添付: シーン3（check がエラーを出す瞬間）。
ねらい: エージェント開発層。

## 3. 計画も lint できる

```
pfdsl check は「出力を持たないプロセス」や「依存の循環」を検出する。
コードの lint と同じことを計画に対してやる、という位置づけ。
工程図のレビューで人が目視で拾っていた類のミスは、機械的に拾える。
```

添付: check のエラー出力スクショ（V003 / V010）。
ねらい: 「計画の lint」という概念の提示。

## 4. TODO リストは依存関係を失う

```
TODO リストは並び順しか持たないので、「どれが何をブロックしているか」を保持できない。
依存関係をグラフとして書いておけば、着手可能な作業の列挙も、ブロッカーを外す次の一手の推薦も計算でできる。
pfdsl ready --best はそれをやる。
```

添付: ready --best の出力スクショ。
ねらい: タスク管理ツール不満層。開発者以外にも届く切り口。

## 5. 図は作った瞬間から腐る

```
作図ツールで描いた工程図は、現実が変わっても更新されず乖離していきがち。
pfdsl は計画がテキストなので、直せば図が再生成される。
git に履歴が残るので、計画の変更を PR としてレビューすることもできる。
```

添付: シーン1-2（テキスト→図が生える）。
ねらい: Mermaid / PlantUML 好きな「テキストで図を書きたい」層。

## 6. 非開発ユースケース: 障害対応

```
本番障害対応のフローも「成果物の依存グラフ」で書ける。
トリアージ記録→応急対応→ポストモーテム→runbook 更新、という知見の還流はフィードバックエッジ >>? で表現する。
pfdsl のリポジトリに実例を置いてある。
```

添付: incident-response.pfdsl のレンダリング画像。
ねらい: SRE 層。examples がそのまま弾になる。

## 7. 非開発ユースケース: なんでも

```
pfdsl は開発計画のために作ったけれど、依存関係のある成果物を積み上げる仕事なら何でも書ける。
リポジトリには採用プロセス・書籍執筆・カンファレンス運営・契約交渉・障害対応の実例を置いてある。
「次にやるべきこと」の計算はどの題材でも同じに動く。
```

添付: 非開発系 example のレンダリング画像を2枚。
ねらい: 非エンジニア・マネジメント層への越境。

## 8. エージェントに計画を渡す

```
エージェントに長期タスクを任せるときの「計画をどこに、どういう形式で持たせるか」問題。
pfdsl の計画は機械可読・機械検証可能なグラフなので、エージェントが自分で読んで着手判断でき、壊れた書き換えは check で弾ける。
人間の仕事は計画の形のレビューに寄る。
```

添付: シーン4（status-set → newly ready）。
ねらい: 切り口2の変奏。役割分担（人間=監査、エージェント=実行）を前面に。

## 9. ガントチャートは日付を約束させる

```
ガントチャートは日付を持つぶん、遅延のたびに引き直しが要る。
pfdsl は日付を書かず、依存関係だけを書く。「今着手できる作業」は成果物の状態から計算するので、引き直しという作業が発生しない。
順序の管理が目的なら、これで足りる場面は多いと思う。
```

添付: ready --best の出力スクショ。
ねらい: 見積もり・スケジュール管理疲れ層。

## 10. done の定義を計画に書く

```
pfdsl では成果物ごとに criteria（完成条件）を計画に書ける。
criteria のない成果物には check が警告を出すので、「完成条件が決まっていない成果物」は機械的に見つかる。
done の解釈違いを、計画の書き忘れとして扱えるようになる。
```

添付: W002 警告の出力スクショ。
ねらい: スクラム・チーム運営層。「Definition of Done を成果物単位で持つ」という語彙で。

## 11. 計画と進捗報告を同じファイルにする

```
pfdsl は成果物の done/wip/todo で工程図を色分けレンダリングできる。
計画と進捗が同じファイルにあるので、報告用の資料を別に作る必要がなく、資料と現実がズレる経路もない。
進捗共有はレンダリング結果を貼るだけになる。
```

添付: ステータス色分け図。
ねらい: 報告業務の多いリード・PM層。

## 12. 「先週から計画どう変わった?」に diff で答える

```
計画がテキストだと、「先週から何が変わったか」に git log と pfdsl diff で答えられる。
増減したノードやエッジが構造差分として出るし、SVG の視覚差分も出せる。
計画の変更を、コードの変更と同じ解像度で追跡できる。
```

添付: pfdsl diff の出力スクショ。
ねらい: 切り口5の変奏。マネジメント層寄りに「変更の追跡可能性」の角度で。

## 13. 手が止まったら最終成果物から逆算する

```
pfdsl の書き方として気に入っているのは、最終成果物からの逆算。
「これを作るには何が要るか」を1行ずつ書き足していくと計画のグラフになり、着手可能な端の作業が計算で出てくる。
チームで運用する前に、まず思考の整理ツールとして使える。
```

添付: シーン1-2（テキスト→図が生える）。
ねらい: 個人開発・副業・同人誌層。チームツールではなく思考ツールとしての入り口。

## 14. プロジェクトは「外から来ないもの」で止まる

```
pfdsl check --audit は、グラフから外部入力（自分たちでは作らない成果物）を列挙する。
プロジェクトが「外から来るはずのものが来ない」で止まる前に、依頼し忘れや待ち依存を計画の検査として棚卸しできる。
```

添付: check --audit の出力スクショ。
ねらい: 受託・複数部署連携の現場。waiting ステータスの話にセルフリプで継げる。

## 15. Mermaid で足りる場合、足りない場合

```
Mermaid で工程図を描く場合との違い。描いて共有するのが目的なら Mermaid で足りる。
pfdsl は図が計算の対象になる点が違う。循環の検出、着手可能な作業の列挙、完了記録後の次タスク推薦。
図の記法というより、計画のデータ構造を書いている。
```

添付: フル GIF。
ねらい: テキスト作図に既に慣れた層の乗り換え。固有名詞入りで検索流入も見込める。

## 16. レイアウト調整という作業をなくす

```
作図ツールで工程図を描くと、内容よりノードの整列や矢印の引き回しに時間を取られがち。
pfdsl はテキストで依存関係だけを書き、配置は Graphviz が計算する。
図の見た目をメンテナンスするという作業自体がなくなる。
```

添付: シーン2（8行→図の物量感）。
ねらい: 作図あるあるの共感ネタ。軽めの合間用。

## 運用メモ

- 各ツイートの末尾に記事 URL を付ける（引用が伸びたときの導線）。
- 反応が付いた切り口はセルフリプライで具体例を1段深掘りする（例: 切り口3が伸びたら --audit の話を継ぐ）。
- 切り口2・8はエージェント系の話題が盛り上がっているタイミングに合わせて投下すると効率がいい。
- 切り口9・15のような対比系は反論リプが付きやすいので、リプで補足対応できる日に投下する。
- 大分類は「デモ系（1, 5, 11, 16）」「計算系（3, 4, 9, 12, 14）」「AI系（2, 8）」「越境系（6, 7, 10, 13, 15）」。同系統を連投せず、系統をローテーションする。

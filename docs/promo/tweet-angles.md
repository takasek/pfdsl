# 告知ツイート 切り口集

同じ記事への導線を、異なる痛点から何度でも着火を試みるためのストック。
1回で全部使わず、反応を見ながら数日おきに別の切り口を投下する運用を想定。
文面はそのまま使える長さ（全角140字目安）に収めてある。添付欄の GIF は gif-storyboard.md の1本、または該当シーンの切り出し。

## 1. 王道デモ（初回告知向け）

```
テキスト8行で工程図が生えて、計画を lint できて、「次にやるべき作業」まで計算で返ってくる。
計画を絵じゃなくコードとして書く DSL、pfdsl を作っている。
（記事URL）
```

添付: フル GIF。
ねらい: 機能の面白さで正面から。初回はこれ。

## 2. AI のプランは書き捨て問題

```
AI に立てさせたプラン、方向転換した瞬間に破綻して書き捨てになりがち。
あれはプランが「検証できない文章」だから。
pfdsl は計画を検査可能なグラフとして持つので、書き換えたら check が矛盾を検出し、ready が次の一手を計算し直す。変更に耐えるプランの置き場所になる。
```

添付: シーン3（check がエラーを出す瞬間）。
ねらい: エージェント開発層。いま一番燃えやすい薪。

## 3. 計画も lint できる

```
コードは lint するのに、計画は lint しないの不思議じゃない?
pfdsl は「何も生み出していないプロセス」「循環した依存」を機械が指摘してくれる。
ホワイトボードの工程図だとレビュアーの目力頼みだったやつ。
```

添付: check のエラー出力スクショ（V003 / V010）。
ねらい: 「計画のバグ」という言葉の引っかかりで RT を狙う。

## 4. TODO リストは依存関係を失う

```
TODO リストの並び順って「なぜこの順か」の情報を全部捨ててるんだよね。
依存関係をグラフで書けば、「今着手可能な作業」も「最下流のブロッカーを外す次の一手」も計算で出せる。
pfdsl ready --best の出力がこれ。
```

添付: ready --best の出力スクショ。
ねらい: タスク管理ツール不満層。開発者以外にも届く切り口。

## 5. 図は作った瞬間から腐る

```
draw.io で描いた工程図、最後に更新したのいつですか。
pfdsl は計画がテキストなので、直せば図が変わるし、git に乗るので計画変更が PR レビューの対象になる。
「図と現実の乖離」が構造的に起きない。
```

添付: シーン1-2（テキスト→図が生える）。
ねらい: Mermaid / PlantUML 好きな「テキストで図を書きたい」層。

## 6. 非開発ユースケース: 障害対応

```
本番障害対応も「成果物の依存グラフ」で書ける。
トリアージ記録→応急対応→ポストモーテム→runbook 更新、って知見を次回対応に還流するループまでフィードバックエッジで表現できる。
pfdsl のリポジトリに実例が入ってる。
```

添付: incident-response.pfdsl のレンダリング画像。
ねらい: SRE 層。examples がそのまま弾になる。

## 7. 非開発ユースケース: なんでも

```
pfdsl、開発計画のツールとして作ったけど「依存関係のある成果物を積み上げる仕事」なら何でも書ける。
リポジトリの実例: 採用プロセス、書籍執筆、カンファレンス運営、契約交渉、障害対応。
「次にやるべきこと」の計算はどの題材でも動く。
```

添付: 非開発系 example のレンダリング画像を2枚。
ねらい: 非エンジニア・マネジメント層への越境。

## 8. エージェントに計画を渡す

```
エージェントに長期タスクを任せるとき、一番困るのは「計画をどこにどう持たせるか」。
pfdsl の計画は機械可読・機械検証可能なグラフなので、エージェントが自分で読んで着手判断して、壊れた書き換えは check が弾く。
人間は計画の形をレビューするだけでいい。
```

添付: シーン4（status-set → newly ready）。
ねらい: 切り口2の変奏。役割分担（人間=監査、エージェント=実行）を前面に。

## 9. ガントチャートは日付を約束させる

```
ガントチャートは「いつやるか」を約束させるけど、本当に欲しいのは「どの順でやるべきか」じゃない?
pfdsl は日付を書かない。依存関係だけ書く。
すると「今着手できる作業」は状態から計算できるので、遅延のたびに日付を引き直す作業が存在しなくなる。
```

添付: ready --best の出力スクショ。
ねらい: 見積もり・スケジュール管理疲れ層。対比系なので反論リプも呼びやすい＝伸びやすい。

## 10. done の定義、決めてますか

```
「それ、doneの定義は?」という議論、pfdsl では成果物ごとに criteria（完成条件）として計画に書いておける。
しかも criteria のない成果物には check が警告を出す。
「完成条件を決めていない成果物」が機械的に炙り出されるの、地味に効く。
```

添付: W002 警告の出力スクショ。
ねらい: スクラム・チーム運営層。「Definition of Done を成果物単位で持つ」という語彙で刺す。

## 11. 進捗報告が1コマンドになる

```
「進捗どうですか」への返事が1コマンドになる。
pfdsl は成果物の done/wip/todo で図を色分けレンダリングできるので、貼るだけで報告が終わる。
報告資料を別に作るから資料と現実がズレる。計画と報告を同じファイルにすればズレようがない。
```

添付: ステータス色分け図。
ねらい: 報告業務に疲れているリード・PM層。

## 12. 「先週から計画どう変わった?」に diff で答える

```
計画がテキストだと、「先週から何が変わったか」に git log と pfdsl diff で答えられる。
増えたノード・消えたエッジが構造差分で出るし、SVG の視覚差分も出せる。
計画変更が「なんとなく変わってた」じゃなく、レビュー可能な出来事になる。
```

添付: pfdsl diff の出力スクショ。
ねらい: 切り口5の変奏。マネジメント層寄りに「説明責任」の角度で。

## 13. 手が止まったら最終成果物から逆算する

```
やることが多すぎて手が止まったときは、最終成果物から逆算するといい。
「これを作るには何が要る?」を pfdsl で1行ずつ書き足すと、いつの間にか計画のグラフができていて、着手可能な端っこの作業まで計算で出てくる。
思考の整理ツールとしても優秀。
```

添付: シーン1-2（テキスト→図が生える）。
ねらい: 個人開発・副業・同人誌層。チームツールではなく思考ツールとしての入り口。

## 14. プロジェクトは「外から来ないもの」で止まる

```
プロジェクトが止まる原因の大半は「外から来るはずのものが来ない」。
pfdsl check --audit は、グラフから外部入力（自分たちでは作らない成果物）を自動で列挙する。
依頼し忘れ・待ち依存の棚卸しが、計画の検査コマンドでできる。
```

添付: check --audit の出力スクショ。
ねらい: 受託・複数部署連携の現場。waiting ステータスの話にセルフリプで継げる。

## 15. Mermaid で足りる人、足りない人

```
Mermaid で工程図を描いている人へ。描いて共有するだけなら Mermaid で足りる。
pfdsl が違うのは図が計算の対象になること。循環の検出、着手可能な作業の列挙、完了記録したら「次はこれ」の推薦。
図法じゃなくて、計画のデータ構造。
```

添付: フル GIF。
ねらい: テキスト作図に既に慣れた層の乗り換え。検索流入も見込める固有名詞入り。

## 16. 矢印の引き回しに人生を使わない

```
作図ツールで工程図を描く時間の半分、矢印の引き回しとノードの整列に消えてない?
pfdsl はテキストを書けばレイアウトは Graphviz が全部やる。
人間が決めるのは依存関係だけ。「見た目の調整」という仕事そのものが消える。
```

添付: シーン2（8行→図の物量感）。
ねらい: 作図あるあるの共感ネタ。軽くて RT されやすい、話題の合間の埋め弾。

## 運用メモ

- 各ツイートの末尾に記事 URL を付ける（引用が伸びたときの導線）。
- 反応が付いた切り口はセルフリプライで具体例を1段深掘りする（例: 切り口3が伸びたら --audit の話を継ぐ）。
- 切り口2・8はエージェント系の話題が盛り上がっているタイミングに合わせて投下すると効率がいい。
- 切り口9・15のような対比系は反論リプを呼びやすい（＝インプレッションが伸びやすい）ぶん、リプで補足対応できる日に投下する。
- 大分類は「デモ系（1, 5, 11, 16）」「計算系（3, 4, 9, 12, 14）」「AI系（2, 8）」「越境系（6, 7, 10, 13, 15）」。同系統を連投せず、系統をローテーションする。

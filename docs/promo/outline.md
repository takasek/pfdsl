# pfdsl 外部向け宣伝文書 構成案

本構成案は、リポジトリ内の既存の宣伝文句を参照せず、ツールを予備知識なしで実際に操作した体験のみに基づいて作成した。
体験した操作: `.pfdsl` ファイルの新規記述、`check --audit --summary`、`ready --best`、`status-set`、`graph --format svg`、`diff`、`fmt`、`explain`、壊れた図に対する検証エラーの確認、samples/examples の閲覧。

## 想定読者と中心メッセージ

- 想定読者は、複数の成果物と工程が絡む仕事の計画・進行を管理する人。
  ソフトウェア開発者を第一想定としつつ、examples が示す通り障害対応・採用・執筆など非開発業務にも届く。
  もう一つの重要読者は、AI エージェントに仕事を任せたい人。
- 中心メッセージ候補: 「計画を『絵』ではなく『コード』として持つ。だから検査でき、diff でき、次の一手を計算できる」。

## 文書構成（8 セクション）

### 1. フック: 計画は作った瞬間から腐る

作図ツールの図は更新されず現実と乖離する。
タスクリストは並び順しか持たず、依存関係と「なぜこの順か」を失う。
この 2 つの痛みを冒頭 3〜4 文で提起する。

### 2. pfdsl とは（3 行紹介 + 最小サンプル）

「入力 >> プロセス -> 出力」だけ覚えれば書き始められることを示す。
`requirements >> design -> spec` の 1 行と、そのレンダリング結果を並べる。

### 3. 体験ウォークスルー（触って得た aha の順に 4 ステップ）

1. **書く**: 数行のプレーンテキストで依存グラフができる。
2. **描く**: `pfdsl graph` で SVG/PNG/PDF を生成。図は常にテキストと同期。
3. **検査する**: `pfdsl check` が「出力のないプロセス (V003)」「循環依存 (V010)」など計画そのもののバグを検出する。
   実際のエラー出力を掲載する。図が「絵」なら起こり得ない体験なので、ここが最初の山場。
4. **回す**: status (todo/wip/done/waiting/suspended) を付けると図が生きた進捗盤になる。
   `ready --best` が「今着手可能なプロセス」を計算し、最下流のブロッカーを外す一手を推薦する。
   `status-set` で完了を記録すると newly ready が即座に返る。ここが第二の山場。

### 4. Plan as Code

プレーンテキストなので git で履歴・ブランチ・PR レビューに乗る。
`pfdsl diff` の構造差分、`fmt`/`reindex` による正規形、stdin・JSON 出力・終了コードによる CI/スクリプト連携を挙げる。

### 5. 実プロジェクトへのスケール

label/owner/criteria/description による成果物の定義充実、`subflow` による階層化、グループ・タグ・スタイル、CJK ラベル対応。
「1 画面に収まらなくなったら」への回答として提示する。

### 6. AI エージェント時代の計画フォーマット

機械検証可能なテキストだから、エージェントが計画を安全に読み書き・運用できる（check が壊れた編集を弾く）。
VS Code 拡張（ライブプレビュー等）と Claude Code プラグインの存在に触れる。

### 7. ユースケース例

docs/examples 由来の題材（障害対応、採用プロセス、書籍執筆、契約交渉など）を 1 行ずつ列挙し、開発専用ツールではないことを示す。

### 8. Getting Started

インストール、最初の 1 ファイル、`check` → `graph` → `ready` の 3 コマンド、リポジトリへのリンク。

## トーンと分量

- トーン: 誇張よりデモ。コマンドと実出力の引用で語る。
- 分量: README 的 1 ページ（1,000〜1,500 語相当）を想定。

# 分岐なし設計方針の明文化 (#9)

## 対象仕様バージョン

v0.0.6 → v0.0.7

## 概要

PFD は制御フローではなく成果物フローを記述する。条件分岐構文は存在せず、この設計判断を仕様に明文化する。

## 仕様変更

### §1 目的 への追記

「PFDSL は制御フロー（if/else、ループ、例外）を記述しない。条件分岐が必要に見える場合は、条件自体を成果物またはプロセスとして明示することを推奨する。」

### 新設セクション §X「条件分岐の不在」

#### PFD が扱わないもの

PFDSL は次を表現しない:

- 条件分岐（if A then B else C）
- ループ（until done）
- 例外フロー（on error goto）

#### 設計判断

**言語を単純に保つ**: 分岐構文を追加すると字句解析・意味論・可視化のいずれも複雑化する。

**明示的モデリングを促す**: 「条件が必要」と感じる場面は多くの場合、設計が不明瞭な状態を示す。PFD の問いは「どの成果物があればどのプロセスが動くか」であり、条件はその成果物の定義に帰着する。

#### 条件的な結果のモデリング指針

条件分岐に見える構造は次のいずれかで表現できる:

**1. 決定成果物として外化する**

```pfdsl
review_result >> approve -> approved_spec
review_result >> reject -> revision_request
```

`review_result` が「承認」か「差し戻し」かという判断そのものが成果物。

**2. 部分成果物（parts）で分岐構造を表す**

```yaml
artifact:
  review_outcome:
    parts: [approved_spec, revision_request]
```

**3. フィードバック入力（`>>?`）で再入力を表す**

```pfdsl
revision_request >>? impl -> code
```

改訂要求が実装プロセスへ戻る補助入力となる。

## 影響範囲

- §1 目的: 一文追記
- 新設セクション: 条件分岐の不在（設計判断と指針）
- 実装変更なし

# 改版チェーンメタデータ仕様案 (#8)

## 対象仕様バージョン

v0.0.6 → v0.0.7

## 概要

Artifact に `revises:` フィールドを追加し、バージョン系列を明示する。単一生成元制約により同一 Artifact を再生成できないため、改版は `spec_v1 → spec_v2` のような別 artifact として表現するのが PFDSL の標準パターン。現状この系列が暗黙的なため、明示フィールドで意図を機械的に利用可能にする。

## 仕様変更

### §2.3 IDメタデータ定義 への追加

```yaml
artifact:
  spec_v2:
    label: 設計書 v2
    revises: spec_v1
```

フィールド定義:
- 型: 成果物 ID（同一ファイル内）
- 個数: 1 Artifact につき 0 または 1 個
- 対象: Artifact のみ

### §15 制約 への追加（§15.9 として）

#### 15.9 revises 制約

1. **参照先存在制約**: `revises:` に指定した ID は同一ファイル内の Artifact ID でなければならない
2. **自己参照禁止**: `revises: self_id` は error
3. **線形チェーン制約**: 改版チェーンは線形でなければならない。ひとつの Artifact を複数の Artifact が `revises:` で参照する（= 分岐した改版）は error
4. **循環参照禁止**: `A revises B, B revises A` は error

### renderer 利用（§2.3 注記追加）

可視化バックエンドは `revises:` チェーンを次のように扱ってよい:
- チェーンの末尾（最新版）のみを表示し中間版を折りたたむ
- チェーン全体をレーンとして並べる

グラフの edge（`>>`/`->`）とは独立した表現層であり、`revises:` は edge を自動生成しない。

## 設計判断

### なぜ edge を自動生成しないか

`spec_v1 >> revise_spec -> spec_v2` というフロー edge と `revises:` は意味が異なる。Edge はプロセスによる生成関係、`revises:` は系列の意図を宣言するメタデータ。前者は省略できないが後者は任意付加情報。両者を混同すると単一生成元制約が曖昧になる。

### なぜ同一ファイル内限定か

multifile 意味論（#5/#6）が未確定のため、クロスファイル参照のパス解決規則を定義できない。v0.0.7 では同一ファイル内に限定し、クロスファイル対応は multifile 仕様確定後に拡張する。

## 影響範囲

- §2.3 IDメタデータ定義: `revises:` 追加
- §15 制約: §15.9 追加
- §16 エラー方針: revises 制約違反は error
- checker 実装: 参照先存在・自己参照・線形性・循環参照の検証ルール追加
- graphviz-exporter: チェーン折りたたみ / レーン表示（任意実装）

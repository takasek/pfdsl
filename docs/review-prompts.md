# PFD レビュープロンプト（A/B カタログ）

任意の PFD を問い詰めるための監査観点。品質ガイドが「書くときのルール」、本カタログは「レビュー時の問い」。各問いは実際に検出された誤りに由来する。一次情報は本ファイル — pfdsl スキル（references/ に同梱）と pfd-retro スキル（ポインタ参照）の両方から使われる。

## A. 図 vs 現実

- **エッジ実在性**: 各エッジについて「このプロセスは本当にこの入力を読むか／この出力を本当にこのプロセスが作るか」（例: README はサンプル描画物から書かれない — リンクするだけ。面接プロセスは応募者の提出物を生成しない）
- **駆動源**: プロセスを実際に起動する主入力はどれか。主入力と還流を取り違えていないか（例: issue 起票の駆動源はレビュー知見でなく対話。知見は `>>?`）
- **名前の一般化水準**: ノード名が実態より狭くないか（例: cli_tool → 実際は packages/ 全体 = toolchain）。逆に広すぎないか
- **偽の不変性**: 生成元なしの「外部入力」は本当に誰にも更新されないか（例: 仕様書は実装知見の issue 経由で改訂される — maintain プロセスが要る）
- **入力充足**: 各プロセス「この入力だけで出力を作れるか」（例: 修正開発に base_code・PoC が無い。ADR 起草に参照先の図が無い → ID 捏造が発生）

## B. 粒度・型

- **万能成果物**: 多数のプロセスに消費される単一の不定形 artifact は型分割のサイン（例: dialogue → お題・提案・判断の3型に分割して発散解消）
- **プロセス実在性（双方向）**: それは独立に起動される作業か、別活動の終端動作か（例: payoff_log 追記は対話の終端動作 — 単体プロセスでない）。逆に、委譲・差し戻し・固有の検証が実際に発生した活動は隠れた独立プロセス（例: ADR 起草は突合検証という固有入力を持つと委譲失敗で判明）
- **自動化はノードでなく description**: 既存ノードの出力・同期を機械化する仕組み（CI・スクリプト）は新プロセスでなく、所有ノードの description に記す（例: 自動 publish は publish_cli の description、close 時 flow 同期は map_deps の description）。新ノードにするのは新しい artifact 型か新依存が生まれるときだけ
- **並列主張への挑戦**: 図が並列と主張する作業群に共有される決定はないか。共有決定が往復で形成されるなら分割せず統合（ADR-0004 基準3）
- **修正案への再挑戦**: 一次修正（上流方針 artifact 化等）が正しい型か再度問う。方針の合否基準を下流作業なしで書けないなら、それは入力でなく出力
- **型違い**: 活動が artifact として、モノがプロセスとして書かれていないか（例: 「対話」は活動 → プロセスに）

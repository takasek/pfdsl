# ADR-0024: retro 実行記録と差分計測可能な起動条件

- Status: Accepted
- Date: 2026-07-03

## Context

pfd-retro の起動条件は pfd-ops プロトコル「定期監査」に4種定義されていた（設計対話が長く続いた後 / ADR が数本たまった時 / 同一 PFD に連続修正が入った時 / セッションの締め際）。

しかし「たまった」「連続」は**前回 retro がいつだったか**の記録が無いため原理的に判定不能だった。
実効的な起動経路は /pfd-cycle の自動実行のみで、
直接 pfd-ops を呼ぶ長期セッションでは気付き依存に逆戻りしていた。
これは retro 自身の D 層「休眠能力」基準（起動条件の定義漏れ）の変種で、条件は定義されているが計測可能でない。

## Decision

1. **実行記録**: pfd-retro は監査完了時、workflow companion の pfd-retro バインディングセクションに実行記録を1行追記する。
   形式: `日付 — 対象範囲 — findings 件数`。
2. **差分計測形の起動条件**: pfd-ops プロトコル「定期監査」の条件を実行記録を基準点とする差分計測形に書き換える（前回以降に新規 ADR が2本以上 / 前回以降に同一 PFD へ修正コミットが3回以上）。
   計測不能な残り2条件（設計対話の長期化・セッション締め際）は補助条件として残す。
3. **休眠能力監査の強化**: retro D 層「休眠能力」に「起動条件は計測可能か（差分計測形か）」の確認を追加する。

companion が無いリポでは記録を省略してよい（/pfd-cycle 経由の自動実行が基準点を代替する）。

## Rationale

1. **基準点なしの閾値条件は死文**: 「N 本たまったら」は起点が無ければ常に判定不能で、実質「気が向いたら」と同じ。
   1行の記録で全閾値条件が判定可能になる（`git log --since` や `docs/adr/` のファイル数比較が機械的に使える）。
2. **記録先は workflow companion が自然**: retro バインディング（カタログ所在・採用状況）が既にそこにあり、運用手続きの散文置き場という companion の役割定義とも一致する。
   専用マーカーファイルは置き場と改訂規約を新設するコストに見合わない。
3. **hook による自動化は今回見送り**: セッション終了 hook で retro を提案する案もあるが、hook 設定は配布対象外（リポ・ユーザー設定に属す）で全採用リポに効かない。
   まず配布可能なスキル文面の範囲で気付き依存を解消する。

## Consequences

- pfd-retro SKILL.md 出力節に実行記録の追記手順を追加した。
- pfd-ops プロトコル「定期監査」を差分計測形に書き換えた。
- `.pfdsl/workflow.md` と scaffold の workflow.md に「retro 実行記録」小節を追加した（scaffold 経由で採用リポにも雛形が届く）。
- 実行記録は増え続ける成果物になる。行数が実用上の限界を超えたら古い行の圧縮規約を検討する（D 層「改訂規約」の対象）。

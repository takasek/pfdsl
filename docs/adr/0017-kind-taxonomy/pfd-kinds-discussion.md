# PFD種別分類の検討メモ

## 問題意識

PFDを書いていると「これは1ファイルに収まるか？別ファイルにすべきか？」という判断が生じる。
pfd-ops スキルは `ecosystem.pfdsl` と `roadmap.pfdsl` の2類型を前提にしているが、実際にPFDを書くと「この関心はどちらにも収まらない」ケースが出てきた。2類型では不足があるため、**必要な類型を判断する知識**をスキルに持たせたい。

## 現時点で見えている種別の軸

まず「PFDの内部特性はどう記述できるか」を帰納的に探るため、4つの仮想プロジェクト（レシピ共有アプリ・AIスキル開発・車両管理システム・カンファレンス）に類型の事前情報を与えず独立してPFDを設計させ、審査エージェントが横断監査した。その結果、PFDの内部特性を記述する3軸が浮かび上がった。

**この3軸は「実際に作るファイル種別」ではなく、種別を判断するときの分析ツール。** 種別の定義は後の「類型の定義」セクションに置く。

### 軸A: 時間的構造

| カテゴリ | 実態 | 典型例 |
|---|---|---|
| 収束（多→一） | 多起点が1つの成果物に絞り込まれる | CFP選考→確定スケジュール |
| 発散（一→多） | 単一起点から複数終端に変換が広がる | レシピ投稿→検索Index＋お気に入り |
| 線形変換チェーン | 直列の多段変換 | GPS→走行ログ→日報→月次レポート |
| 定常ループ | 終端が起点に戻る繰り返し | 対話→知見→ガイド更新→スキル再生成→対話 |
| 静的依存マップ | 「誰が何を必要とするか」の地図（時間軸が主役ではない） | 成果物の生態系（ecosystem） |

**背景:** 4エージェントが独立設計した際、全員が「収束型」という語を使ったが、
指している図の形が3種類あった。

- Noodleエージェント: 「レシピ投稿 → 検索Index と お気に入り の2方向に広がる」を「収束型」と呼んだ（実態は発散）
- FleetOpsエージェント: 「GPS → 走行ログ → 日報 → 月次レポートの直列変換」を「収束型」と呼んだ（実態は線形チェーン）
- Zenithエージェント: 「多数のCFP申請 → 1つの確定スケジュールに絞り込む」を「収束型」と呼んだ（これが真の収束）

同じ語が異なる図形を指していたため、語を使い分けるための実態ベースの分類が必要と判明した。

### 軸B: 主語

| カテゴリ | 実態 |
|---|---|
| データ変換型 | データ・成果物が主語、人は黒子 |
| 人間活動型 | チーム・役割が主語、成果物は合意や判断の外化 |
| 知識蓄積型 | スキル・ガイドなど能力成果物の世代的蓄積 |

### 軸C: 寿命

| カテゴリ | 実態 |
|---|---|
| 実装フェーズ限定 | 完成したら参照が減る（roadmap的） |
| 定常運用 | 完成という概念がなく永続する |
| イベント固有 | 1回のイベントで完結し廃棄される |

## 類型の定義（草稿）

各類型は「この類型が答える問い」と「ない場合に見えなくなること」で定義する。
プロジェクト開始時にこのリストを参照し、必要な類型を選ぶ。

---

### ecosystem（成果物の生態系）

**答える問い:** この成果物は誰が使うか。消費者のない成果物が紛れ込んでいないか。

**ない場合に見えなくなること:** 誰も参照しない文書が増殖する。新しい成果物を追加するとき「本当に要るか」の判断基準がない。成果物間の依存が暗黙になり、廃止・変更の影響が読めない。

**特徴:**
- 時間軸より「依存の地図」が主役
- 消費者を書けない成果物を検出する終端監査が主な用途
- statusは通常書かない（進捗ではなく存在依存の地図が目的）

**典型的に含むもの:** ドキュメント類・スキル・仕様書・サンプル・ADRなどの知識成果物とその消費関係

---

### roadmap（実装ロードマップ）

**答える問い:** 何を何の前に作る必要があるか。今着手できる作業はどれか。

**ない場合に見えなくなること:** 依存を無視した着手が起きる。並列化できる作業と待つべき作業の区別がつかない。「何が終われば次に進めるか」が頭の中にしかない。

**特徴:**
- 時間軸あり・収束型（すべてのartifactがdoneに向かって進む）
- artifactのstatusによる進捗可視化が主な用途
- 完成したら参照が減る（寿命が有限）

**典型的に含むもの:** 実装タスク・マイルストーン・依存関係にある成果物群

---

### workflow（定常ワークフロー）

**答える問い:** この作業はどう繰り返されるか。誰が何をトリガーに何を行うか。

**ない場合に見えなくなること:** プロセスが属人化し再起動不能になる。「なんとなくやっている」定常作業の全体像が見えない。新メンバーがどこから入ればいいかわからない。

**特徴:**
- 時間軸あり・循環型（終わりがなく繰り返す）
- 人・システムどちらが実行するかは自動化レベルによって変わるため区別しない
- statusは通常書かない（サイクルに「完了」の概念がない）

**典型的に含むもの:** スプリントサイクル・レビュー→マージ→デプロイのループ・改善サイクル・月次運用サイクル

---

### runtime-pipeline（実行時データフロー）

**答える問い:** システムが動くとき、データは何に変換されるか。変換の境界はどこか。

**ない場合に見えなくなること:** どの変換ステップで情報が落ちるか・加工されるかが不明。機能追加時の変更影響範囲が読めない。障害がどこまで波及するかの依存関係が見えない。

**特徴:**
- 時間軸あり（データの流れる方向に時間が進む）
- 主語はデータ・成果物。人・チームは黒子
- workflowと似ているが「何を回すか」ではなく「何が変換されるか」が主役

**典型的に含むもの:** 受信→処理→保存→配信のチェーン・バッチ処理・ETLパイプライン

---

## 類型の選び方（問診リスト草稿）

プロジェクト開始時に以下を確認する。Yesなら対応する類型を作る。

| 問い | 類型 |
|---|---|
| 知識・能力成果物（ドキュメント・スキル・仕様書）が複数あり、誰が使うか管理したいか？ | ecosystem |
| 実装すべき作業に依存関係があり、着手順を管理したいか？ | roadmap |
| 定常的に繰り返す作業サイクルがあるか？ | workflow |
| システムがデータを受け取り変換して出力するパイプラインがあるか？ | runtime-pipeline |

複数Yesなら複数ファイルを作る。全部Noなら1ファイルで足りる。

**workflow か runtime-pipeline か迷ったら:**
人・チームの判断やトリガーが主役 → workflow / データの変換経路が主役 → runtime-pipeline。同一ドメインに両方存在してよい（例: 週次再学習サイクル=workflow、ログ→モデル変換=runtime-pipeline）。

---

## 先行概念との対応

- **Kruchten 4+1**: 同一システムを関心ごとに別図で表す。「何を分けるか」ではなく「誰の何の問いに答えるか」が分割基準
- **IDEF0 vs IDEF3**: IDEF0=関数変換（時間軸なし）、IDEF3=プロセスフロー（時間軸あり）。PFDSLの構文はIDEF0に近いが、roadmapのようなIDEF3的用途にも使われている
- **Zachman列軸**: What（データ構造）/ How（機能変換）/ When（順序）/ Who（組織）/ Why（動機）。PFD種別は「列違い」として解釈できる

## 廃止した類型

### ~~stakeholder-handoff~~（廃止）

**廃止理由:** 答えようとしていた問い「誰から誰へ何が渡るか」を分解すると、何が渡るか=artifact、誰から誰へ=ownerフィールド、いつ=エッジの順序、という既存概念の組み合わせで表現できる。種別が独自に持つべき「他のどの種別でも答えられない問い」が存在しなかった。また、「境界の種類」で種別を切ろうとしており、「問いの単位」で切る他4種別の設計原則と異質だった。

**検証:** FleetOpsコンプライアンス提出・Noodle API合意・PMDA薬事申請の3ケースをsubagentで4種別に再設計し、表現しきれなかった問いがゼロであることを確認した（→ `tmp/pfd-kinds-discussion.md` 内の議論ログ参照）。

## 設計上の決定事項

**種別分類の対象スコープ**
この種別分類は `.pfdsl/` ディレクトリで**プロジェクトとして永続管理するPFD**を対象とする。イベント当日の手順書・ランブックに同梱する一回性のPFD（例: カンファレンス当日オペレーション）は記法として妥当だが、種別分類の管理対象外。そのような「イベント固有・使い捨て」のPFDを4種別のどれかに無理に収める必要はない。

**ファイル vs group の使い分け**
類型がファイルに対応する。同一類型内の細分はgroupで行う。「類型が違えばファイルを分ける、同一類型内の整理はgroup」。

**statusの使い方**
roadmapではartifactの進捗可視化に使う（todo→wip→done）。それ以外の類型ではstatusを書かないか、類型の目的に合った意味で限定的に使う。種別をまたいだstatusの統一定義は不要。

**合意プロセスの配置**
組織境界をまたぐ引き継ぎも workflow か runtime-pipeline に吸収する。組織境界は artifact / process の `owner` フィールドで表現し、種別を増やす理由にしない。合意（チーム間API合意・外部提出・差し戻しループ）はどれも workflow の問い「この作業はどう繰り返されるか」で答えられる。

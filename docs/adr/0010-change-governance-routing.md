# ADR-0010: 変更ガバナンスの経路設計

- Status: Accepted
- Date: 2026-06-12

## Context

生態系 PFD には変更対象として仕様（`spec`）、スキル（`skill`）、
品質ガイド（`quality_guide`）などが存在する。
それぞれ変更に必要な手続きの重さが異なる。

仕様は外部消費者（ツール実装者・ユーザー）との互換性に関わる。
一方、スキルや品質ガイドはリポジトリ内部の学習ループを形成し、
速度がそのまま学習効率になる。

「どの変更経路が必要か」を規約文書に書くと運用者が暗記を要求される。
この制約をエッジ形状——すなわち入力接続の有無——として PFD に恒久化できるかを検討した。

## Decision

変更対象によって入力経路を分ける。

- **仕様改訂**は `gh_issues`（GitHub Issues）を唯一の入力とする。
  経路: `proposals >> issue_management -> gh_issues >> maintain_spec -> spec`。
  `decisions` から `maintain_spec` への直結エッジは存在しない。

- **スキル・品質ガイドの改善**は `decisions`（対話の判断）から直結する。
  経路: `decisions >> improve_skill -> skill`。
  issue 管理ゲートを挟まない。

この接続パターンを図に固定することで、変更経路は `graph` コマンドの出力から
機械的に読み取れる状態にする。

## Rationale

1. **仕様は外部消費者と互換性に関わる**: 提案の審査・恒久記録・依存整理（issues_flow
   での起草順序管理）が必要。即時改訂は破壊的変更のリスクを高める。

2. **スキルは内部学習ループで速度が価値**: ゲートを挟むと実践→ガイド改訂→再実践の
   サイクルが鈍る。改善の機会損失がそのまま方法論の停滞になる。

3. **ガバナンスのエッジ恒久化**: 「この変更はどの手続きか」という問いに対し、
   図を参照すれば答えが出る。規約文書の暗記・最新化コストを削減できる。
   エッジが規約そのものになる。

## Consequences

- `gh_issues` を経由しない仕様改訂は、図への違反として検出対象になる。
  循環検出 lint（issue #4）が整備されれば、経路違反も同様に自動化できる。
- 変更対象を新たに追加するときの判断基準:
  「外部消費者がいるか」→ いれば issues ゲート、いなければ decisions 直結。
- `docs/issues_flow.pfdsl` の起草順序管理と接続し、
  仕様変更の優先順位付けは issues_flow 側で担う。

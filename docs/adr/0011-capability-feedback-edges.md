# ADR-0011: 能力成果物の世代還流

- Status: Accepted
- Date: 2026-06-12

## Context

スキル・ツールチェーンのような「能力」をなす成果物は、
自分自身の生産チェーンの上流プロセス（実践レビュー・設計対話）の道具としても使われる。

具体例として、`toolchain` を実践レビュー（`review_examples`）の入力に
通常エッジ（`>>`）で追加しようとした。しかしこの接続は
`review_examples` → `findings` → `discuss` → `decisions` → `improve_skill`
→ `skill` → `toolchain` という循環を Primary graph に生じさせた。

`check` コマンドは現状循環を検出しない（issue #4 で lint 化予定）ため、
循環の発生は静的には気付かれにくい。ルールとして明文化する必要があった。

一方、スキル（`skill`）は実践レビューの通常入力にできる——
スキルの生産チェーンは知見の `>>?` でしか閉じず、
通常エッジを追加しても Primary graph に循環が生じないためである。

## Decision

能力成果物が自分の生産チェーンの上流プロセスに入力されるときは、
**前世代スナップショット**として `>>?`（フィードバック入力）で表す。

通常エッジ（`>>`）が許されるのは、
その成果物の生産チェーンがフィードバックエッジ経由でしか閉じない場合のみ。

適用例:
- `toolchain >>? review_examples` — ツールチェーンは自分の生産チェーン上流に戻るため `>>?`
- `findings >>? discuss` — レビュー知見は対話へ還流するため `>>?`
- `skill >> review_examples` — スキルの生産チェーンは知見の `>>?` でしか閉じないため `>>` 可

## Rationale

1. **自己改善ループは本質的に世代をまたぐ**: ADR-0002 形態3（定常サイクル）の適用。
   能力成果物が自身の改善に使われるサイクルは「版を列挙できない繰り返し」であり、
   `>>?` が正しい表現形態になる。

2. **プロセスが使うのは現行世代の能力**: その実行が生む「次世代」とは別物である。
   `>>?` のフィードバック入力はこの世代差を型として表現する。
   通常エッジで書くと「同一インスタンスが自分を改善する」という意味になり、
   実態と乖離する。

3. **循環検出の代替基準**: checker は循環を検出しない（issue #4 で lint 化予定）ため、
   当面この規則が人間・agent 側の判断基準になる。
   「追加しようとした通常エッジが生産チェーンを閉じるか」を図上で確認する手順が定着する。

## Consequences

- 生態系 PFD では `toolchain >>? review_examples`、`findings >>? discuss` が確定する。
- 循環検出 lint（issue #4）が導入されれば、通常エッジによる循環は自動的に検出され、
   この規則は機械的に促される形になる。
- 新しい能力成果物を生産チェーン上流のプロセスに接続するときは、
  「その成果物の生産チェーンは `>>?` でしか戻らないか」を確認してからエッジの種類を決める。

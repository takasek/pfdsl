# ADR-0008: グループは存在様式で切る

- Status: Accepted
- Date: 2026-06-12

## Context

PFDSL の `group` はレイアウト補助であり、Graphviz の cluster に対応する（spec §2.8）。
意味論への影響はないが、何を軸にグループを切るかは図の情報密度を左右する。

`docs/artifact_ecosystem.pfdsl`（生態系 PFD）で「設計対話」プロセスの5出力を分類したとき、お題・提案・判断の3成果物は「対話」クラスタに、ADR・効果局面ログの2成果物は「知識記録」クラスタに落ち着いた。
「同一プロセスの出力は同じグループにまとめるべきでは」という問いが立ち、グループ分割基準を明文化する必要が生じた。

## Decision

グループの軸は**成果物の存在様式**（住処・寿命・消費局面）とし、生成元プロセスではない。

- セッション限りの対話内シグナル（status 管理対象外、実体はセッションログ）と、リポジトリに永続するファイル（status 管理対象、後のセッションが消費）は、生成元が同じでも別クラスタに置く。
- プロセスは主活動の場のクラスタに置く。

## Rationale

1. **エッジとの情報分担**: 生成関係はエッジが既に完全に表現している。グループまで生成元で切ると情報が重複し、代わりに存在様式（誰がいつどこで消費するか）の情報が図から失われる。

2. **クラスタ境界をまたぐ出力エッジの意味**: 対話クラスタから知識記録クラスタへ抜けるエッジは「外化」そのものの図示になる。公開・配布クラスタへ入るエッジは「外部への公開」の図示になる。境界越えのエッジが状態遷移を可視化する。

3. **公開・配布クラスタの妥当性**: README・npm CLI・サンプル描画物は生成プロセスがバラバラだが「外部消費者向け終端」という存在様式で同居できる。生成元基準では1クラスタに収まらない。

4. **ADR-0004 との類縁**: ADR-0004（プロセス粒度の所有権境界）と同様、グループも構造（エッジ）が表現しない直交軸を担う。エッジ＝関係、グループ＝存在域、という役割分担になる。

## Consequences

- ADR・効果局面ログは、生成元が「設計対話」プロセスであっても「知識記録」クラスタに留まる。
- 新ノード追加時のグループ判断は「どこに住むか・誰がいつ消費するか」で行い、「どのプロセスが作ったか」では行わない。

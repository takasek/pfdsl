# ADR-0018: 成果物の門番は双方向 — 手段成果物は終端たりえない

- Status: Accepted
- Date: 2026-06-20

## Context

pfd-ops プロトコル5「成果物の門番」は当初「消費者を書けない成果物は作らない」という **消費者側の片方向監査**だった。これは「出力ノードが宙に浮く（誰も使わない成果物）」を防ぐ。

しかし逆向きの欠落を捕捉できなかった。#5/#6 の仕様案起草（PR #126）では `draft_multifile_specs -> [hierarchy_spec, presets_spec, multifile_policy]` を起草し、 `integrate_multifile -> spec_v008` まで roadmap にあったが、**`spec_v008` を消費する実装プロセスが無かった**。
`spec_v008` は roadmap 上で消費者なしの終端ノードになっていた。

仕様は必ず実装される。`spec_v008` は真の終端（納品物）ではなく、実装という後続を持つ**手段成果物**。
明らかに必要な後続タスク（v0.0.8 実装ツールチェーン）がグラフから失われ、 PR を出す時点で次のノードが用意されていなかった。v0.0.7 では `spec_v007` に `implement_spec_features -> featured_toolchain (#52)` が繋がっていたのと非対称だった。

## Decision

門番を**双方向**にする。

- **(a) 消費者側**（従来）: 消費者を書けない成果物は作らない。
- **(b) 後続側**（新規）: 終端を名乗れるのは**真の納品物**（公開物・外部提出物・運用される成果物）のみ。
  **手段成果物（仕様・設計・計画・提案）は終端たりえない**。手段を出力・計画した時点で、 それを消費する後続プロセスをプレースホルダ（todo）でもグラフに登録する。
  明らかに必要な後続が欠けた終端 artifact は門番違反。

pfd-ops の終端ゲートに確認項目を追加する:
「出力 artifact が手段なら後続プロセスがグラフに在るか確認。無ければ todo プレースホルダで登録」。

## Rationale

1. **手段と納品物の区別が判定基準**: spec / design / plan / proposal は「次に使われる」ことが
   存在理由。これらが消費者を欠くのは設計の盲点であり、図から後続タスクが消える。
   対して publish 済みパッケージ・公開記事・運用ツールチェーンは真の終端で、消費者がなくてよい。

2. **プレースホルダで十分**: 後続の詳細（実装 issue の中身）は手段が確定してから書くのが忠実
   （統合が提案の盲点を露出する — ADR-0013 の対称）。だが**ノードの存在**は手段を出した時点で 確定できる。詳細未定でも todo ノードを置けば、全体経路が図に残り、後続着手者が見失わない。

3. **片方向監査の死角を埋める**: 「消費者なし成果物を作らない」は新規作成時のチェックだった。
   既存の手段成果物（roadmap に前から在る `spec_v008` 等）が後続を欠く状態は、 作成時チェックをすり抜けて滞留する。双方向化と終端ゲート項目で毎サイクル再点検する。

## Consequences

- pfd-ops プロトコル5を双方向門番に書き換え、終端ゲートに後続確認項目を追加した。
- 現状修復として `.pfdsl/roadmap.pfdsl` に `implement_multifile -> multifile_toolchain`（todo,
  入力 `[spec_v008, featured_toolchain]`）を登録し、`spec_v008` の終端違反を解消した。
  実装 issue の起票は v0.0.8 統合後（spec_v008 確定後）に行う。
- review-prompts A「偽の不変性」（生成元なし外部入力の点検）と対称の点検観点。
  終端ノードについても「これは真の納品物か、後続ノード未登録か」を問う。

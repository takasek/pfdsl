# ADR-0004: プロセス粒度の決定基準 — 構造3則＋社会1則

- Status: Accepted
- Date: 2026-06-11

## Context

プロセスをどこまで分割するかについて、本家 PFD の目安は「見積もり可能な最小単位」であり、
停止基準として曖昧である。分割しすぎると図が煩雑になり、粗すぎると依存が隠れる。

実践では「会場・スポンサー手配」「双方レビュー」のような並記名を持つプロセスが現れ、
そのたびに判断基準のなさが問題になった。

## Decision

4基準を採用する。

**(1) 上限: 時間的凝集で束ねない**
複数の論理的に独立した作業を「同時期だから」という理由で1プロセスにまとめない。
束ねた全入力が全出力をゲートする偽依存が生まれる。
並記名（「○○・△△」「双方□□」）は束ねのサイン。

**(2) 下限: 新依存が現れない分割はしない**
分割しても境界をまたぐ新しい依存関係が現れない場合、図に情報が加わらない。
割らなくてよい。

**(3) 分割不能: 相互依存は1プロセスに**
決定が往復で形成される相互依存関係は、DAG では循環として書けない。
無理に分割すると循環が暗黙依存として隠れる。
→ 1プロセスに複数出力を持たせ、共有決定を決定記録 artifact に外化する。

判定テスト: 上流方針の合否基準を下流作業なしに書けるか。
書けなければ「上流方針」は入力ではなく出力（相互依存の証拠）。

**(4) 社会軸: 所有権境界は構造基準を上書きする**
引き継ぎ点（担当者・チームの境界）は、依存基準によらず正当な分割根拠となる。
そこで artifact が契約になる。単一所有者内部では構造基準(1)(2)(3)のみ適用。

## Rationale

マイクロサービス設計と同型: artifact=API契約、暗黙依存禁止=共有DB禁止、
相互依存の無理な分割=分散モノリス、基準(4)=Conway の法則。
逆 Conway: フロー先行設計→owner 割当という順序が自然に導ける。

補論: 図の粒度と実行の粒度は分離する。AI エージェントが作業者でも
図は粗くせず、実行時に部分グラフをバッチングする。

## Consequences

- スキル品質ガイドに4ルールとして収録
- `docs/issues_flow.pfdsl` の「階層×プリセット仕様起草」が
  「並列→上流方針→共同起草」と3段階で収束した経緯が実例
  （git log の `fix(docs)` 2コミット参照）

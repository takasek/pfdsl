# ADR-0002: 改版の表現 — 単一生成元制約下の3形態

- Status: Accepted
- Date: 2026-06-11

## Context

PFD の単一生成元制約（V001: 同一 artifact を複数のプロセスが生成してはならない）により、
同一 artifact の再生成をそのまま記述することはできない。

しかし実務には改版・レビューループ・定常サイクルが不可避に存在する。
書籍執筆レビュー、契約交渉、ML モデル再学習などが典型例である。
これらを適切に表現する統一ルールが必要だった。

## Decision

以下3形態を状況に応じて使い分ける。**併用禁止**。

1. **別 artifact 形態**: 版がベースライン（承認・配布・契約の対象）になる場合
   → `spec_v1 → spec_v2` のように別 artifact として定義する

2. **収束ループ形態（`>>?`）**: 同一フェーズ内の繰り返し収束がある場合
   → `指摘 >>? 元プロセス` のように後ろ向きフィードバックで表現する

3. **定常サイクル形態（`>>?`）**: 版を列挙できない無限繰り返し（ML 再学習等）
   → 同様に `>>?` を使い、サイクルの性質をコメントで補足する

補則: `>>?` は後ろ向き（下流成果物→上流プロセス）のみ。
既に `>>` で繋いだ対への重複 `>>?` は冗長であり禁止する。

## Rationale

実践ラウンド2で実際に発生した誤り2種が動機となった。

- **誤り1（書籍執筆例）**: `>>? write` と revise プロセスの併存 — 同一現実の二重表現
- **誤り2（契約交渉例）**: 前向き方向への `>>?` の重複使用

3形態の使い分けは「版の意味が何か」を問うことで決まる:
ベースラインが必要なら artifact を分ける、収束であれば `>>?`、
無限サイクルも `>>?` だがコメントで意図を明示する。

## Consequences

- `revises:` メタデータ（issue #8 で設計中）が版チェーン畳み込み・diff 整列の足場となる
- 実例: `docs/examples/contract-negotiation.pfdsl`、`docs/examples/ml-model-dev.pfdsl`

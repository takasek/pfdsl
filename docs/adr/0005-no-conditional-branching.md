# ADR-0005: 条件分岐の不採用

- Status: Accepted
- Date: 2026-06-11

## Context

フロー記述には条件分岐（成功/失敗、合格/不合格、承認/差し戻し）が 欲しくなる場面が頻出する。テスト通過・レビュー承認・閾値判定などが典型例である。

これらを `if/else` 的な分岐構文で表現することを言語仕様に加えるかどうかが問題となった。

## Decision

条件分岐は言語仕様に導入しない。

PFD はフローチャートではなく**成果物の依存グラフ**である。
分岐が欲しくなるのは、プロセス設計（成果物定義）の誤りのシグナルと解釈する。

変換パターン:
- 「テスト合格なら→」ではなく、テスト結果報告書を常に出力し、
  不具合票（空でありうる成果物）を `>>?` で元プロセスへ還流させる
- 「承認されたら→」ではなく、承認記録を成果物として定義し、
  後続プロセスはその承認記録を入力とする（承認記録が存在しなければ実行されない）

## Rationale

分岐を導入すると以下が壊れる:

1. **DAG の静的解析性**: ランク計算・到達性解析・正準順序（トポロジカルソート）が
   実行時条件に依存するようになる
2. **状態機械との重複**: 分岐＋ループは状態機械の表現力と本質的に同じであり、
   PFD が本来持つべきでない関心事を取り込む
3. **ワークフローエンジン化**: エンジン依存の実行セマンティクスが意味論に混入する

本家 PFD も分岐を持たない。この決定は言語仕様を最小に保つという方針とも整合する。

分岐したい衝動は多くの場合、「成果物が何であるべきか」を再考することで解消できる。

## Consequences

- 言語仕様を最小に保つ（新構文追加なし）
- issue #9（仕様書への明文化）の根拠文書となる
- レビュー時の問い「分岐が欲しくなったらどの成果物が未定義か」が使えるようになる

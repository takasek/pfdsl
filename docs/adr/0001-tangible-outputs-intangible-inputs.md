# ADR-0001: 成果物の有形性 — 出力は検証可能なモノ、入力は不定形可

- Status: Accepted
- Date: 2026-06-11

## Context

PFD（Process Flow Diagram）の成果物として「理解」「合意」のような不定形を許すか否かは、
方法論の内部一貫性に関わる問題である。

一律禁止すると、レビュープロセスが実際に依存する「レビュアー知識」「ドメイン知識」を
入力として記述できなくなる。これは「暗黙依存を図に表す」という PFD の根本目的と
自己矛盾を起こす。

## Decision

非対称規則を採用する。

- **出力（生成される成果物）**: 保管・検証可能なモノのみ許可
  - 「理解」→ 理解資料（ドキュメント）に外化
  - 「合意」→ 議事録・承認記録に外化
- **入力（フロー外リソース）**: 不定形を許可
  - 「レビュアー知識」「顧客との関係性」などは不定形のまま記述可

## Rationale

不定形出力を禁じる理由は3つある。

1. **完了判定不能**: 「理解が深まった」の達成基準を客観的に定義できない
2. **status 管理不能**: done/wip/todo の判定に外部判断が必要になる
3. **引き継ぎ不能**: 後続プロセスへの確実な受け渡しができない

XDDP（Extended Derivative Development Process、清水吉男）のスペックアウトは
「理解の外化」の制度化として、この方向性と整合している。

一方、不定形入力の明示はむしろ**依存の可視化**として価値がある。
「暗黙依存禁止」の精神は、依存をゼロにすることではなく見えるようにすることにある。

**境界条件の判定テスト**: 不定形をフロー内プロセスの出力にしたくなったら、
それは文書化のサインである。補助テスト: 作業名成果物（「応急処置」）は
「保管できるモノか」で弾き、「暫定対応記録」に変換する。

## Consequences

- スキル（SKILL.md）の品質ガイドに2ルールとして収録
- `docs/examples/incident-response.pfdsl` 等の実例で「暫定対応記録」として実証済み
- 「合意形成」のような曖昧な artifact 名が現れたとき、外化を促すレビュー基準となる

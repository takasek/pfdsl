# ADR-0006: 品質担保の二層構造 — ルールで防げるミスと防げないミス

- Status: Accepted
- Date: 2026-06-11

## Context

品質ガイド（スキル）整備後に行った実践ラウンド2で、ガイドの例文に明記済みの
ミスが再発した:

- **入力充足漏れ**: 修正開発プロセスに `base_code` が入力として存在しない
  （ガイドの例文そのものに記載されていた誤り）
- **終端監査の不実施**: 消費されていない artifact の放置が2例

これらはガイドを読んでいても防げなかった。ルールのみによる品質担保の限界が
実証された形となった。

## Decision

品質担保をルールとツールの二層に分ける。

**ルール（ガイド）の担当**: 設計判断が必要なミス
- プロセス粒度の選択
- 改版形態（別 artifact / `>>?`）の選択
- 成果物命名と外化の要否
- 所有権境界での分割判断

**ツール（lint）の担当**: 機械的に検査可能な性質（issue #4 で実装予定）
- 循環依存の検出
- 終端監査（消費されていない artifact の検出）
- 入力充足の手がかり（孤立している artifact の検出）
- 孤立 parts の検出
- 冗長な feedback（`>>?` の重複）の検出

## Rationale

「列挙的・網羅的な確認」を要するミスはルール提示だけでは防げない。
これはルールの質の問題ではなく、認知的な確認作業の限界である。

逆に、実践→レビューのループは有効に機能した:
どのミスがルールで消えてどれが残るかを分離計測することで、
lint のターゲット項目とその優先度を実証的に導出できた。

「ルールは機械に任せられないものだけ担当する」という分業が、
長期的にスキルの品質ガイドを引き締めることにもなる。

## Consequences

- issue #4 の lint 項目に優先度根拠が与えられた
- スキルの点検ルールは lint 実装までの暫定的な手動代替として位置づける
- 実践ラウンドの価値が「ガイド品質向上」から「ツール要件導出」へも拡張された

# ADR-0021: 外部ツール専用フィールドは spec に入れない — pfdsl 単体価値が判定基準

- Status: Accepted
- Date: 2026-06-27

## Context

pfd-tools 連携準備として `schedule:` ブロック（process に workVolume / reworkRatio / resources / startCondition / milestone、artifact に availableTime / maxRevision）を spec に追加する案（#220）を起票し、checker・サンプル・proposal まで実装した。

しかしこのフィールド群は、値の意味づけが完全に外部ツール（pfd-tools）に存在し、pfdsl 自身のモデルには何も寄与しない。グラフ意味論（依存・生成・改版）に影響せず、可視化にも現れず、`check` の検証対象も「pfd-tools が解釈できる形式か」という外部都合の型チェックにとどまる。ブロックの**存在**が連携可否のフラグになるだけで、pfdsl 単体のユーザーには空のメタデータ置き場にしか見えない。

加えてサブフィールドが多く意味論が重い。pfdsl の spec・checker・サンプル・skill 配布物すべてに恒久的な維持コストを課す一方、pfdsl 単体の価値は増えない。

## Decision

**値の意味づけが完全に外部ツールに依存し、pfdsl 単体に価値を与えないフィールドは spec に入れない。**

採否の判定テスト: 当該フィールドが pfdsl の少なくとも1つの一次関心（**グラフ意味論 / 可視化 / `check` の本質的検証**）に寄与するか。3つすべてに寄与せず「外部ツールがそう解釈する」ことだけが存在理由なら、spec に入れない。

`schedule:` はこのテストに落ちるため不採用とする。#220 は not planned でクローズし、実装 PR・proposal・サンプルを取り下げ、roadmap から schedule チェーンを削除する。

## Rationale

1. **spec の凝集を守る**: pfdsl の spec はプロセスフロー（成果物の変換グラフ）を記述する言語の定義である。外部ツールの見積もりパラメータ置き場を兼ねると、フィールドの意味が「pfdsl のモデル」と「pfd-tools の入力形式」の二系統に分裂し、抽象度がぶれる。

2. **維持コストと単体価値の非対称**: 新フィールドは spec・checker・サンプル・skill 配布物に恒久コストを課す。そのコストは pfdsl 単体ユーザーの価値で正当化されるべきで、外部連携のためだけのフィールドはこの非対称を持ち込む。

3. **連携は別レイヤで実現できる**: pfd-tools 連携が必要なら、pfdsl ネイティブで単体価値を持つフィールド（例: ノード採番のように pfdsl 単体でも `renum` として機能するもの）に意味を載せるか、変換アダプタを外部に置く。spec 本体に外部専用フィールドを彫り込む必要はない。

## Consequences

- pfd-tools 連携準備として起票した2件のうち、`schedule:`（#220, 外部専用）は不採用、`index:` + `renum`（#221）は pfdsl 単体でノード採番として機能するため維持する。`spec_v009` は index 単独統合に縮小する。
- 新フィールド提案のレビュー観点に「pfdsl 単体価値テスト」が加わる。pfd-ops 終端ゲートの後続門番（ADR-0018）が「消費者を書けるか」を問うのに対し、本 ADR は spec フィールドについて「pfdsl 自身がこの値を消費するか」を問う。
- 外部ツール連携は将来、単体価値を持つフィールドへの相乗りか外部アダプタで再設計する。

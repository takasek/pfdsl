# ADR-0027: 仕様相互参照を安定 ID アンカーに移行する

- Status: Accepted
- Date: 2026-07-09

## Context

`docs/spec/spec.md` の相互参照は現状「§15.8」のような節番号の直書きである。
節の挿入・並べ替えが起きると番号がずれ、参照は追随しない。
節番号は位置に依存する値であり、安定参照たり得ない。

#326 では前方参照の stale 検出をマーカー方式へ移行済みである。
実装（`scripts/lib/forward-ref-marker-check.mjs`）は既に `[[SPEC_<slug>?]]` という前方参照マーカーと、見出し行末尾の `(SPEC_<slug>)` という id 定義マーカーの組を扱っている。
本 ADR はこの bracket 構文を相互参照全般の安定アンカーとして正式採用し、構文の詳細・曖昧性回避・レンジ規則・ID の性質を規定する。

## Decision

### 構文

- `(SPEC_xxx)` — ブロック要素に書くと、そのブロックが ID `SPEC_xxx` の定義になる。
- `[[SPEC_xxx]]` — strict 参照。対応する定義が存在しないと lint エラーになる。節番号直書きを置き換える本来の用途。
- `[[SPEC_xxx?]]` — 許容参照。対応する定義が存在しなくてもエラーにしない。
  「まだ定義されていないが将来定義される」前方参照に使う。
  後から定義が現れた場合は「解消済み候補」warning を出す。
  これは #326 で導入済みの forward-ref/implements 突き合わせロジックがそのまま該当する。
  旧称の forward-ref と implements という2語彙は、`?` の有無という1つの記法差に統合される。

### 曖昧性回避

- 定義マーカーは `SPEC_` という固定プレフィックスで識別する。
  見出し行に他の括弧句（全角括弧を含む）が同居していても、`SPEC_` で始まる括弧のみを定義として拾う。
  例: `## foo（説明） (SPEC_001)` は `SPEC_001` のみを定義として認識する。
- fenced code block 内は検出対象外とする。
  `spec.md` には fence 内に `.pfdsl` 本文が埋め込まれるため。
- inline code span 内も検出対象外とする。
  本 ADR 自身のように、構文例をインラインコードで引用するドキュメントが自己参照的に lint 対象になることを避けるため。
- ID の字種は `SPEC_[A-Za-z0-9_]+` とする。
  issue #328 本文は `[A-Z0-9_]` という大文字例を示すが、#326 で既に導入済みのマーカー `SPEC_crossfile_semantics` は小文字 slug であり、曖昧性回避の実体は `SPEC_` プレフィックスそのものにある。
  したがって大文字限定は課さない。
  この偏差は意図的な決定として本 ADR に明記する。

### レンジ規則（ID が指す範囲）

共通原則は「マーカーが属する直近のブロック要素を特定し、次の兄弟境界まで」である。

- 見出し（level N）→ 次の level ≤ N の見出し直前まで。
- list item → item 本体に加え、ネストした子リスト・子 table・item 内の複数 paragraph を含め、次の兄弟 item 直前まで。
- table 行 → その行（横断する全セル）。
- 素の paragraph → 空行区切りのブロック単体。

GFM の table セル内はインライン要素のみのため、「table 行内に list item」は原理上発生しない。
明示的スコープ外として、段落内の一文・item 内の一部分のような inline 粒度のアンカーは対象としない。
本 ADR が定義するのは block-level granularity のみである。

### ID の性質

- ID は不透明・永続・欠番許容とする。
  連番順序のような位置的意味を持たせない。
- renumber 機能は採用しない。
  ドリフトの根本原因は「ID が位置的意味を持つ」ことにあり、renumber はその位置的意味を維持するための機能であるためドリフトを再導入してしまう。
  さらに renumber を行うと、git 履歴・過去 issue・PR 議論・LLM の記憶内に残る旧 ID がすべて dangling になる。
  リポジトリ内の参照は書き換えられても、外部の参照は書き換えられない。
- `[[...]]` は wikilink 互換の記法とする。
  Obsidian 等の既存エコシステムでジャンプ・backlink・dangling 検出といった機能がそのまま付いてくる。

## 検討したが不採用の案

- **#326 のコメントマーカー方式（HTML コメント）を相互参照全般に拡張する案**: HTML コメントは本文非可視のため、引用可能な安定アンカーとして機能しない。
- **見出し限定とし list item・table 行対応を見送る案**: 実運用では table 行単位・箇条書き単位で参照したいニーズがあり、見出し限定では実用に耐えない。
- **renumber ツールを提供する案**: 上述の通り「ID に位置的意味を持たせない」という思想と根本的に矛盾する。

## Consequences

- 本 ADR と同時に最小 lint（ID 一意性チェック・`[[SPEC_xxx]]` の dangling 参照チェック）を導入する。
  実装は `scripts/check-spec-ids.mjs`、`make check-docs` 経由で CI 実行する。
- `spec.md` 全節への遡及的 ID 付与、および既存の節番号参照を ID 参照へ置き換える移行作業は本 ADR のスコープ外とする。
  別 issue として起票し段階移行する（roadmap の `migrate_spec_id_refs` プレースホルダが該当する）。
- 「ID 発行・複数 repo 対応」等の専用ツール化は時期尚早と判断する。
  実 ID の手動管理が実際に痛くなってから検討する（検討記録は issue #328 のコメントに残る）。
- レンジ規則は当面の lint（一意性・dangling 検査）には未実装とする。
  一意性・dangling 検査には範囲計算そのものが不要なため。
  将来「ID 指定ブロック取得」ツールを実装する際の規範として、本 ADR がレンジ規則の定義を持つ。

# spec v0.0.11 机上レビュー + CLI 実測ログ

ADR-0020 の付随資料（第2弾）。`boundary-validation-log.md`（v0.0.8 時点の具体例トレース）の続編として、
v0.0.11 の spec 全体を「過剰な制約 / 暗黙の制約 / 論理矛盾 / 文書フロー / その他」の5軸でレビューし、
怪しい箇所は CLI（`node packages/cli/dist/cli.js check`）で実測した記録。
subflow の agent 実書き実験は sibling の `subflow-agent-probe.md` を参照。

判定記号: ❌ = 矛盾・欠陥（spec 修正を要する） / ⚠️ = 未定義・暗黙（明示決定を要する） / 💭 = 設計議論の余地（現状維持も可）。

## 1. 論理矛盾

### F1 ❌ feedback 除外原則と open input 定義の非対称（階層を跨ぐフィードバックが表現不能）

§2.9.3 は「フィードバック入力（`>>?`）は境界整合の対象外」を原則として掲げ、
**terminal 側**はそれを反映している（「`>>` でも `>>?` でも消費されない」「フィードバックのみで消費される artifact は terminal ではない」）。
しかし **open input 側**の定義は「生成元プロセス（`->`）を持たない artifact」のみで、feedback の除外規定がない。

その結果、子フローが「外部で生成された成果物を `>>?` で受ける」ことが一切できない:

```pfdsl
# parent — E1: 親は feedback を宣言（境界対象外のはず）
---
process:
  P: { subflow: ./child.pfdsl }
---
order >> P -> out
issues >>? P
```
```pfdsl
# child — issues は生成元なし・>>? でのみ消費
order >> work -> out
issues >>? work
```

実測: `error [V025] parent inputs ["order"] ≠ child open inputs ["issues","order"]`。
親が `issues >>? P` を宣言してもしなくても error（E1 / E1b）。
つまり「qa の bug_report が実装詳細フローの中の工程に還流する」という、
PFD で最も普通のフィードバックループは、展開プロセスの境界をひとたび跨ぐと書けない。
原則（feedback は階層 I/O 契約に含めない）と操作的定義（open input が feedback 消費 artifact を巻き込む）の矛盾。

**提案**: open input の定義を terminal と対称にする —
「open input artifact = 生成元プロセスを持たず、**かつ通常入力（`>>`）で消費される** artifact。
生成元がなくフィードバックでのみ消費される artifact は横断的ループの要素であり境界に出さない」。
これで E1（親子両方で宣言）も E1b（子のみ宣言）も pass になり、原則と定義が一致する。
実装は `computeOpenInputs`（`packages/core/src/multifile.ts`）が
`kind === "input"` の消費集合を追加で見るだけ。

### F2 ❌ 診断コード V025 の二重割当（spec §2.8.4 vs 実装 subflow 境界）

spec が名指しするコードは V025 = 「group `parent` チェーン循環」（§2.8.4）のみ。
一方、実装は subflow 境界違反の**全種**（集合不一致・非単射・side 越境・dangling key）にも
V025 を使っている（`multifile.ts`。#148 で V021–V028 を採番した際、
#183 の group 循環 V025 と衝突したまま残った）。
同一コードが「group 循環」と「subflow 境界不一致」の2つの無関係な違反を指す。

**提案**: subflow 境界違反に専用コード（例: V029 系）を割り当て直すか、group 循環側を改番する。
併せて F8（コードレジストリ不在）を参照。

### F3 ❌ 「terminal artifact」概念の三重定義

同じ言葉が3箇所で異なる述語を指している:

| 箇所 | 定義 | feedback のみで消費される artifact は |
|---|---|---|
| §3.3（モデル） | 「どのプロセスにも入力されない成果物」 | 曖昧（>>? が「入力」かは未規定） |
| §2.9.3 / §15.11（subflow 境界） | `>>` でも `>>?` でも消費されない | terminal **ではない** |
| `check --audit`（終端監査、実装） | feedback edge は意図的に無視 | terminal **である** |

audit 側の実装（`audit.ts` の「feedback edges intentionally ignored」）は
消費者監査としては合理的（`>>?` しか消費者がいない成果物は「納品先が疑わしい」ので列挙してよい）だが、
spec 上は §3.3 の一文しか根拠がなく、subflow 境界の terminal と同名のまま逆の判定をする。

**提案**: §3.3 を「終端成果物 = 通常入力（`>>`）で消費されない成果物。
フィードバック消費の有無で二つの部分クラスに分かれ、監査（audit）はフィードバック消費を無視し、
subflow 境界判定はフィードバック消費を除外する（§2.9.3）」のように、
述語を2つに分けて明示的に命名する（例: *audit-terminal* / *boundary-terminal*）。

### F4 ❌ §15.8 / §15.9 の「マルチファイル仕様（将来版）に委ねる」が stale

v0.0.8 で §2.9 マルチファイル意味論が新設済みなのに、
§15.8（location のクロスファイル参照）と §15.9（クロスファイル revises）は
いまだ「将来版に委ねる」と書いている。読者は「マルチファイル仕様はもう在るのでは？」と混乱する。

**提案**: 「§2.9 のマルチファイル意味論は `subflow:` / `extends:` のみを対象とし、
`location:` / `revises:` のクロスファイル意味論は引き続き対象外（将来の拡張）」と書き換え、
§2.9 冒頭にもスコープ（subflow / extends に限る）を1文明記する。

## 2. 暗黙になっており定義したほうがよい制約

### F5 ⚠️ 境界判定の対象が「edge 参加 artifact」に限られることが未規定

子フローに edge を持たない node-decl（例: `memo`）があるとき、
spec の字義（open input = 生成元なし / terminal = 消費なし）では `memo` は
open input **かつ** terminal になり、境界不一致で error になるはず。
実測は **OK**（`computeOpenInputs` / `computeTerminals` が edge 参加 artifact のみを走査するため）。
実装の挙動は妥当だが spec から導出できない。

**提案**: §2.9.3 に「境界判定は edge に参加する artifact のみを対象とする。
孤立 node-decl・frontmatter のみで宣言された artifact は open input / terminal のいずれにも数えない」を明記。

### F6 ⚠️ プロセス完全性（§15.2）の「1入力」に `>>?` が数えられるかが未規定

`a >>? P; P -> b`（通常入力ゼロ・feedback のみ）は実測 **OK** — 実装は `>>?` を「1入力」に数える。
spec は「少なくとも1入力・1出力」としか言わず、feedback が入力にカウントされるか読み取れない。
さらに subflow と交差すると: 通常入力ゼロの展開プロセスは「子の open input 集合 = ∅」を要求され、
edge を持つ DAG の子フローでは事実上満たせない（F1 修正後も同様）。

**提案**: §15.2 に「入力には `>>?` を含む」と明記。
subflow 展開プロセスについては「通常入力（`>>`）を1つ以上持つべき」の note を §2.9.3 に置く。

### F7 ⚠️ frontmatter のみで宣言された artifact は黙って消える（V020 と非対称）

frontmatter `process:` に宣言して edge 不参加なら **error**（V020、§15.10）。
ところが frontmatter `artifact:` に宣言して本文に現れない ID（typo 等）は、
実測でノードにもならず normalize にも現れず、警告すらない（E10）。
チェーン削除の残骸・typo の検出という §15.10 の動機は artifact 側にも等しく当てはまる。

**提案**: 対称の warning（例: W006「frontmatter 宣言 artifact が本文に不参加」）を追加するか、
少なくとも §15.10 に「artifact は対象外（孤立成果物宣言と区別できないため）」と非対称の理由を明記する。

### F8 ⚠️ 診断コードレジストリが spec に無い

spec が名指しするコードは V001–V003 / V007 / V008 / V020 / V025 / V031 / W001–W005 程度で、
実装が発行する V004–V006 / V009–V019 / V021–V028 は spec に現れない。
§16 エラー方針は「条件の散文リスト」でコード列が無く、実装との突合が人力になる
（F2 の V025 衝突が見逃されたのはこの死角の帰結）。

**提案**: §16 を「コード / 条件 / severity / 定義節」の表に改める。
`check --json` の出力コードと spec 表の一致を CI で検査できる形にする（#138 と同型の機械化）。

### F9 ⚠️ bare-id の禁止文字リストに `,` が無い

§4.1 の禁止文字は `[ ] ; # "` と演算子のみ。カンマが無いため、字義上は `x,y` が単一の bare-id になり
`[a,b]` の集合記法と両立しない。実測は `error [P005] Expected >> or >>? after artifact, got COMMA` —
lexer はカンマを常に区切りトークンとして扱う。

**提案**: §4.1 の禁止文字に `,`（および空白類が区切りであること）を追記。

### F10 ⚠️ boundary リネーム下のメタデータ権威が未定義

§2.9.3 のメタデータ権威は「境界 artifact は親子で**同一 ID** を共有し、食い違いは親優先」と
同一 ID 前提で書かれている。`boundary: { order: incoming_order }` のとき、
親 `order` と子 `incoming_order` のメタデータ（status / label / criteria）が食い違ったら
どちらが正本か、規則の文面が届いていない（恒等照合の時代の文章のまま）。

**提案**: 「メタデータ権威は**実効対応**（明示マップ ⊕ 恒等）で対応づけられた境界 artifact 対に適用し、
親側宣言を優先する」と書き換える。

### F11 ⚠️ §15.3 strict feedback 検査の方向が未規定

§15.3 は「A が Primary Graph 上で **P に関連する到達可能成果物**であること」— 「関連する」「到達可能」の
向きが読み取れない（P→A? A→P? 無向?）。CLI の `--strict` は
「feedback source が target process から到達可能でなければ error」と方向を決め打ちしている。

**提案**: 「P から Primary Graph の有向辺を順方向に辿って A に到達できること」と方向を明記。

### F21 ⚠️ W003（status 非単調）の subflow 跨ぎ評価が未規定

親で `spec_doc >> impl -> code`（impl は subflow 持ち）、子で `spec_doc >> write_code -> ...` のとき、
W003 の「出力が done なのに入力が done 未満」はどのファイルの status 宣言で評価されるのか、
クロスファイルで評価されるのかが書かれていない（agent 実験Aの被験者が独立に指摘）。
実装はファイル単位でしか評価しないはずだが、境界 artifact のメタデータ権威（親優先）と
組み合わせると「平坦化ビューでは警告が出るがファイル単位では出ない」ケースが構成できる。

**提案**: 「W003 を含む status 系検査はファイル単位で閉じる（V001–V003 と同様、§2.9.1 の原則に従う）」を
§2.9.1 に1文追記。

## 3. 過剰と思われる制約

### F12 💭 子フローの副産物 terminal も親出力への露出を強制される（全単射の代償）

E4: 子が `order >> work -> [out, work_log]` のとき、親が `work_log` を受けないと error。
全単射は清水法の階層整合（親の外部 I/O = 子の外部 I/O）として原理的に正しいが、
「ログ・帳票などの副産物を持つライブラリ子フローの再利用」では親ビューが副産物で汚れる。
agent 実験 B（`subflow-agent-probe.md`）では被験者がこの罠を正しく回避できたが、
「親側の出力 edge を子の terminal 数に合わせて増やすのが正攻法」という誘導が spec に無い、
という指摘が被験者からも出た。

**提案（2段階）**: まず §2.9.3 の粒度差の段落に
「子の terminal が親の出力より多い場合、親の出力 edge を増やして全て露出させるのが正攻法
（N:M マップに走らない）」の1文と例を追加する（規則変更なし）。
「terminal を境界に出さない」ための注釈（例: 子側での `internal` マーク）は
階層整合を弱めるため、実需が再度観測されるまで導入しない。

### F13 💭 status 語彙の説明が git ワークフロー固有

§2.7.1 の gloss（「ブランチ・PR がオープン」「main にマージ済み」）は DSL 仕様としては
ドメイン過剰。PFD は非ソフトウェア工程（規制提出物・帳票等）にも使う。

**提案**: gloss を一般語（未着手 / 生産中 / 完了・受入済み）にし、git の言い回しは例として括弧書きに落とす。

### F14 💭 revises の分岐禁止（§15.9）

「複数の Artifact が同一 Artifact を revises」は error だが、現実の版系列は分岐しうる
（同一 v1 から社内版と提出版を派生等）。線形リスト制約は「最新版はどれか」を機械判定するための
意図的な単純化と読めるが、その動機が spec に書かれていない。

**提案**: 制約は維持しつつ、§15.9 に理由（最新版の一意決定のため。分岐が要るなら別 ID 系列に分ける）を1文追加。

## 4. 文書としての不自然な流れ

- **F15 §16 が §15 に無い規範を含む**: W001（parts メンバー edge 不参加）と W003（status 非単調）は
  §16 のリストにしか定義がない。§15 に定義節を持つ W002 / W005 と扱いが割れている。
  → W001 を §15.5 に、W003 を §15.6 か新節に移し、§16 は表化（F8）して参照専用にする。
- **F16 §2.3 が雑居ビル化**: 「IDメタデータ定義」の節に document-level の `basePath` と、
  意味論の深い `subflow:` / `boundary:` が同居。subflow/boundary は §2.9.3 への前方参照だけ残して
  フィールド表程度に痩せさせ、basePath は §2.2 直後の独立小節にする方が読み順が自然。
- **F17 §15.4 と §16 の重複 edge の齟齬**: §15.4「無視してよい」/ §16「warning可」。
  どちらも任意だが、読者には「無視するのか警告するのか」が二度定義に見える。1箇所に寄せる。
- **F18 §14 のタイトル「（推奨）」**: fmt / diff の安定性はこの順序に依存しており、
  実装上は事実上 normative。「推奨」の語が実装者に選択の余地があると誤読させる。
  「正準順序（fmt が従う規範。他処理系は任意）」等に改める。
- **F19 subflow 規範の分散**: 境界規則の normative な文が §2.3（フィールド）/ §2.9.3（意味論）/
  §15.11（検証）の3箇所に部分重複して散っている。F1 の非対称が生まれたのも、
  terminal の除外規定を §2.9.3 に足した際に open input 側の文が3箇所とも据え置かれたため。
  定義は §2.9.3 に一本化し、§15.11 は「§2.9.3 の述語を検証する」+ error 列挙のみにする。
- **F20 subflow の絶対パス禁止の error が §15.11 / §16 に無い**: §2.3 は「絶対パス・URL 不可」と言い、
  extends 側は §15.12-4 と §16 に error 行があるのに、subflow 側の対応行が両方に無い
  （実装は V021 で error）。対称に追記する。

## 6. 仕様・ツール改善につながるアイデア（レビュー副産物）

1. **spec ↔ checker のコード整合を CI で検査**: `check --json` が発行しうる全コードの一覧を
   core からエクスポートし、spec §16 の表（F8）との差分を `check-docs` 系で検出する。
   V025 衝突（F2）はこの検査があれば混入時点で落ちていた。
2. **agent 実書きプローブの定常化**: 「spec だけを読ませた sonnet に書かせ、CLI を正解器にして採点する」
   手法（`subflow-agent-probe.md`）は、ADR-0020 の手トレースが取りこぼす
   「読者がどこで誤読するか」を検出する。仕様の大きな追加時のレビュー観点として
   `docs/review-prompts.md` に C 系（仕様ユーザビリティ）として追記する余地がある。
3. **`check` の境界不一致メッセージに差分ヒント**: 現状は両集合の全列挙。
   `missing in parent: [...] / extra in parent: [...]` 形式にすると、agent 実験で観測した
   自己修正ループ（何が原因かの特定）が速くなる。
4. **`fmt --mode flows` と subflow の関係の文書化**: 正規形が subflow を保持しない（§13）以上、
   構造 diff は subflow 版とインライン版を別物と判定する。この帰結（リファクタリングで
   フローを子に切り出すと diff が全張り替えになる）を §13 に注記しておくと驚きが減る。

# ADR-0003: update 意味論の不採用 — 可変リソースはスナップショット、update はビュー

- Status: Accepted
- Date: 2026-06-11

## Context

清水吉男の本家 PFD には update 概念（既存成果物を書き換える操作）が存在する。
DB・本番環境・サービング中モデルなど、継続的に変化する可変リソースを PFD でどう表現するかという問題がある。

update 意味論を導入すると以下が壊れる:
- **ランク計算**: 「更新後の自分」を入力とする循環が生じる
- **単一生成元制約**: 同一 artifact への複数書き込みをどう扱うか曖昧になる
- **DAG 性**: 循環参照によりトポロジカルソートが不能になる
- **diff 安定性**: スナップショット比較の基準点が失われる

## Decision

update 意味論は導入しない。

可変リソースは**時点固定スナップショット artifact** として表現する:
- DB の状態 → 日次ダンプ
- 本番環境 → リリース版
- サービング中モデル → トラフィックスナップショット

update の直感（「このモデルが継続的に更新される」という視覚的表現）は、 描画側のビューで提供する: `revises:` チェーンを1ノード＋自己更新風に畳む。

## Rationale

意味論とビューの分離により一長一短を両取りできる。

スナップショット表現の利点:
- 各時点が検証可能なチェックポイントとなる
- 計画・進捗管理という PFD の目的に本質的に合う
- ランク・到達性・正準順序の静的解析性を保つ

実行時の状態遷移は状態機械の領分であり、PFD の守備範囲外である。
「フローを実行する」と「フローを設計する」は別の関心事として分離する。

## Consequences

- スキルのスナップショットルールとして品質ガイドに収録
- issue #8 のコメントに設計指針として記録済み
- 実例: `docs/examples/ml-model-dev.pfdsl` の `model_snapshot_*` artifact 群

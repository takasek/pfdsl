# ADR-0022: ファイルを書き換える CLI コマンドは gofmt 出力モデルに従う

- Status: Accepted
- Date: 2026-06-27

## Context

`pfdsl reindex`（#221）の設計で、ファイルを書き換えるコマンドが2種類の出力を持つことが問題になった。
ひとつは**書き換え後のファイル本体**、もうひとつは**何をどう変えたかのレポート**（`index:` の `old→new` マッピング）である。
レポートは「変更を他ファイルに波及させたい」「git 未コミットでも変化を確認したい」という動機から重要だが、両方を同時に stdout に載せると衝突する。

当初案は「レポートを stderr 固定」だったが、機械可読データ（`--json`）を stderr に流すのは Unix の慣習（stdout=データ、stderr=帯域外の診断）に反する。

## Decision

**ファイルを書き換える pfdsl CLI コマンドは gofmt 出力モデルに従う。**

- **既定（`--write` なし）**: 変換後のファイル本体を stdout に出す（プレビュー・パイプ用）。ファイルは書き換えない。
- **`--write`**: ファイルをインプレースで書き換える。本体はディスクへ行くので、stdout は空き、そこに**変更レポート**を載せる。
- **`--check`**: 書き換えず、変換で何か変わるなら exit 1（CI のドリフト検出）。
- **`--json`**: レポートを機械可読 JSON で出す（波及・外部連携用）。
- **stderr は診断・エラー専用**。一次データ（本体・レポート）を stderr に流さない。

副作用（書き込み）と stdout の中身を分離するのが核心である。
本体とレポートを同時に stdout へ載せようとするから衝突する。
モードごとに stdout の中身を切り替えれば、stderr に逃がす必要がなくなる。

## Rationale

1. **1モード1次データ**: stdout の中身は「そのモードで要求された有用な出力」1種類に保つ。`--write` 時は本体がディスクへ行くため stdout が空き、レポートを載せられる（gofmt `-w -l` と同型）。

2. **stderr の役割を守る**: stderr は人間向けの帯域外メッセージ専用とし、パイプ可能な機械可読データを流さない。`pfdsl reindex f --write --json > changes.json` が `2>` なしで成立する。

3. **`fmt` との一貫**: `pfdsl fmt` が既に「既定 stdout / `--write` インプレース / `--write` は stdin 不可」を採用済み。`reindex` も同型にすることで CLI 全体の文法が揃う。`--write` 時に本体でなくレポートが出るのは gofmt 先例どおりで、変則ではなく定石。

## Consequences

- `fmt` と `reindex` が同じ出力モデルを共有し、将来ファイルを書き換えるコマンド（autofix 等）も同じ規約に従う。
- 変更レポートは `--json` で機械可読になり、index 変更の他ファイル波及や外部ツール連携に使える。
- `reindex` 固有: レポートは `pfdsl diff` の語彙（`+` 新規 / `~` 変更 `旧 → 新`）に合わせ、process は `P{index}`・artifact は `D{index}` のプレフィックスで示す。
- `--write` は stdin（`-`）と併用不可。`--check` と `--write` は併用不可（読み取り専用 vs 書き込みの衝突）。

# ADR-0014: 条件付きゲート項目は反実仮想テストで判定する

- Status: Accepted
- Date: 2026-06-14

## Context

pfd-ops のワークサイクル終端ゲートに
「PFD が効いた局面があれば `docs/pfd_payoff_log.md` に追記した」という項目がある。
「あれば」という条件節を持つが、完了形のチェックリスト項目として並ぶと
「埋めるべき欄」として作用する。

実際にあるサイクルで後付けの payoff 草稿が生成された。
却下された実物を以下に引用する（本 ADR を git 履歴に頼らず単体で読めるようにするため）:

> ## 知見が lint 要件を完全指定する
> - 局面: `implement_lints` プロセスの入力に `findings_r12`（実践ラウンド知見）を依存として登録していた
> - 効果: 実装開始時に「何を作るか」ではなく「知見から何が導出されるか」という問いに変換された。
>   W001（parts member without edges）は spec 条文（§17.4）単独でなく、実際の執筆で踏んだ
>   浮いたノード問題から来ている。依存グラフが「参照すべき知識」を作業開始前に明示する機能を果たした
> - 参照: `.pfdsl/plan.pfdsl` の `implement_lints` エッジ

この草稿は反実仮想テストで落ちる。「依存グラフがなければこの判断は違ったか？」を問うと、
issue 本文だけで同じ実装に到達しており、`findings_r12` が依存にあった事実は本物でも
結果を変えていない。効果は捏造だった。草稿は却下した。

これは payoff_log 固有の問題ではなく、「あれば〜した」型の
条件付きゲート項目に共通する病理である。条件の充足判定が主観に委ねられ、
チェックリストの充足圧力が「該当なし」より「何か書く」を選ばせる。

## Decision

条件付きゲート項目は、条件の充足を**反証可能な操作テスト**で先に判定する。
デフォルトは不充足（書かない）。

payoff の判定テスト:

> 依存グラフ／プロセス分解がなければ、この判断・作業は違っていたか？
> Yes かつ「どこがどう違ったか」を1行で引用できる時のみ記録する。
> 引用できなければ該当なし。

## Rationale

1. **完了形チェックリストは充足を誘発する**: 埋める方が完了感があり、
   「該当なし」を選びにくい。圧力は文言を「したものだけ書く」へ反転させない限り消えない。

2. **verification-before-completion と同型**: 証拠（引用）なき主張は嘘。
   payoff も体感の引用が証拠であり、引用できない payoff は存在しなかったものとして扱う。

3. **ADR-0004 の粒度判定テストと同型**: 「上流方針の合否基準を下流作業なしで書けるか」
   と同じく、判断を反証可能な操作テストに落とすと自己欺瞞が機械的に弾かれる。
   方法論は判定を主観から操作テストへ移すことで信頼性を得る。

## Consequences

- pf-ops 終端ゲートの payoff 行を反実仮想テスト付きに改訂する。
  pfd スキルは別リポジトリで管理されるため、本 ADR は pfdsl 側の決定記録であり、
  スキルへの反映は別タスクで行う。
- 同型の条件付き項目（「実行中に発見した新プロセス・成果物を計画 PFD に追記した」等）にも
  同じ判定の適用余地がある。
- `docs/pfd_payoff_log.md` 冒頭に判定規約を1行加え、ログ単体でも反実仮想テストが参照できるようにする。
  後付けが構造的に除去され、公開記事（issue #12）の素材として残るエピソードが本物に限られる。

# PFD レビュー観点（A/B/C カタログ）

レビュー観点のカタログ。品質ガイドが「書くときのルール」、本カタログは「レビュー時の問い」。各問いは実際に検出された誤りに由来する。一次情報は本ファイル — pfdsl スキル（references/ に同梱）と pfd-retro スキル（ポインタ参照）の両方から使われる。A/B は任意の PFD を、C は normative 仕様文書（自リポで保守する DSL・プロトコル・API contract・スキーマ等）を問い詰める。リポ固有の具体例は各リポの review-perspectives instance（例: `.pfdsl/review-perspectives.md`）で育てる。

## A. 図 vs 現実

- **エッジ実在性**: 各エッジについて「このプロセスは本当にこの入力を読むか／この出力を本当にこのプロセスが作るか」（例: 派生文書が生成物でなく元データから書かれていないか。実際には受け取らない提出物を生成物に持っていないか）
- **駆動源**: プロセスを実際に起動する主入力はどれか。主入力と還流を取り違えていないか（例: 起票・登録の駆動源は判断・対話で、レビュー知見はそれに沿う還流）
- **名前の一般化水準**: ノード名が実態より狭くないか（例: 単一コンポーネント名が実際は一群全体を指す）。逆に広すぎないか
- **偽の不変性**: 生成元なしの「外部入力」は本当に誰にも更新されないか（例: 仕様・定義文書は運用知見を経て改訂される — 保守プロセスが要る）
- **入力充足**: 各プロセス「この入力だけで出力を作れるか」（例: 派生成果物に基となる素材が欠けていないか。参照 ID の捏造は入力欠落のサイン）

- **status 単調性**: 出力 artifact が `done` なのに入力 artifact が `done` 未満の対はないか（machine-checkable。チェッカがあれば検出するが status 未宣言 artifact は対象外で人手 backstop）
- **同種対称性**: 既存の同種 artifact（同グループ・同接頭辞等）と同じ下流消費者を新 artifact も備えているか。欠落は「追加者が全消費者を列挙しなかった」形で発生する

## B. 粒度・型

- **万能成果物**: 多数プロセスに消費される単一の不定形 artifact は型分割のサイン（例: 総称的な「やりとり」を複数の役割型に分割し発散を解消）
- **プロセス実在性（双方向）**: 独立起動の作業か、別活動の終端動作か（例: ログ追記は親活動の終端動作で単体プロセスでない）。逆に、委譲・差し戻し・固有の検証が発生した活動は隠れた独立プロセス
- **自動化はノードでなく description**: 既存ノードの出力・同期を機械化する仕組み（CI・スクリプト）は新プロセスでなく所有ノードの description に記す。新ノードは新 artifact 型か新依存が生まれるときだけ
- **並列主張への挑戦**: 図が並列と主張する作業群に共有される決定はないか。共有決定が往復で形成されるなら分割せず統合
- **修正案への再挑戦**: 一次修正（上流方針の artifact 化等）が正しい型か再度問う。方針の合否基準を下流作業なしで書けないなら入力でなく出力
- **型違い**: 活動が artifact として、モノがプロセスとして書かれていないか

## C. 仕様・制約（spec を問い詰める）

対象は図でなく normative 仕様文書の内部整合。リポが DSL・プロトコル・スキーマ等の仕様を自リポで保守する場合に適用（pfd-retro のゲート条件）。仕様固有の実行手順・具体例は各リポの review-perspectives instance（例: `.pfdsl/review-perspectives.md`）のバインディングが指す。

- **対称性の欠け**: 対になる定義・規則（入力↔出力、開放入力↔終端、継承↔埋め込み等）で、片側にだけ除外規定・error 行が欠けていないか。非対称は表現不能な入力を生む
- **原則と操作的定義の一致**: 節冒頭の原則を後続の操作的定義が実際に実現しているか。原則だけ読んだ読者と定義だけ読んだ読者が同じ判定に到達するか
- **字義 vs 実装**: 定義を字義通りに適用したとき、実装と食い違う入力を構成できないか。どちらに倒すか明示決定があるか
- **同名異義**: 同じ用語が節・ツールごとに異なる述語を指していないか
- **読者到達性**: 精読者が正解に到達できるか、そして**どう到達したか**。回避行動（情報を捨てて通す）・類推・確信度「中」の推測による正答は、pass でも仕様の欠陥の証拠

自リポの仕様監査で得た具体例・機構はこの観点の育て先＝各リポの review-perspectives instance（例: `.pfdsl/review-perspectives.md`）に蓄積する。

# PFDSL 連載記事

PFDSL の問題意識と設計根拠を説明する公開技術記事（連載・日本語）。issue
[#12](https://github.com/takasek/pfdsl/issues/12) に対応し、リポの生態系図上は `write_article` の出力 `i12_article`。
この README は連載の**構成プラン（設計書）**。本文はこの後 `docs/articles/NN-slug.md` に書く。

**書き方の基本方針（R5 で確定、R8 で構成を痛み階段へ再設計）**: 序・hook で釣らない。伏線/回収/「○○へ橋渡し」式の物語的レトリックを使わない。
**素直な技術記事として「痛み（問題）→ 解決」で書く**。免責・N=1 開示・反実仮想の限界・自己批判の連打は誠実さでなく言い訳/ノイズとして削ぐ（残すのは下記2点のみ）。
読者の現場への適用手順（「あなたのリポジトリで」）は**別記事**で本連載のスコープ外。

## 解く問題（記事全体の頂点・一文）

> **プロジェクトの進め方 — 誰が何を作り、何に依存して完了するか — が暗黙知のままだと、人も AI も毎回文脈の口移しでしか参加できない。
> プロセスを成果物依存グラフとしてテキストに外化し機械検査に載せれば、文脈ゼロの参加者が図を読むだけで作業を選び、回し、監査できる。**
> — PFDSL はそのための言語と運用。

逆ピラミッド: この頂点（＝第4部の到達点の約束）を第1部の冒頭近くで言い切る。
以降の各部はこの到達点に必要な性質を痛み駆動で1つずつ実証する。各部末で要点を再宣言しない。

## 読者

プロセスをコードと同じく**テキスト・版管理・CI に載せたい開発者**（git/diff/レビュー/CI を回し、AI エージェントにも仕事を振り始めた層）。
PFD の前提知識は仮定しない。第1部の入口だけ「依存関係は誰の仕事にもある」と広く入り、以降は開発者の既知概念（git・diff・CI・レビュー・subagent）で内部用語を橋渡しする。

## 連載の構成 — 痛み階段（R8）

4部 = 読者の4つの痛み。**各部は「痛み → 解 → 実演・実話 → 払うコスト（負例1件）」の完結構造**。
実話は `docs/pfd_payoff_log.md` から各部に配分する（下のエピソード配分表が正）。旧構成（言語設計を前半・実践を後半）は実話が後半に偏り前半が内向きの設計解説になったため廃止。

| 部 | 痛み | 解（実演する性質） |
|---|---|---|
| 第1部 | 今、何に着手できるかを毎回頭の中で計算し直す | 成果物依存グラフ + status → 着手可能集合の機械導出 |
| 第2部 | 図もロードマップ文書も腐る | テキスト性 → diff・レビュー・CI に載る |

## 第1部 節構成 — 「今、何に着手できるか」

冒頭に頂点の一文を言い切る（連載の道筋段落は数文以内・目次代わり）。

1. **痛み** — 書き捨てでないプロジェクトには依存関係が生じる（誰の仕事にもある＝広い入口）。
   依存把握が頭の中にしかないと、着手判断・優先順位づけ・新提案の影響評価を毎回頭の中でやり直す。
2. **成果物で考える** — プロセスでなく成果物を明示すると「何を生んで完了したか」が検証可能になる — これが清水吉男の PFD の核（PFD をここで1文定義）。
   README の最小例で最小記法（`A >> P -> B`・集合記法・status frontmatter）を導入する。
3. **着手可能集合の機械導出** — 入力が全て done のプロセス = 今着手できる作業。`ready` 相当の導出を動作で見せる。
   実話2件: オープン issue 8件を図に落とすと並列着手9本が status から即答できた／新提案の受け入れは既存グラフへの1チェーン追加で並列性・接点・合流点が確定した。
   **N=1 の一文はこの最初の実話の前に一度だけ置く**（「以降の実例はすべて著者が自リポジトリを数週間運用した自己観察で、判定は著者による。効果測定でなく『この条件でこう効いた』の記録」）。
4. **払うコスト（負例1件）** — 前倒しモデリングの初期コスト。
   フラットな ToDo リストと違い、最終成果物から遡って依存を書き切る手間を先に払う。

## 第2部 節構成 — 「図は腐る」

1. **痛み** — 作図ツールの図・ロードマップ文書は更新が続かず、実体と乖離して死蔵される。
   乖離した図は参照されなくなり、参照されない図は更新されない。
2. **テキストだから git に載る** — 座標を持たずグラフ構造だけを書く。変更が diff になり、diff がレビュー対象になり、履歴が残る。
   既存記法の不足はここに**従属的に**吸収する（各1〜2文・比較を主題化しない）: 依存管理の代表 WBS は包含ツリーで*依存グラフではない*（生成エッジを持たない）。作図ツールは絵を描くが diff を持たない。DFD・先行例（SWET の PFD Draw Tool）・diagram-as-code は各1文。
3. **検査が CI に載る** — `check` を動作で見せる。
   「絵を出すだけなら Graphviz でできる。PFDSL が足したのは絵でなく、PFD 固有の制約を構文として検査する層だ」を、Graphviz では出ないエラーが PFDSL では出る**対比デモ**として実演内に示す。
   CI に組み込めば図の腐敗が検出可能になる。
4. **払うコスト（負例1件）** — テキスト化してもグラフと作業実体（issue 等）のドリフトは残り、同期を保つ監査ステップが要る。
   テキスト化はドリフトを消すのでなく、検出可能にする。

## 第3部 節構成 — 「構造の見落としは人が拾えない」

1. **痛み** — ガイドとレビューを整備してもミスは再発する。
   実話: 品質ガイドの例文に明記済みの入力欠落が、その直後の実践ラウンドで再発した。消費されない成果物の放置も2例。
   列挙的・網羅的な確認は人の認知に向かない — ルールの質の問題ではない。
2. **PFD 固有の意味論を機械検査する** — 単一生成元・DAG 性・終端健全性・孤立プロセス・完了基準の欠落を検査する（テキストで CI に載ること一般でなく「何を検査するか」が PFD 固有）。
   ルールと機械検査の**二層分業**をここで1段落: ルールは設計判断（粒度・命名・外化の要否）だけを担当し、列挙的確認は全部機械に送る。
   実話3件: 作った警告が自分の roadmap の完了基準欠落を直撃した／運用で見つけた見落とし様式（チェーン削除後の孤立プロセス）を lint 要件に変換し、以後は機械検出になった（閉ループ）／文脈を持たない外部レビュアー（AI）にリポ構造を監査させたら、指摘14件が全て機械検査の届かない散文層に集中し、機械化済みの領域からは新規欠陥が出なかった — 分業の防波堤が指摘の分布として可視化され、次に機械化すべき位置まで特定された。
3. **検査可能性を守るために入れない機能** — 統一理由は「文脈ゼロの読者（人も機械も）が静的に読み切れる図であることを壊すから」。
   update（スナップショット + ビューで代替。**改版3形態と `>>?` をここで図つき導入** — 別 artifact／`>>?` 収束／定常サイクル）／条件分岐（合否・承認は承認記録・結果報告書 + `>>?` へ変換。清水 PFD も元々持たない）／subroutine（似た Process は横断ラベルで代替）。
4. **図そのものが検査装置になる** — 機械検査に載る手前で、図に書くこと自体が構造を暴く。
   実話3件: 「並列で起草できる」という直感が図で反証され、相互依存の発見につながった／仕様の境界ケースを図の上でトレースした検証ラウンドがそのまま11のテストケースになった（仕様テキストだけでは非自明ケースを事前網羅できなかった）／相互依存を1プロセス複数出力に畳んだ roadmap が、後続着手者が罠を踏む前に正解構造を渡した。
5. **払うコスト（負例1件）** — 粒度の読み違えは残る。
   粒度ルールに従って一度は統合したプロセスが、委譲した一括起草が存在しない参照 ID を捏造して初めて「隠れた独立プロセスだった」と分かり、分割し直した。
   図が事前に正解を出したのでなく、失敗が入力不足を露出させた。

## 第4部 節構成 — 「AI に文脈を口移しする」

1. **痛み** — エージェントにも新規参加者にも、毎回プロジェクトの文脈を口移しで説明する。
   委譲した subagent は暗黙知を踏み抜く。進め方が人の頭の中にしかない限り、参加コストは人数と回数に比例する。
2. **運用を PFD に焼き付ける** — プロジェクトの運用そのものを外化する: **roadmap**（何を何の前に作るか・今何に着手できるか。status を持つ）と **workflow**（この作業はどう繰り返されるか・誰が何をトリガーに何を行うか）。
   それを**リポ非依存の汎用スキル**が読んで回す。層設計は平易語で圧縮: 汎用手順は PFD 概念だけから導ける（着手判断＝入力が全 done・進捗＝status 更新・1サイクル1プロセス）／リポ固有の中身は図の隣の説明ファイルに逃がす／issue 管理の流儀は選んで載せるプリセット。
   スキル名（pfd-cycle / pfd-ops / pfd-retro）はここで一度だけ紹介し、以降「運用ハーネス」で受ける。
3. **実証** — 冒頭に N=1 の射程を一句（「これも同じ自リポの自己観察」）。
   文脈ゼロの AI エージェントが roadmap/workflow を読んで着手可能集合を導出し、1サイクル完走。その過程で設計者の見落とし2件（issue クローズの timing・チェーン未定義による孤立プロセス）を露出させ、直して再実行したら消えた。
   外部リポへの初適用でリポ非依存を確認: スコープ変更2件を完了基準の1行編集とチェーン1本の追記で吸収し、再構成が不要だった。
4. **払うコスト（負例1件）** — 焼き付けてもルール文言に書かれない意図（暗黙スコープ）は残る。
   実話: 運用ルールの「open のもの」という文言が意図した適用対象を持たず、誤適用が起きた。定期監査が「あるべき記述の不在」としてこれを拾い、文言が修正された。
5. **総括 — 3条件の命名（連載で唯一ここ）** — ①成果物と生成依存エッジの明示（第1部が実演）②PFD 意味論の機械検査（第3部）③テキスト性（第2部）。
   三つが揃うと、文脈を持たない参加者が図を読むだけで作業を選び、回し、監査できる — 頂点の一文の実証が完了する。
   **「回収」「伏線」等のメタ語は使わない**。命名と対応の提示だけで締める。

## 書き方の規律

- **痛み（問題）→ 解決**。各部・各節は「次に解く痛み・小問題」から淡々と始める。前部を引き継ぐ宣言（「ここまでが…だが」）や次部への予告（「第N部で詳述」）を書かない。
- **物語的メタ装置を使わない**: 「橋渡し」「コーダ」「蝶番」「結」「回収する」「連載を閉じる」式の構造語を本文に持ち込まない。否定形での自己言及も書かない。
  第4部§5の3条件命名は総括であって回収レトリックではない — 命名と対応提示のみで書く。
- **免責は最小2点のみ**（下記以外の自己批判・方法論的懺悔は本文に書かない）:
  1. N=1 の一文を第1部の最初の実話の前に一度（文言は第1部§3）。第4部§3の頭に射程の一句を引き継ぐ。
  2. 各部に「払うコスト」型の負例を**1件だけ**（第1部: 前倒しモデリングの初期コスト／第2部: ドリフト同期監査／第3部: 粒度の読み違え／第4部: 暗黙スコープ）。読者が同じ道具で実際に踏む落とし穴に限る。
- **記号・略号を本文に出さない**: `ADR-NNNN`/`V001`/`W002`/`V020`/`L1〜L4`/`cold subagent`/`companion` 等は平易語に開き、出典は脚注へ。
- **dogfooding という語を使わない**: 「著者が自リポジトリで」等の平易語で受ける（未定義語を増やさない）。
- **表は最小**: 短い対比（3項目以上・各セル1行）だけ表。物語・事件・反実仮想は散文。本 README の整理用の表は本文に出さない。
- **用語の初出マップ**: PFD の1文定義・最小記法・status・着手可能集合＝第1部／`check`（動作として）・diff・CI＝第2部／単一生成元・DAG・終端健全性＝第3部§2／`>>?`・改版3形態＝第3部§3（図つき）／roadmap・workflow（種別）・運用ハーネス＝第4部§2／3条件の命名＝第4部§5のみ。

## 執筆者メモ（本文に出さない・self-check 用）

provenance 事故・altitude 混同を防ぐための控え。本文には出さず、書くときの照合に使う。

- **清水軸（keep/tighten/drop/add）×扱い軸（課す/採らない）は直交**。継承（keep）を機械検査として課す（単一生成元）も、継承した不在を継続して採らない（条件分岐）もある。
  keep: 有形性(0001)・単一生成元(V001)・DAG・条件分岐の不在(0005)／tighten: 粒度4基準(0004)／drop: update(0003、清水軸で唯一の意味論変更)／add: テキスト DSL 化・tooling・`tags`(0019)・終端門番の運用側(0018)・最小コア＋拡張(0021)。
- **provenance スコープ（vs 清水 PFD）**: 清水の意味論に対し意図的に変更（drop）したのは update 不採用ただ一つ。他は継承（keep）か追加（add）。
  「追加」概念が PFDSL 内部で版を重ねて意味論を変える（`tags` の 0019 など）のは**内部版進化**で別軸。「PFDSL は自分の意味論を一度も変えていない」と誤読しないこと。
- **二層と二段は別レベル**: 第3部§2の「ルール対 機械検査」の二層（ADR-0006）と、「言語の `check` 対 運用側の点検」の二段（ADR-0018）は別で、後者は前者の機械検査の内訳。
  運用側の点検（終端ゲート・定期監査）は第4部§4で追加的に出す。第3部の機械検査が不完全だったと読ませない — 網を運用面へ広げる話であって覆す話ではない（誤読防止の接続を1文）。
- **採否**: ADR-0008（group 指針）は本連載から外す。ADR-0011（能力成果物の世代還流）は第3部で1文触れる程度。
  `schedule:` 全廃棄（ADR-0021）は第3部§3で任意・1文まで（実装を一周してから判定基準が効いた、の一例。主題化しない）。
- **2026-07-06 防波堤エントリの使い方**: 「監査機構自身を監査する再帰」の学習部分は内向き話なので本文に出さない。
  指摘の修理状況（#351/#352 は未解決）に触れず「分布の実証」までで切る。使用は第3部§2のみ（第4部の「監査できる」の傍証に使い回さない）。
- **有形性（ADR-0001）の置き場**: 第3部§2の機械検査の前提として1段落（出力は検証可能なモノ、入力は不定形可。「応急処置」→「暫定対応記録」。理解の外化は清水の XDDP の spec-out と同方向）。独立節にしない。

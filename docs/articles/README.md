## 解く問題（記事全体の頂点・一文）

> **プロジェクトの進め方 — 誰が何を作り、何に依存して完了するか — が暗黙知のままだと、人も AI も、毎回誰かに文脈を説明してもらわないと参加できない。
> プロセスを成果物依存グラフとしてテキストに外化し機械検査に載せれば、文脈ゼロの参加者が図を読むだけで作業を選び、回し、監査できる。**
> — PFDSL はそのための言語と運用。

逆ピラミッド: この頂点（＝第4部の到達点の約束）を第1部の冒頭近くで言い切る。
以降の各部はこの到達点に必要な性質を層ごとに1つずつ実証する。各部末の結び節は要点の機械的な再宣言ではなく、部冒頭の論点への答えを言い切る（R12）。

## 読者

プロセスをコードと同じく**テキスト・版管理・CI に載せたい開発者**（git/diff/レビュー/CI を回し、AI エージェントにも仕事を振り始めた層）。
PFD の前提知識は仮定しない。第1部の入口だけ「依存関係は誰の仕事にもある」と広く入り、以降は開発者の既知概念（git・diff・CI・レビュー・subagent）で内部用語を橋渡しする。

## 第1部 節構成 — 「PFDをテキストDSLにして処理系に載せる」（仮題）

冒頭に頂点の一文を言い切る（連載の道筋段落は数文以内・目次代わり・4部構成）。

1. **痛み2つを並置する**（別の角度の痛みとして明示的に2つ挙げる — R11 レビュー指摘）。
   ①既存のタスク管理は依存関係を第一級で表現できない: ToDo リストはフラットで項目間の前後関係を持たない。
   WBS は包含ツリーで生成の依存を持たない。その弱点は皆了解していて実務ではガントチャートへ変換するが、変換が暗黙でトレーサビリティが確保できず、実行順は複数ありうるのに1パターンへ収束させてしまう。
   DFD はデータの流れであって作業の完了・着手可否を扱わない。
   共通の分析: どの手法も「成果物」と「何から何を作るか（生成依存）」を第一級の要素として持たない。
   ②依存の絵は作図ツールで描けるが、図が腐る: 更新の手間が続かず、実体と乖離し、参照されなくなる。
   ①は表現力の欠如、②は維持の失敗。
2. **成果物依存グラフをテキストで書く** — 成果物を主語にすると「何を生んで完了したか」が検証可能になる（清水吉男の PFD をここで1文定義）。
   最小記法（`A >> P -> B`・集合記法・status frontmatter）。メタデータは status のみ最小導入し、他のフィールドは第3部まで出さない。
3. **着手可能集合の機械導出** — 入力が全て done のプロセス = 今着手できる作業。図と status だけから機械的に導出でき、誰が読んでも同じ答えになる。
   R13: 「実際に機械化されている」例示として `ready --best` の実出力をここに置く（第4部から移設）。図に種別宣言 `type: roadmap` を1行足して渡す（種別の詳細は第4部、と1文で流す）。`*` 推薦＝合流点を解くプロセス、の説明も添える。CLI 出力は実採取。
   事例素材: オープン issue 8件を図に落とすと並列着手9本が即答（2026-06-11）／新提案の受け入れが既存グラフへの1チェーン追加で確定（2026-06-11）。
4. **処理系に載る** — 座標を持たずグラフ構造だけを書くから、変更が diff になりレビューと CI に載る（痛み②への答え）。
   Graphviz は絵を描くだけだが、PFDSL は PFD の意味論を検査する: 単一生成元違反の**対比デモ1件**（NG の図 + 実際の CLI 出力。Graphviz なら描けてしまう）。
   コミット時に構造の壊れが検出される（実体との乖離は検査では捕まらない — 負例参照）。先行例（SWET の PFD Draw Tool）・diagram-as-code は各1文で従属的に触れる。
   負例1件をこの節末尾に畳み込む: テキスト化してもグラフと作業実体（issue 等）のドリフトは残り、同期を保つ監査の手間は消えない。テキスト化はドリフトを消すのでなく、検出可能にする。
5. **書く段階で構造の誤りが見つかる** — 検査より手前で、書くと即座に検査・描画のフィードバックが返ることで、PFD が思考整理の道具になる。AI との協調でこの強みがさらに効く。
   事例素材: 並列起草の直感が図で反証され相互依存が見えた（2026-06-11）／相互依存の先回りモデル化が後続の罠を未然回避（2026-06-20）／境界ケースを図の上でトレースした検証がそのまま11のテストケースになった（2026-06-21。具体1例を必ず展開）。

## 第2部 節構成 — 「グラフに入れた制約、入れなかった機能」（仮題）

1. **論点提示 + 痛み** — 第1部で図は処理系に載った。この部の論点は、その処理系に何を検査させるかをグラフの制約としてどう決めたか（入れたもの・入れなかったもの）。
   痛み: 構造の見落とし（重複した生成元・切れたチェーン・死んだ宣言）は列挙的確認でしか見つからず、列挙的確認は人の認知に向かない。
2. **入れた制約 = 機械検査** — 単一生成元・循環の禁止・孤立プロセスの排除。
   各検査を **NG/Good の実 pfdsl 対 + 実際の CLI 出力 + なぜ問題になるか** の3点セットで示す（R11 レビュー指摘: 実際の PFD が見たい・NG/Good で比較したい）。
   単一生成元は第1部のデモを1文で受けて深掘り（どちらを正とすべきか決まらない・名前の衝突）。循環は着手順序が導出不能になる。孤立プロセスは削除の取り残しが死蔵される。
   前提として有形性（出力は検証可能なモノ・入力は不定形可）を1段落。ルールと機械検査の**二層分業**もここ（設計判断は人・列挙的確認は機械）。
   事例素材: 機械化済み領域からは新規欠陥が出ず指摘が散文層に集中した防波堤（2026-07-06）。
   負例1件をこの節末尾に畳み込む: 粒度の読み違えは検査で防げない。粒度ルールに従って一度は統合したプロセスが、委譲した一括起草が存在しない参照 ID を捏造して初めて「隠れた独立プロセスだった」と分かり、分割し直した。図が事前に正解を出したのでなく、失敗が入力不足を露出させた。
3. **入れなかった機能** — 統一理由は「文脈ゼロの読者（人も機械も）が静的に読み切れる図であることを壊すから」。
   update（スナップショットで代替。**改版3形態と `>>?` をここで図つき導入** — 別 artifact／`>>?` 収束／定常サイクル）／条件分岐（合否・承認は結果報告書・承認記録の成果物化 + `>>?` へ変換。清水 PFD も元々持たない）／サブルーチン（似た Process は横断ラベルで代替）。

## 第3部 節構成 — 「メタデータで図を運用につなぐ」（仮題）

1. **論点提示 + 痛み** — グラフと status だけでは運用に足りない情報（完了条件・実体の所在・実行手段）がある。
   図の傍らに自然言語で書き添えることはできるが、非形式的で機械が処理できない（R11 レビュー指摘）。
   PFDSL は frontmatter の形式フィールドにすることで、人にも AI にも処理可能にする。図を読むだけに完結せず、運用に必要な情報へアクセスし更新するときに効力を発揮するのがこの部の主題。
2. **criteria → 完了判定の資産化** — 完了基準を成果物に事前宣言し、欠落は機械が警告する（実際の CLI 出力つき）。
   後続のセッション・別の担当者が合否を判定できる資産になる。
   事例素材: 警告の対象拡大が自分の図の不備を直撃（2026-06-19）。
   負例1件をこの節末尾に畳み込む: 機械が検査するのは criteria の存在だけで、質は保証しない。形骸化した完了基準は残りうる。
3. **location / command → 図から実体への接続** — 成果物の実体ファイル・issue・実行コマンドへ図から直接アクセスできる。
4. **道具が豊かになる** — メタデータが形式化されているから道具が読める。VSCode 拡張を具体で見せる: ホバーで label・owner・status 等を表示／`location:` がリンクになり図のノードから実体ファイルへクリックで飛べる／編集中のリアルタイム検査／編集に追随するライブプレビュー（SVG）／`.dot`・`.svg` への画像書き出し。
   中心機能は**双方向ジャンプ**: プレビューのノードをクリックするとエディタの該当エッジへ、続けてクリックすると frontmatter 定義へトグルでジャンプ。逆にエディタのカーソル移動はプレビューの該当ノードをフォーカスする。
   グラフィカルエディタの一覧性とテキストエディタの編集力・diff 可能性のいいとこどりであり、第1部の「テキストだから処理系に載る」を手放さずに図の操作性を得ている、という位置づけで書く。
   連載で拡張に触れるのはここだけなので、この節でしっかり展開する（R11 追レビュー）。進行を回す手順書（第4部の主題）への言及は1文に留める。
5. **自由記述の逃がし先** — 読者の疑問「メタデータだけでは横断的な情報が書けないのでは」「frontmatter だけだとつらくないか」に答える節として紙幅を割く（R11 追レビュー。「軽く」を撤回）。
   役割分担を示す: frontmatter は1ノード1値の形式データに限り、ノード横断・図そのもの・図の外との接続に関する記述は図の隣の自由記述ファイルが担う。
   実 companion（`.pfdsl/roadmap.md`）由来の特徴的事例を仮想事例化して2〜3件: ①タイミング規約（issue のクローズと進捗の確定はマージ時に行い、PR 作成時点では行わない — 個々のノードでなく全ノードに横断するルール）／②グラフの編集自体の規則（公開プロセスの入力列挙は実装成果物の追加と同一サイクルで更新する。怠ると公開チェーンから切れる — 図のメンテナンス規則はどのノードの frontmatter にも置けない）／③外部の一次情報との同期手段の宣言（issue 管理のバックエンドと同期監査スクリプトの指定）。

## 第4部 節構成 — 「運用そのものをPFDに書く」（R13 で洗練）

部の軸（一文）: **エージェントが回すループの1周を、決定論部分の機械化と AI の判断の分業でどう設計するか。**

1. **痛み** — エージェントにも新規参加者にも、毎回プロジェクトの文脈を口頭で説明し直す。
   進め方が頭の中にしかない限り参加コストは人数と回数に比例し、AI はセッションが終わると説明を保持しない。
2. **答える問いで分かれる3種類の図** — **roadmap**（何を何の前に作るか・今何に着手できるか）・**workflow**（この作業はどう繰り返されるか・誰が何をトリガーに何を行うか）・**runtime-pipeline**（システムが動くときデータは何に変換されるか）。
   種別は「この図が答える問い」で切る。分類の由来（複数の仮想プロジェクト設計を横断審査して帰納 — ADR-0017）は本文では1文 + 脚注に留める。
3. **図を読んで1サイクル回す運用ハーネス** — リポ非依存の汎用スキルが読んで回す。層設計は平易語で圧縮: 汎用手順は PFD 概念だけから導ける（着手判断＝入力が全 done・進捗＝status 更新・1サイクル1プロセス）／リポ固有の中身は図の隣の説明ファイルに逃がす／issue 管理の流儀は選んで載せるプリセット。
   スキル名（pfd-cycle / pfd-ops / pfd-retro）はここで一度だけ紹介し、以降「運用ハーネス」で受ける。
   1サイクル＝**選択→実行→反映→報告**の4段階。
4. **決定論的な部分は機械が返し、判断だけを AI が担う（R13 の核）** — 1サイクルの各段階に決定論的な支援コマンドが対応することを、段階に沿って示す:
   - 選択＝`ready --best`（着手可能集合と合流点解放の推薦。導出のデモは第1部§3 に移設済みなので1文参照）。ただし一次情報を読んで設計未確定なら実装に入らない判断は AI
   - 進捗＝`status-set`（status 更新と同時に、新たに解放された着手可能プロセスを列挙して返す）
   - 反映＝`check --audit`（終端成果物の列挙 — 手段成果物が終端に残っていないかの門番監査）・`audit-sync`（roadmap と他の flow 図の todo 突合 — 図間のずれ検出）。知見をどの文書に振り分けるかは AI
   - ほか `diff`（構造差分）・`explain`（診断コードの仕様参照）等。
   位置づけ: 第2部の**二層分業**（設計判断は人・列挙的確認は機械）の運用版。「列挙・突合・導出」は毎サイクル発生する定型作業であり、ここを機械に倒すことで AI の文脈を判断だけに使える。
   コマンド名の羅列にしない — 各コマンドは「どの段階の、どの列挙作業を肩代わりするか」とセットで1〜2文ずつ。
5. **結び** — 説明の繰り返しが図と手順書を読むことに置き換わった。1周は文脈ゼロの担い手でも回る。ただし外化された運用は書いた通りにしか実行されない → 第5部予告1文。

## 第5部 節構成 — 「実行と振り返りで運用を直し続ける」（R13 で洗練）

部の軸（一文）: **周回を重ねると、規定の穴が露出し、ループ自体が良くなっていく — この形をコーディングエージェントのループ設計の文脈に位置づける。**

1. **ループ設計という関心（R13: 前提説明を独立節で厚く）** — 対応表より先に、読者がループ設計の議論を知らない前提で説明する:
   コーディングエージェントの分野で、1回ずつ指示を出すことから「停止条件を満たすまで作業サイクルを繰り返す仕組みの設計」へ関心が移った（2026）。
   Anthropic は公式ブログでループを起動の契機と停止条件で分類し（ターンベース・ゴールベース・時間ベース・プロアクティブ）、目標達成判定や周期実行を製品機能として持つ。この実践はコミュニティで loop engineering とも呼ばれる（呼称の出どころは Osmani — 「公式が推奨する loop engineering」とは書かない）。
   設計要素: 何をきっかけに動くか・1周で何をするか・できたことをどう検証するか・いつ止まるか。
   難所は**検証**と**状態**: 完了を機械的に判定できなければ実行者の完了報告を受け入れるしかない（公式解説も決定論的基準を勧める）。周回間の記憶はセッションに残らないから、状態は会話の外（ファイル・git 履歴）に置くしかない（Anthropic のハーネス解説）。
2. **PFDSL はループが読む状態と判定基準を宣言する** — 前節の設計要素に第1〜4部の仕組みを対応づける:
   1周の単位＝着手可能集合から1プロセス／完了判定＝criteria+check／状態の置き場＝git 上のテキスト（各周回は文脈ゼロでよい）／停止条件＝着手可能集合が空。
   位置づけの限定: PFDSL はループの実行機構（周期起動・達成判定モデル・再投入）を持たず、各周回が読む状態と判定基準の宣言に徹する。実行機構はエージェント側の機能が担い、両者は別々に選べる部品。
3. **運用を実行すると規定の穴が見える** — 実行（ops）が規定の穴を露出させ、振り返り（retro）が拾い、規定の改訂か言語検査への要件化で塞ぎ、再実行で確認する閉ループ。
   ループ設計の語彙で受ける: 検証役の分離（生成する側と評価する側を分ける maker-checker）に対応し、周回の失敗を次の周回と将来のループに還流する。
   事例素材: 文脈ゼロエージェントの1サイクル完走と穴2件（issue クローズの timing・チェーン未定義による孤立プロセス）の露出 → 改訂 → 再実行（2026-06-14）／文言で防げない型（孤立プロセス）が言語検査の要件へ（2026-06-19）。
   負例1件をこの節末尾に畳み込む: 外化してもルール文言に書かれない意図（暗黙スコープ）は残る。定期監査（retro）が「あるべき記述の不在」として特定し、文言が修正される — これ自体がループの一周でもある。
4. **リポジトリを替えてもループは回る** — 外部リポへの初適用（2026-07-03）を運用ハーネスのリポ非依存性の実証として書く: `criteria` 検査が完了条件を後続セッションの資産にし、スコープ変更2件を完了基準の1行編集 + チェーン1本の追記で吸収した。
5. **結び（連載の結び）** — 各部で外化したものを並べ、冒頭の頂点の一文がこの積み重ねの上に成り立つことを述べる。物語語（回収等）は使わない。
   出典（脚注）: Getting started with loops（公式）・gihyo 紹介記事・Osmani "Loop Engineering"・Effective harnesses for long-running agents（状態外部化）。

## 書き方の規律

- **痛み（問題）→ 解決**。各部・各節は「次に解く痛み・小問題」から淡々と始める。前部を引き継ぐ宣言（「ここまでが…だが」）や本文中の節での次部予告（「第N部で詳述」）を書かない。ただし結び節末尾の1文予告は可（R12）。
- **物語的メタ装置を使わない**: 「橋渡し」「コーダ」「蝶番」「結」「回収する」「連載を閉じる」式の構造語を本文に持ち込まない。否定形での自己言及も書かない。
- **各部の冒頭1段落で論点を提示する**（R11）: この部が何の問題にどうアプローチしたかを先に言う。論点提示のために前部の結論へ1文だけ言及するのは可（それ以上の引き継ぎ宣言は従来どおり禁止）。
- **検査の説明は NG/Good の実 pfdsl 対 + 実際の CLI 出力 + なぜ問題になるか、の3点セット**（R11）: CLI 出力の捏造は禁止 — 必ず実行して採取する。
- **免責は各部の負例1件のみ**（それ以外の自己批判・方法論的懺悔は本文に書かない）:
  各部に負例を**1件だけ**（第1部: ドリフト同期監査／第2部: 粒度の読み違え／第3部: criteria の形骸化／第4部: 暗黙スコープ）。読者が同じ道具で実際に踏む落とし穴に限る。
  独立の「払うコスト」末尾節は立てず、関連する節の末尾に畳み込む（2026-07-08）。
- **言葉を擬人化しない・気取った比喩を使わない**（2026-07-08 ユーザーレビュー）: 「構造を暴く」「焼き付ける」「文脈の口移し」「穴を踏み抜く」「言い当てる」「軌を一にする」「同じ方向を先に歩いていた」のような表現は、平易な動詞・普通の言い方に開く。無生物主語が動作する文は、人・エージェントを主語にするか受け身でない平叙に直す。
  見出しも対象: 「criteriaが完了判定を資産にする」型は「criteriaで〜」の手段格に直す（2026-07-08）。
- **見出し**: H1 は「タイトル（PFDSL 連載 第N部）」の括弧形式。地の文・見出しにダッシュ（—）を使わない・並列の中黒を使わない等、整形は japanese-tech-writing に従う。
- **記号・略号を本文に出さない**: `ADR-NNNN`/`V001`/`W002`/`V020`/`L1〜L4`/`cold subagent`/`companion` 等は平易語に開き、出典は脚注へ。
- **dogfooding という語を使わない**: 「著者が自リポジトリで」等の平易語で受ける（未定義語を増やさない）。
- **表は最小**: 短い対比（3項目以上・各セル1行）だけ表。物語・事件・反実仮想は散文。本 README の整理用の表は本文に出さない。
- **用語の初出マップ**: PFD の1文定義・最小記法・status＝第1部§2／着手可能集合＝第1部§3／`check`（動作として）・diff・CI＝第1部§4／単一生成元・循環・孤立プロセス・有形性・二層分業＝第2部§2／`>>?`・改版3形態＝第2部§3（図つき）／criteria＝第3部§2／location・command＝第3部§3／roadmap・workflow・runtime-pipeline（種別）・運用ハーネス＝第4部§2。

## 執筆者メモ（本文に出さない・self-check 用）

provenance 事故・altitude 混同を防ぐための控え。本文には出さず、書くときの照合に使う。

- **清水軸（keep/tighten/drop/add）×扱い軸（課す/採らない）は直交**。継承（keep）を機械検査として課す（単一生成元）も、継承した不在を継続して採らない（条件分岐）もある。
  keep: 有形性(0001)・単一生成元(V001)・DAG・条件分岐の不在(0005)／tighten: 粒度4基準(0004)／drop: update(0003、清水軸で唯一の意味論変更)／add: テキスト DSL 化・tooling・`tags`(0019)・終端門番の運用側(0018)・最小コア＋拡張(0021)。
- **provenance スコープ（vs 清水 PFD）**: 清水の意味論に対し意図的に変更（drop）したのは update 不採用ただ一つ。他は継承（keep）か追加（add）。
  「追加」概念が PFDSL 内部で版を重ねて意味論を変える（`tags` の 0019 など）のは**内部版進化**で別軸。「PFDSL は自分の意味論を一度も変えていない」と誤読しないこと。
- **二層と二段は別レベル**: 第2部§2の「ルール対 機械検査」の二層（ADR-0006）と、「言語の `check` 対 運用側の点検」の二段（ADR-0018）は別で、後者は前者の機械検査の内訳。
  運用側の点検（終端ゲート・定期監査）は第4部§3〜4で追加的に出す。第2部の機械検査が不完全だったと読ませない — 網を運用面へ広げる話であって覆す話ではない（誤読防止の接続を1文）。
- **採否**: ADR-0008（group 指針）は本連載から外す。ADR-0011（能力成果物の世代還流）は第2部で1文触れる程度。
  `schedule:` 全廃棄（ADR-0021）は第2部§3で任意・1文まで（実装を一周してから判定基準が効いた、の一例。主題化しない）。

# PFDSL 連載記事

PFDSL の問題意識と設計根拠を説明する公開技術記事（連載・日本語）。issue
[#12](https://github.com/takasek/pfdsl/issues/12) に対応し、リポの生態系図上は `write_article` の出力 `i12_article`。
この README は連載の**構成プラン（設計書）**。本文はこの後 `docs/articles/NN-slug.md` に書く。

**書き方の基本方針（round 5 で確定）**: 序・hook で釣らない。伏線/回収/「○○へ橋渡し」式の物語的レトリックを使わない。
**素直な技術記事として「問題 → 解決」で書く**。免責・N=1 開示・反実仮想の限界・自己批判の連打は誠実さでなく言い訳/ノイズとして削ぐ
（残すのは下記2点のみ）。読者の現場への適用例（「あなたのリポジトリで」）は**別記事**で本連載のスコープ外。

## 解く問題（記事全体の頂点・一文）

> **プロセス（誰が何を作り、何に依存して完了するか）を、コードと同じく diff・レビュー・CI に載せて機械に守らせたいのに、
> それができる記法が無い。** — PFDSL はこの問題への解。

逆ピラミッド: この一文を第1部の冒頭近くで言い切り、以降はその実証として淡々と並べる。各部末で要点を再宣言しない。

## 読者

プロセスをコードと同じく**テキスト・版管理・CI に載せたい開発者**（git/diff/レビュー/CI を回す層）。
PFD の前提知識は仮定しない。最初の節だけ「依存関係は誰の仕事にもある」と広く入り、以降は開発者の既知概念
（git・diff・CI・レビュー）で内部用語を橋渡しする。

  次部への予告（「第N部で詳述」）を書かない——技術記事の節は問いで始めれば接続は自明。
- **物語的メタ装置を使わない**: 「橋渡し」「コーダ」「蝶番」「結」「回収する」「連載を閉じる」式の構造語を本文に持ち込まない。否定形での自己言及（「これは伏線ではない」「回収ではない」式）も書かない。
- **免責は最小2点のみ**（下記以外の自己批判・方法論的懺悔は本文に書かない）:
  1. 第3部冒頭に N=1 を一文（「自リポ1個を数週間運用した自己観察で、判定は著者による。効果測定でなく『この条件でこう効いた』の記録」）。
     第4部冒頭にも「これも同じ自リポの自己観察」と射程を一句引き継ぐ（再宣言でなく引き継ぎ。別ファイルゆえ第3部の一文は届かない）。

## 第1部 節構成 — 「成果物を、テキストで」（解の提示）

冒頭に連載の道筋を1段落（hook でなく目次代わり・数文以内、長くしない）。続いて解く問題（上の一文）を言い切る。

1. **なぜ成果物で考えるか** — 書き捨てでないプロジェクトには依存関係が生じる（誰の仕事にもある＝広い入口）。
   成果物を明示すれば「何を生んで完了したか」が検証可能になる——これが清水吉男の PFD の核（PFD をここで1文定義）。
2. **既存の記法に足りないもの**（§2・§3 を統合・1メッセージ）— **収束文を節頭に先出し**（ここは結論先出しが効く）:
   「既存の依存記法は、テキストで diff できるか／PFD の整合を機械検査できるか のどちらかを必ず欠く。」 以下はその実証。
   依存管理の代表 WBS は包含ツリーで*依存グラフではない*（生成エッジを持たない）。作図ツールは絵を描くが diff・機械検査を持たない。
   フローチャート・DFD・先行例（SWET の PFD Draw Tool）・diagram-as-code は**各1文で従属的に**触れる（比較を主題化しない）。
3. **PFDSL の最小例と `check`** — README の最小例で記法を導入し、`check` で単一生成元検査を**動作で**見せる。
   「絵を出すだけなら Graphviz でできる。PFDSL が足したのは絵でなく、PFD 固有の制約を構文として検査する層だ」を、
   Graphviz では出ないエラーが PFDSL では出る**対比デモ**として実演内に示す。
4. **満たすべき3条件と命名** — 実物を見せた後にここで名付ける。
   **① 成果物とその生成依存エッジの明示**（単一生成元のもとで）— 完了が検証可能。
   **② PFD 意味論の機械検査** — 単一生成元・DAG 性・終端健全性という PFD にしかない整合条件を機械判定する（テキストで diff/CI に
   載ること一般でなく「何を検査するか」が固有）。
   **③ テキスト性** — 座標を持たずグラフ構造だけを編集し diff・レビュー・git に載る。
   要点（記事全体で一度だけ強く言い切る）: **①②③が揃うと、完了が検証でき、レビューでき、文脈を持たない参加者でも加われる。**

## 第2部 節構成 — 「何を課し、何を入れないか」（解の設計判断）

「課す/採らない」は ADR の整理軸であって構成の主役にしない。本文は散文で、判定基準（DAG の静的解析性を壊さないか／機械検査できるか／
pfdsl 単体に価値があるか）を一度示し、各論を1段落ずつ語る。A群/B群の表・直交2軸表は本文に出さない（→執筆者メモ）。

1. **機械検査で課す性質**（各1段落の散文）— 成果物の有形性（出力は検証可能なモノ、入力は不定形可。「応急処置」→「暫定対応記録」。
   理解の外化は清水の XDDP の spec-out と同方向）／単一生成元（生成元の一意性が diff・改版・責任の基準点）／終端健全性（消費者なし終端・
   孤立プロセスを検査で検出）／プロセス粒度（時間的凝集禁止・新依存なし分割禁止・相互依存は1プロセス・所有権境界。マイクロサービス同型）。
2. **あえて入れない機能**（各1段落）— update（DB・本番で欲しくなるがスナップショット＋ビューで代替。循環を生み DAG 性・単一生成元・diff
   安定性を壊す）／条件分岐（合否・承認は承認記録・結果報告書＋`>>?` へ変換。静的解析性を壊し状態機械と重複。清水 PFD も元々持たない）／
   subroutine（似た Process は横断ラベル `tags` で代替。1:1 leveling から逸脱）。理由を「静的解析性と最小性を壊すから」で統一。
3. **改版の表現** — update を採らず単一生成元も守る前提で改版をどう書くか。3形態（別 artifact／`>>?` 収束／定常サイクル）。
   外部ツール専用フィールドを core に入れない（`schedule:` 等。「open な拡張と minimal な core の両立」）もここに1段落で吸収。
4. **設計判断の限界（1文で締める）** — ルール（人の設計判断）だけでは列挙的・網羅的な確認を要するミスは防げず、機械検査との二層が要る
   （自分でこの言語を使ってみたところ base_code 入力欠落・未消費成果物の放置が再発した。少数事例ゆえ示唆。未定義の dogfooding 語は使わない）。
   ※この「ルール対 機械検査」の二層と、第4部の「`check` 対 運用点検」の二段は別レベル＝後者は前者の機械検査の内訳。本文で同一視しない。

## 第3部 節構成 — 「図を、設計とテストとレビューに使う」（解を使う・言語）

冒頭に N=1 注記を一文（上記）。dogfooding を1文定義。図が3つの装置として働いた、を素直に並べる。

1. **図が設計判断に効いた** — 着手可能集合の機械導出（入力が全 done のプロセス＝並列着手リストを即答）／新提案の受け入れ
   （1チェーン追加で並列性・接点・合流点が確定）／スキーマ分岐（location 単数か辞書かを原則だけで3分割即決）。
2. **図がテストに化けた** — 仕様の境界ケースを具体例にトレースした検証ラウンドが、そのまま実装のテストケースになった
   （boundary 検証の具体例が 11 テストケースに）。仕様テキストだけでは非自明ケースを事前網羅できなかった。
3. **図がレビューを可能にした** — 「並列で起草できる」という直感が図で反証され、並列→上流方針→共同起草と収束（相互依存の発見）。
   roadmap が相互依存を1プロセス複数出力で畳んでいたため後続着手者が罠を踏まずに済んだ。
4. **払うコスト（負例1件）** — 機械検査はタダではない。`.pfdsl` を1つ編集するたびに snapshot 更新と鮮度検査が走り、フィールドを
   足せば fixture を二重更新する義務が増える。検査可能性は、検査対象を最新に保つ手作業コストと引き換えに得ている。

## 第4部 節構成 — 「文脈を知らない参加者に運用を任せる」（解を使う・運用）

運用記述もテキスト＋機械検査に載せた話。冒頭に「これも同じ自リポの自己観察」の一句（N=1 の射程引き継ぎ）。cold subagent の事件は**連載でここだけ**に置く。素っ気なく終える（要点を再び叫ばない）。

1. **運用もテキスト＋機械検査に載せる** — pfd-cycle / pfd-ops / pfd-retro は PFDSL の上に足した運用層（言語仕様ではない、と1文）。
   運用の判断材料（着手可能な成果物・依存・状態）が記述側に出る。
2. **文脈ゼロの参加者が着手判断できた** — 文脈を持たない参加者に運用を任せたら、人間が見落としていた設計の穴を2件
   （issue クローズの timing・チェーン未定義による孤立プロセス）あぶり出し、直して再実行したら消えた。文脈ゼロの AI エージェントは
   この「暗黙文脈なしに加われる」を最も分かりやすく示す一例（主役でなく検証手段）。
3. **失敗が構造的欠落として現れる** — 3本の PR が続けて追記を漏らした。文章なら「また忘れた」で終わるが、PFDSL では追記漏れは
   「あるはずのエッジがグラフに無い」という同じ形をとり、同型の1パターンとして1回の監査でまとめて挙がる。この形の失敗を構造的欠落と呼ぶ。
4. **運用ルールを機械検査へ格上げ** — 孤立プロセスの混入は最初は運用側の点検が拾っていた。人が回すルールゆえ回す人が代われば素通りしうる。
   そこで同じ点検を言語層の `check` に移し機械検出にした。検査には二段ある——言語の `check` が見る欠落と、運用側が点検する欠落だ。
   **本文に必ず誤読防止の接続を1文置く**: これは第1部②（機械検査）を覆すのでなく、終端健全性の一部（手段成果物が後続を欠く欠落）を
   当初は運用側の人手点検に委ねており、その一部を `check` 側へ移した話だ。
5. **払うコスト（負例1件）** — `schedule:` フィールドを checker・サンプル・テスト・snapshot まで作り込んでから「pfdsl 単体に価値を
   与えない」と全廃棄した。設計の判定基準を持っていても、効いたのは実装を一周した後だった。道具があっても設計判断ミスは事前には防げない。

## 執筆者メモ（本文に出さない・self-check 用）

provenance 事故・altitude 混同を防ぐための控え。本文には出さず、書くときの照合に使う。

- **清水軸（keep/tighten/drop/add）×扱い軸（A 課す/B 採らない/コーダ）は直交**。継承（keep）を機械検査として課す（単一生成元＝A）も、
  継承した不在を継続して採らない（条件分岐＝B）もある。A群/B群に keep が混じるのは矛盾でなく二軸が直交するため。
  keep: 有形性(0001)・単一生成元(V001)・DAG・条件分岐の不在(0005)／tighten: 粒度4基準(0004)／drop: update(0003、清水軸で唯一の意味論変更)／
  add: テキスト DSL 化・tooling・`tags`(0019)・終端門番の運用側(0018)・最小コア＋拡張(0021)。
- **provenance スコープ（vs 清水 PFD）**: 清水の意味論に対し意図的に変更（drop）したのは update 不採用ただ一つ。他は継承（keep）か追加（add）。
  「追加」概念が PFDSL 内部で版を重ねて意味論を変える（`tags` を Artifact 専用→両種別に広げた 0019 など）のは**内部版進化**で別軸。
  「PFDSL は自分の意味論を一度も変えていない」と誤読しないこと。
- **検査の二段（言語層／運用側）**: 言語の `check`（単一生成元・DAG・終端健全性／`--audit`・V020）と、運用側の点検（手段成果物が後続を欠く
  欠落＝ADR-0018(b)・pfd-ops 終端ゲート）。第1部②は言語の機械検査として提示し、運用側の点検と「運用側で見つけ言語側へ昇格させる」運動は
  第4部で**追加的に**出す（第1部に前借りの予告を書かない／第1部②が不完全だったと読ませない＝加える話であって覆す話ではない）。
  なお第2部の「ルール対 機械検査」の二層（ADR-0006）と、この「`check` 対 運用点検」の二段（ADR-0018）は別レベル＝後者は前者の内訳。本文で包含関係を一言示す。
- **採否**: ADR-0008（group 指針）は本連載から外す。ADR-0011（能力成果物の世代還流 `>>?`）は第3部で1文触れる程度（自己適用が図に
  現れる一例。「自己適用そのもの」と等号で結ばない）。`/code-review` の PR スコープ外バグ検出は PFD ハーネスの dogfood でないため中核に据えない。

# PFDSL 連載記事

ADR 群を素材に、PFDSL の問題意識と設計根拠を説明する公開記事（連載）。
issue [#12](https://github.com/takasek/pfdsl/issues/12) に対応し、リポの生態系図上は
`write_article` プロセスの出力 `i12_article` にあたる。

この README は連載の**方針と構成のインデックス**であり、執筆の前提を一箇所に固定するための文書。
各部の本文を書く前に、ここで軸・読者・NG・構成を確定させてから書く。
**この README は設計書であり、ここにある表・記号・メタ注記の多くは「執筆者の self-check」であって本文へそのまま
持ち込むものではない**（→「執筆規律」「執筆者メモ」を参照）。

## 読者

主たる読者は **プロセスをコードと同じくテキスト・版管理・CI に載せたい開発者**（git/diff/レビュー/CI を日常的に回す層）。
PFD の前提知識は仮定しない。第1部の入口（最初の節）だけは「依存関係は誰の仕事にもある」と広く構えて共感の入口とするが、
以降は開発者の既知概念（git・diff・CI・レビュー）で内部用語を橋渡しする。
これにより「前提知識なし」と「内輪用語」の矛盾を、「開発者の既知概念で内部用語を定義する」という解ける問題にする。

## 軸（書くこと）

- **そもそもツール自体がない状態での問題**。
  プロセスを形式化したいのに、それを支え・制約を守らせる手段が存在しない、という状況そのものの問題。
- その**解としての PFDSL**。
- **制約をツールで管理することは重要な要求**。
  形式化は規約・人手レビューだけでは形骸化する。
  単一生成元・DAG 性・終端健全性などを機械が検査できる必要がある。

## 書かないこと（NG）

- 社内・SWET の GUI ツールとの差分・比較を**主題にしない**。要求の実在を示す先行例として一度だけ触れる。
- 「作図ツールの限界」を中核に据えない。差別化は独立した比較章を作らず、既存の論証ライン
  （WBS→PFD、`check` 実演）に1段落ずつ寄生させ、優劣を煽らず PFD/PFDSL へ収束させる。
- AI 臭いポエム調・過剰な比喩・情緒的レトリックを使わない。**淡々と具体的に書く**。
- 伏線回収・自己弁護のレトリック（「回収する」「連載を閉じる」「これは飾りではない」「比喩はしない」式）を本文に置かない。
  結論は淡々と事実として述べる。

## 執筆規律（本文化の規則・敵対レビュー反映）

連載を「正しい設計書」から「宣言した読者が完走する文章」に変えるための規則。

- **hook を冒頭に**: 連載は第0部（序）で具体的な事件から入る（下記）。各部冒頭にもミニ hook を1つ置いてから理論に入る。
- **表は最大「1部1表」**: 対比が3項目以上 かつ 各セルが1行で収まる場合のみ表。**1セルに `／` で複数エピソードを詰めない。
  事件・反実仮想・物語は必ず地の文（散文）で書く**。本 README の C群/D群 表は執筆者の整理用——本文では散文に展開する。
- **記号・略号を本文に出さない**: `ADR-NNNN`／`V001`／`W002`／`V020`／`②-a`／`②-b`／`L3`／`cold subagent`／`companion` 等は
  本文では平易語に開く（出典は脚注へ）。概念は散文で説明し、記号は読者に渡さない。
- **メタ装置を本文に出さない**: 「結」「橋渡し」「採否メモ」「provenance 注記」「再分類表」は執筆者の self-check。
  本文には結果だけを地の文で出す。
- **用語オンボーディング順序**（初出定義の位置）:
  - PFD … 第1部 入口で1文定義（「成果物の依存を1枚の図にする清水吉男の方法論」）
  - 単一生成元／DAG／終端健全性／`check` … 第1部の `check` 実演で動作とともに初出（抽象語でなく動きで定義）
  - 正準の3要求（①②③）… 第1部 末尾で命名（実物を見せてから名付ける）
  - `>>?` … 第2部 B（条件分岐の変換）で図つき初出。第1部で先出ししない
  - dogfooding … 第3部 冒頭で1文定義
  - 運用ハーネス／pfd-cycle・ops・retro／検査の二層（言語層 vs 運用層）… 第4部で初出。第1部では出さない

## 連載の構成（暫定・柔軟）

本数は固定せず、序＋4部で進める。

| 部 | 状態 | 責務 / テーゼ |
|---|---|---|
| 第0部（序）| 確定 | 事件から入る hook。文脈ゼロの参加者（AI）が設計の穴をあぶり出した事件→なぜ可能か（テキスト＋機械検査）→連載の地図。3段落・約400字。 |
| 第1部 | **確定** | ツールが無い問題 → 解としての PFDSL → そこから正準の3要求（成果物の明示・PFD 意味論の機械検査・テキスト性）を言語化。5節。 |
| 第2部 | **確定** | 制約の設計を「課す/採らない」で二分（A群＝課す／B群＝採らない）。清水軸（keep/tighten/drop/add）とは直交（詳細は執筆者メモ）。 |
| 第3部 | **確定** | 言語そのものの dogfooding。図が設計判断器・テスト生成器・レビュー装置として働いた自己観察（N=1）。効かなかった局面も書く。 |
| 第4部 | **確定** | 運用ハーネス（pfd-cycle / ops / retro）の dogfooding。テキスト化＋機械検査で運用が暗黙文脈・気付きから独立する。1記事に保つ。 |

## 第0部（序）節構成（確定）

仮タイトル: **「設計の穴を、文脈ゼロの参加者があぶり出した」**（後で詰める）

連載全体の hook。約3段落・400字以内。事件→理由→地図の順。

1. **事件** — 文脈を持たない参加者（ここでは文脈ゼロの AI エージェント）に自分のプロジェクト運用を任せたら、
   人間が見落としていた設計の穴を2件あぶり出した。直してもう一度やらせたら、穴は消えていた（第4部 D1 の平易化）。
2. **なぜ可能だったか（一文）** — プロジェクトの成果物・依存・状態が全部テキストで書いてあり、機械がそれを検査できたから。
   だから文脈を持たない参加者でも加われた。**AI はこの「文脈ゼロでも参加できる」を最も分かりやすく示す一例**であって、
   記事の主役ではない（AI を前面に出しすぎない・第4部まで深掘りしない）。
3. **連載の地図** — その「テキストで機械検査できるプロセス記述（PFDSL）」をゼロから作った話をする。PFD も DSL も知らなくていい。
   第1部で道具を作り、第2部で設計判断を見せ、第3・4部で自分自身に使った結果（効いた所と効かなかった所）を見せる。

## 第1部 節構成（確定・5節）

仮タイトル: **「成果物を、テキストで」**（後で詰める）

実物（道具）を見せてから抽象（3要求）に名前を付ける順にする。差別化は独立節を作らず本文に1段落ずつ織り込む。

1. **なぜ「成果物」で考えるのか** — 書き捨てでないプロジェクトには依存関係が生じる（誰の仕事にもある＝広い入口）。
   成果物を明示すれば「何を生んで完了したか」が検証可能になる——これが清水吉男の PFD の核（PFD をここで1文定義）。
2. **既存の地図に足りないもの** — 依存管理の代表は WBS。WBS は PMBOK 定義では成果物指向だが*包含ツリー*であって
   *依存グラフではない*（「どの成果物がどのプロセスからどの成果物を生むか」の生成エッジを持たない）。
   フローチャート（順序は分かるが関連が希薄）と DFD（状態更新を許し循環しうる）は**各1文で従属的に**触れるに留める
   （独立段落にしない）。
3. **制約は人手では守れない／テキストでないと回らない** — PFD の整合維持は人手では労力大。SWET の PFD Draw Tool は
   単一生成元・二部性を「そもそも書けないようにする」ことで制約をツール強制した（要求の実在を示す先行例として一度だけ）。
   だが手動編集前提でテキスト表現を持たず、diff・レビュー・版管理（git）・AI への受け渡しができない。
   見出しは「ツール不在」と誇張せず「テキスト化・版管理・AI 受け渡しができる PFD 実装が無かった」と正直に書く。
4. **PFDSL — 最小例と `check`** — README の最小例で記法を導入し、`check` で単一生成元検査を実演（用語は動作で初出定義）。
   「絵を出すだけなら Graphviz でできる。PFDSL が足したのは絵でなく、PFD 固有の制約を構文として検査する層だ」を
   実演の中に1段落で織り込む（独立節 §7.5 は作らない）。
5. **正準の3要求とテーゼ** — 実物を見せた後にここで命名する。
   **① 成果物とその生成依存エッジの明示**（単一生成元のもとで）— 完了が検証可能。
   **② PFD 意味論の機械検査** — テキストで diff でき CI に載る記述は珍しくない。固有なのは検査できること一般でなく
   **何を**検査するか——単一生成元・DAG 性・終端健全性という PFD にしかない整合条件を機械判定する点（一般 diagram-as-code に溶けない）。
   **③ テキスト性** — 座標を持たずグラフ構造だけを編集し diff・レビュー・git・AI 受け渡しを可能にする。
   テーゼ: ①②③が揃って初めて「検証可能・レビュー可能・AI 共有可能」なプロセス記述になる。後の各部はこの3要求に繰り返し立ち返る。
   （②の「機械検査」が実は言語層と運用層の二段である、という精緻化は第4部 D3 で事件として導入する。第1部では②を一つにまとめ、
   読者にまだ存在を知らせていない運用層を前借りさせない。）

## 第2部 節構成（確定）— 制約の設計を「課す/採らない」で二分

**背骨テーゼ**: PFDSL は2方向で制約を選ぶ。(A) グラフが満たすべき性質を機械検査可能な形で**課す**。
(B) 静的解析性と最小性を壊す機能は**採らない**。判定基準は「DAG の静的解析性を壊さないか」「機械が検査できるか」
「pfdsl 単体に価値があるか」。各 ADR はこの基準を別々の誘惑に適用した記録。

各論点は本文では「清水 PFD に対し残した/締めた/捨てた/足した（keep/tighten/drop/add）」で軽く触れる。
ただし詳細な分類（清水軸 × 課す/採らない軸の直交2軸表）は執筆者メモに置き、本文には出さない。

**A群 — 課す制約（言語層でグラフを検証可能にする）**（本文では表のまま：短い対比4項目）

| 節 | 設計問題 → 決定 | 裏づけ・例 | 出典(脚注) |
|---|---|---|---|
| A1 成果物の有形性 | 「理解」「合意」のような不定形を出力に許すか → 非対称規則（出力は検証可能なモノ／入力は不定形可） | 完了判定・status・引き継ぎが成立する条件。「応急処置」→「暫定対応記録」。理解の外化は清水自身の XDDP の spec-out と同方向 | ADR-0001 |
| A2 単一生成元 | 同一成果物を複数プロセスが生成してよいか → 禁止 | 生成元の一意性が diff・改版・責任の基準点 | V001 |
| A3 終端健全性 | 出力ノードが宙に浮く → 消費者なし終端／孤立プロセスを検査で検出 | 形骸化の構造的検出。後続門番（手段成果物が後続を欠く）は運用側の話ゆえ第4部へ送る | ADR-0018 |
| A4 プロセス粒度 | どこまで分割するか → 4基準（時間的凝集禁止／新依存なし分割禁止／相互依存は1プロセス／所有権境界） | 偽依存と隠れ循環の排除。マイクロサービス同型 | ADR-0004 |

**B群 — 採らない選択（最小性と静的解析性を守る）**（本文では**散文**。B1〜B3 は3件なので表にせず1段ずつ）

- **update 意味論（drop）** — DB・本番・サービング中モデルで欲しくなるが採らない。スナップショット＋ビューで代替。
  update は循環を生み DAG 性・単一生成元・diff 安定性を壊すため（ADR-0003）。**清水 PFD にあった意味論を意図的に落とした唯一の例**。
- **条件分岐（継続して採らない）** — 合否・承認・閾値で欲しくなるが採らない。承認記録・結果報告書＋`>>?` へ変換して成果物定義で解消。
  分岐は静的解析性を壊し状態機械と重複（ADR-0005）。清水 PFD も元々持たず、PFDSL はその不在を継続し変換パターンを規定した。
- **subroutine（却下）** — 似た Process の再利用で欲しくなるが却下、横断ラベル `tags` で代替。
  1:1 leveling から逸脱し図の現実地図性を損なう（ADR-0019）。**却下の動機は清水 DFD の 1:1 leveling を守ること**。

**橋渡し — 改版の表現（0002）**: 「update を採らない」と「単一生成元」を両立させると改版はどう書くか。3形態
（別 artifact／`>>?` 収束／定常サイクル）。A と B の制約から演繹される帰結として、B 群の後・結の直前に置く。

**コーダ — 最小コアと開かれた拡張（0021）**: B1〜B3 が「意味論を採らない」のと種類が違う不採用。メタデータ拡張自体は
開く（未宣言タグ・フィールドは許容）が、意味づけが完全に外部ツール依存でグラフ意味論/可視化/`check` に寄与しない
フィールド（例: `schedule:`）は core に彫り込まない。「open と minimal の両立」を短く添える。

**蝶番 — ルールは実践で限界を露呈した（ADR-0006 の物語）**: ここまでが「ルール（設計判断）」の話。
だが品質ガイド整備後の実践で、*ガイドに明記済みのミスが再発した*——修正開発プロセスに `base_code` 入力が無い
（誤りはガイドの例文そのものにあった）、消費されない成果物の放置が2例。根拠は少数事例（計3件）ゆえ一般法則でなく
**示唆**として扱うが、教訓は明瞭——列挙的・網羅的な確認を要するミスはルール提示だけでは防げない（ルールの質でなく認知の限界）。
そこで品質担保を**ルール（判断）＋ lint（機械検査）の二層**に分け、2ラウンドの比較で lint バックログと優先度を方向づけた。
方法論が自分自身に適用されて検証された具体例——これが第3・4部が dogfooding の話になる理由。

**結（本文では淡々と）**: 課す/採らないは同じ基準の表裏。ルールの限界（ADR-0006）が tooling と自己適用を要請する——第3部へ。

## 第3部 節構成（確定）— 言語そのものの dogfooding（pfdsl）

仮タイトル候補: **「設計図を仕様のテストに変えた」** / 「PFDSL で PFDSL を作る」 / 「自分の図を、自分の開発で使ってみた」（後で詰める）

冒頭にミニ hook（例: 「『並列で起草できる』と思った2本の仕様。図に描いた瞬間、それが嘘だと分かった」＝C3 の事件）。

**背骨テーゼ（本文では観測の記述として淡々と）**: PFDSL を PFDSL 自身の開発で使ったら、図が3つの装置として働いた——
設計判断器・テスト生成器・レビュー装置。第1部の3要求（明示・機械検査・テキスト性）が揃っていたことで説明がつく。

**誠実な前置き（地の文・防御的にしない）**:
- N=1 開示（文言）: 「以下は PFDSL リポジトリ1個を 2026年6月の数週間にわたり自分で運用した自己観察であり、判定はすべて
  著者による。一般化された効果測定ではなく、『この条件下でこう効いた／効かなかった』の記録として読んでほしい。」
- 反実仮想テストとその限界（文言）: 「これらの局面は1つのテストを通している——『依存グラフが無ければこの判断は違ったか。
  違いを1行で引用できるか』（ADR-0014）。実際にこのテストで落として捨てた草稿もある（依存登録が lint 要件を完全指定したと
  いう草稿は、issue 本文だけで同じ実装に到達できたため却下）。ただし限界も明記する——Yes/No を判定するのは著者本人の内省で、
  第三者の反証を経ていない。自分で反例と気づけた1件は落とせたが、気づけなかった確証バイアスはこのテストを素通りする。」
- 3装置の事後整理（文言）: 「設計判断器・テスト生成器・レビュー装置という3分類は、観測したら3つだった、ではない。
  第1部の3要求に対応させて事後に整理した枠で、効果局面ログには3つに収まらない記録もある（複数のレビュー指摘が1つの整合解に
  収束した『代数的レビュー』、統合と再分離を往復した粒度判断など）。きれいに1対1に見えるのは、そう対応づけられる事例を選んで
  並べたからである。」

**3つの効能（本文では各1〜2段落の散文。下表は執筆者の整理用・本文に表で出さない）**:

| 装置 | 実エピソード | 反実仮想 |
|---|---|---|
| C1 設計判断器 | 着手可能集合の機械導出（並列着手リストを9本即答）／新提案の受け入れコスト低減（1チェーン追加で並列性・接点・合流点が確定）／location 単数か辞書かを原則だけで3分割即決 | 文章ロードマップなら毎回頭の中で依存と並列性をやり直す |
| C2 テスト生成器 | ADR-0020 の boundary-validation-log の具体例が 11 テストケースにそのまま変換 | 仕様テキストだけでは非自明ケースを事前網羅できなかった |
| C3 レビュー装置 | 暗黙依存のレビュー可能化（「並列起草できる」直感が図で反証され並列→上流方針→共同起草と収束）／相互依存の先回りモデル化（後続着手者が「並列主張」の罠を踏まずに済む）／自リポの criteria 欠落を即時警告 | 直感の依存主張がレビュー対象にならず、起草が衝突して初めて相互依存に気付く |

**効かなかった局面（実在の負例・C 群の散文に溶かす。独立見出しにしない）**:
- *粒度の手戻り*: 知見外化プロセスを粒度ルールに従って一度統合したが、委譲した一括起草が存在しない ID を捏造して初めて
  「隠れた独立プロセスだった」と分かり、再分離した（ADR-0009 の4段階変遷）。図が事前に正解を出したのではなく、失敗が入力不足を露出させた。
- *機械検査の維持コスト*: `.pfdsl` を1つ編集するたびに snapshot 更新と鮮度ゲートが走り、フィールドを足せば fixture を二重更新する
  義務が増える。検査可能性は、検査対象を最新に保つ手作業コストと引き換えに得ている。

**結（本文では淡々と）**: 効いた局面の共通構造は、第1部の3要求が揃っていたことで説明がつく。
能力成果物の世代還流（ADR-0011）はその一例として触れるに留める。第4部（運用ハーネスの dogfooding）へ。

## 第4部 節構成（確定）— 運用ハーネスの dogfooding（pfd-cycle / pfd-ops / pfd-retro）

仮タイトル候補: **「文脈を知らない参加者に運用を任せる」** / 「書き忘れを、構造で止める」（後で詰める。AI を表に出す案「設計の穴を、AI が2件あぶり出した」は序の hook と重複するので避ける）

冒頭にミニ hook（序で出した事件の運用面を受け直す）。

**背骨テーゼ（buzzword 確定文言・結論を前借りせず観測として）**: 第1〜3部までで PFDSL を「言語」として説明した。
第4部は、その言語の上に運用ハーネス（pfd-cycle / pfd-ops / pfd-retro）を載せて運用そのものを回したときに観測したことを扱う。
運用層をテキストに書き機械検査にかけると、運用の判断材料（着手可能な成果物・依存・状態）が記述側に出る。本部は自リポでの
自己観察として、(a) 文脈を持たない参加者が着手集合を導出できた局面、(b) 書き忘れがグラフ上のエッジ/ノードの不在として
検出できた局面、(c) 個別対応していた滞留をルールや lint に移せた局面を順に挙げる。文脈ゼロの AI エージェントはこの3点を
一度に検証できる素材なので一例だけ置く（主張ではなく検証手段）。AI 同士の意思疎通など、エピソードで裏づけられない一般化はしない。

**N=1 開示（運用層向け・地の文）**: 「第3部と同じく観測対象は自リポ1個・短期間で、運用したのも判定したのも著者本人である。
他リポでの再現は確認していない（後述の『運べる構造』は設計上の主張で、移植実績ではない）。」

**運用層の出自（provenance・本文では1文）**: pfd-cycle / pfd-ops / pfd-retro という運用層は清水 PFD にも第1〜3部の言語仕様にも
無い、PFDSL の上に**足した**もの。「言語の効能（第3部）」と「言語の上に載せた運用の効能（本部）」を混同しない。

**4つの観測（本文では各1〜2段落の散文。下表は執筆者の整理用・本文に表で出さない）**:

| 観測 | 実エピソード | 反実仮想 |
|---|---|---|
| D1 文脈ゼロの参加（AI はテストケース）| cold subagent で運用を dogfood → 設計盲点2件（issue close timing・チェーン未定義の孤児 process）を露出 → 改訂後に再実行したら両欠陥が消滅 | 暗黙文脈に依存する運用なら、文脈ゼロの参加者は着手判断すらできない |
| D2 失敗を「構造的欠落」として検出 | 3PR 連続の追記漏れを「あるべきエッジの不在」として同型一括検出／列挙ドリフト検出／運用ルールの暗黙スコープ欠落を検出 | 文章ドキュメントなら「更新忘れ」の個別指摘で終わり、同型件を束ねられない |
| D3 検査の二層と昇格 | 孤立 process は最初は運用側の門番が拾っていた（人が回すルール）。同じ点検を言語層の `check` 検査（V020）へ移し機械検出に格上げ | 気付きに依存したままなら検出者が代わると同じ滞留が再発する |
| D4 他リポへ運べる構造（補強・短く）| 汎用プロトコルとリポ固有バインディングを分離し配布可能に | 運用知識がリポ固有にハードコードされると移せない（※「運べる構造がある」までで効果実証ではない、と明示）|

D3 で「検査には言語が `check` で見る欠落と、運用側で点検する欠落の二段がある」を**事件として初めて導入**する
（第1部の②の精緻化をここで回収）。本文では `②-a/②-b` の記号でなく「言語層の検査／運用側の門番」と平易に書く。

**構造的欠落の地の文ドラフト（実例先行→要約語は最後に一度だけ）**: 「同じ時期、3本の PR が続けて追記を漏らした。文章
ドキュメントなら『3回目も忘れた』で終わる。だが PFDSL では追記漏れは『あるはずのエッジがグラフに無い』という同じ形をとり、
3件は同型の1パターンとして1回の監査でまとめて挙がった。本稿ではこの『あるべきエッジ/ノードが無い』形の失敗を構造的欠落と呼ぶ。」

**昇格の地の文ドラフト**: 「孤立 process の混入は最初、運用側の門番が拾っていた。人が回すルールゆえ回す人が代われば素通りしうる。
そこで同じ点検を言語層の検査に移し、`check` が `.pfdsl` 単体で孤立 process を機械検出するようにした。運用ルールで見つけた欠落を、
言語の検査に格上げした形になる。」

**効かなかった局面（実在の負例 A・運用層の散文に溶かす）**:
- *原則が事後にしか効かなかった*: `schedule:` フィールドを checker・サンプル・テスト・snapshot まで作り込んでから、
  「pfdsl 単体に価値を与えない」と判断して全廃棄した（ADR-0021 / revert、十数ファイル削除）。B群の判定基準（pfdsl 単体価値テスト）は
  持っていたのに、効いたのは実装を一周した後だった。図やツールがあっても設計判断ミスは事前には防げていない。

**結（buzzword 確定文言・「回収/閉じる」を排除）**: テキスト性＋制約の機械検査＋失敗様式の構造化が揃うと、運用は「誰かの気付き」に
依存しなくなる。PFDSL は人間の作図言語にとどまらず、文脈を持たない参加者（新任の人間でも、文脈ゼロの AI でも）が同じ記述を読んで
運用に加わり、失敗が構造として現れる基盤になる。第1部で挙げた3要求が、運用の場面でも同じ形で現れることを確認して本連載を終える。

**脚注（記事末・主題化しない、buzzword 確定文言）**: 「注: 手順の外化を扱う近年の研究に Externalization in LLM Agents
（arXiv:2604.08224, 2026）がある。本稿はこれを論拠にはせず、近い現象を自リポで1例観測したと記すに留める。この種の運用層は
"harness engineering" と呼ばれることもある。」

## 執筆者メモ（本文に出さない・self-check 用）

本文へ持ち込まないが、provenance 事故・altitude 混同を防ぐために執筆者が常時参照する。

**清水軸（keep/tighten/drop/add）と PFDSL の扱い軸（A 課す/B 採らない/コーダ）は直交する2軸**。継承（keep）を機械検査として
課すこと（単一生成元＝A）も、継承した不在を継続して採らないこと（条件分岐＝B）もある。A群/B群に keep 項目が含まれるのは
矛盾でなく二軸が直交するため。

| 項目 | ADR | 清水軸 | 扱い軸 |
|---|---|---|---|
| 成果物の有形性 | 0001 | keep（XDDP spec-out と同方向） | A 課す |
| 単一生成元 | V001 | keep | A 課す |
| DAG 性／循環不在 | — | keep | A 課す |
| 条件分岐の不在 | 0005 | keep（清水も持たない）＋add（変換パターン規定） | B 採らない |
| プロセス粒度4基準 | 0004 | tighten（「見積もり可能な最小単位」を精緻化） | A 課す |
| update 意味論 | 0003 | **drop（清水軸で唯一の意味論変更）** | B 採らない |
| subroutine | 0019 | keep を守る却下（1:1 leveling 保護）＋ `tags` は add | B 採らない |
| 終端門番（後続側）| 0018 | add（清水にない運用層） | 第4部（運用層）|
| 最小コア＋開く拡張 | 0021 | add | コーダ |

**provenance スコープ（vs 清水 PFD）**: このフレームの軸は「清水 PFD に対し何を変えたか」。清水の意味論に対し意図的に
**変更（drop）**したのは update 不採用ただ一つ。他は継承（keep）か、清水に無い概念の**追加（add）**（テキスト DSL 化・tooling・
`tags`・運用層・メタデータ拡張）。なお「追加」概念が PFDSL 内部で版を重ねて意味論を変える（`tags` を Artifact 専用→両種別に
広げた ADR-0019 など）のは**内部版進化**で別軸。「PFDSL は自分の意味論を一度も変えていない」と誤読しないこと。

**要求②の二層（言語層 vs 運用層）**: ②-a 言語層 validator（単一生成元・DAG・終端健全性を `check`/`--audit`/V020 で判定）と
②-b 運用層門番（手段成果物が後続を欠く欠落を pfd-ops 終端ゲートで点検＝ADR-0018(b)）。第1部では②を一つに提示し、二層の区別と
「②-b で見つけ②-a へ昇格させる」運動は第4部 D3 で事件として導入する（提示位置の決定。論理としては両層が常に存在）。

**採否メモ**: ADR-0008（group 指針）は本部から外す。ADR-0011 は第3部結で軽く触れる。ADR-0006 は第2部の蝶番。
`/code-review` の PR スコープ外バグ検出（payoff log 2026-06-15）は PFD ハーネスの dogfood でないため中核に据えない。

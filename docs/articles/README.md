# PFDSL 連載記事

ADR 群を素材に、PFDSL の問題意識と設計根拠を説明する公開記事（連載）。
issue [#12](https://github.com/takasek/pfdsl/issues/12) に対応し、リポの生態系図上は
`write_article` プロセスの出力 `i12_article` にあたる。

この README は連載の**方針と構成のインデックス**であり、執筆の前提を一箇所に固定するための文書。
各部の本文を書く前に、ここで軸・読者・NG・構成を確定させてから書く。

## 読者

エンジニア一般。プロセス（仕事の段取り）は職種を問わず誰もが日々扱うものなので、間口は広く取る。
「プロセス設計に関心のある人」には限定しない。PFD の前提知識は仮定せず、導入から書く。
ただし第2部以降は徐々に PFDSL 固有概念に踏み込むため、内部用語（ADR 番号・V001・`>>?`・dogfooding 等）は
**初出で定義するか脚注に逃がす**。間口は第1部で最大に取り、後半は「PFD に関心が芽生えた読者」へ段階的に絞る。

## 軸（書くこと）

- **そもそもツール自体がない状態での問題**。
  プロセスを形式化したいのに、それを支え・制約を守らせる手段が存在しない、という状況そのものの問題。
- その**解としての PFDSL**。
- **制約をツールで管理することは重要な要求**。
  形式化は規約・人手レビューだけでは形骸化する。
  単一生成元・DAG 性・終端健全性などを機械が検査できる必要がある。

## 書かないこと（NG）

- 社内・SWET の GUI ツールとの差分・比較を**主題にしない**。
  要求の実在を示す先行例として一度だけ触れるに留める。
- 「作図ツールの限界」を記事の中核に据えない。差別化は独立した比較章を作らず、既存の論証ライン
  （WBS→PFD、`check` 実演→テーゼ）に1段落ずつ寄生させ、優劣を煽らず PFD/PFDSL へ収束させる。
- AI 臭いポエム調・過剰な比喩・情緒的レトリックを使わない。**淡々と具体的に書く**。

## 言語

日本語で書く。これは CLAUDE.md「ユーザーの目に触れる公開ドキュメントは英語」への**意図的な例外**。
対象が日本の方法論（清水吉男の PFD）で、読者・媒体が日本語のため、明示的に日本語を選ぶ。
ファイル名・slug は英語。

## 配置・命名

- 本 README — 連載の方針と構成のインデックス。
- 各部の本文 — `docs/articles/NN-slug.md`（例: `01-tangible-text.md`）。`NN` は連番、`slug` は英語。

## 連載の構成（暫定・柔軟）

本数は固定せず、暫定 3〜4 部で柔軟に進める。第1部のみ確定、以降は暫定。

| 部 | 状態 | 責務 / テーゼ |
|---|---|---|
| 第1部 | **確定** | ツール不在の問題 → 要求の導出 → 解としての PFDSL。テーゼ: 成果物の明示・PFD 意味論の機械検査・テキスト性が揃って初めて、検証可能でレビュー可能で AI と共有できるプロセス記述になる。 |
| 第2部 | **確定** | 制約の設計を「課す/採らない」で二分する。A群＝グラフを検証可能にするために**課す**制約、B群＝静的解析性と最小性を守るために**採らない**機能。清水軸（keep/tighten/drop/add）とは直交する2軸で扱う。下の節構成を参照。 |
| 第3部 | **確定** | 言語そのものの dogfooding（pfdsl）。PFDSL を PFDSL 自身の開発で使ったら、図が設計判断器・テスト生成器・レビュー装置として働いた（自リポ1個の自己観察）。下の節構成を参照。 |
| 第4部 | **確定** | 運用ハーネス（pfd-cycle / pfd-ops / pfd-retro）の dogfooding。テキスト化＋機械検査で運用が暗黙文脈・気付きから独立する。1記事に保つ（D1〜D3 が一貫した1テーゼ、D4 は補強）。下の節構成を参照。 |

## 第1部 節構成（確定）

仮タイトル: **「成果物を、テキストで」**（後で詰める）

1. **効能の動機（入口）** — 書き捨てでないプロジェクトには必ずタスクの依存関係が生じ、その管理が求められる。
   職種を問わず誰もが踏む状況として導入する。
2. **既存モデルの欠落** — 依存管理の代表は WBS。WBS は PMBOK 定義ではむしろ成果物指向
   （deliverable-oriented hierarchical decomposition）であり「成果物の語彙がない」わけではない。
   だが WBS は成果物の*階層分解（包含ツリー）*であって*依存グラフではない*——「どの成果物がどの
   プロセスを介してどの成果物から生成されるか」という生成エッジを持たず、順序・依存は別途
   スケジュール／ネットワーク図へ逃がす（そこは作業指向に戻り、成果物の検証可能性が薄れる）。
   フローチャートも「順番は分かるが関連性の情報が希薄」（SWET 記事の指摘）。
   またデータフロー図（DFD）とも見た目は近いが、DFD はデータストアの読み書き（状態更新）を許し循環を含みうる。
   PFDSL はプロセスを「入力成果物から出力を作る純粋関数」とみなし update を意味論から外す（ADR-0003）ため
   必ず DAG になり、着手順も宙に浮いた成果物も静的に判定できる。
   → PFD の追加手は、成果物とその生成依存エッジを単一生成元のもと1枚のグラフに統合し機械検査可能にする点。
3. **成果物中心モデル＝PFD** — 成果物を明示すれば完了が検証可能になる。これが清水吉男の PFD の核。
4. **制約のツール管理は実在要求で、先行例が満たしていた** — PFD の整合維持は人手では「かなりの労力」。
   SWET の PFD Draw Tool は単一生成元・二部性を「そもそも書けないようにする」ことで制約をツール強制した。
   比較が主題ではなく、要求の実在を示す先行例として一度だけ触れる。
   （注: 第1部のテーゼ見出しは「ツール不在」ではなく「テキスト化・版管理・AI 受け渡しができる PFD 実装が
   無かった」と正直に書く。先行ツールは存在したので「不在」と誇張しない。）
5. **残った欠落＝生成 AI 時代の新要求** — Draw Tool はブラウザ上の手動編集が前提（生成 AI 以前）。
   テキスト表現を持たないため diff・レビュー・版管理（git）・AI への受け渡しができない。
   （diagram-as-code 系（Structurizr 等）が「テキスト形式は AI がパースしやすく architectural drift を
   検出できる」と同方向を主張している事実に一度だけ触れる——比較は主題化しない。）
6. **要求の導出＝正準の3要求（PFDSL）** — 以降の部はこの3つを“回収”で呼び戻すので、ここで一度確定する。
   **① 成果物とその生成依存エッジの明示**（単一生成元のもとで）— 完了が検証可能になる。
   **② PFD 意味論の機械検査（二層）** — テキストで diff でき CI に載る記述は今や珍しくない。PFDSL に固有なのは
   検査できること一般ではなく、何を検査するか。**②-a 言語層**: 単一生成元・DAG 性・終端健全性（消費者なし終端
   `check --audit`／孤立宣言プロセス V020）を `.pfdsl` 単体に `check` が判定する。**②-b 運用層門番**: validator では
   捕捉しきれない「手段成果物が後続プロセスを欠く」欠落（ADR-0018(b)）を運用ハーネス（pfd-ops 終端ゲート）が
   点検する。②-b は第4部の運用層に属し言語仕様でない、と先に断る。
   **③ テキスト性** — 座標を持たずグラフ構造だけを編集し、diff・レビュー・版管理（git）・AI への受け渡しを
   可能にする（メタデータ拡張を開く柔軟性も③の系）。
7. **解：PFDSL** — README の最小例で記法を導入し、`check` で単一生成元などの制約検査を実演する。
7.5. **描画と検査は別の仕事** — 図を「描く」手段は既に豊富で、PFDSL の SVG も内部は Graphviz の系譜に乗る
   （絵を出すこと自体は新しくない）。だが描画器が知るのは「ノードとエッジの並べ方」だけで、単一生成元・DAG・
   終端健全性は問わない——それらは PFD という方法論の意味論で、汎用の描画記法に語彙がないからだ。PFDSL が
   足したのは絵でなく、この PFD 固有の制約を構文として `check` で機械検査する層。描画器と PFDSL は競合でなく
   担当が違う（前者は見た目、後者は PFD の整合性）。固有名は Graphviz の自己開示のみ、比較カタログにはしない。
8. **第1部テーゼ** — 正準の3要求（①明示・②PFD 意味論の機械検査・③テキスト性）が揃って初めて
   「検証可能・レビュー可能・AI 共有可能」なプロセス記述になる、と結ぶ。②は固定点でなく**運用層(②-b)で発見し
   言語層(②-a)へ昇格させる動的二層**である（第3部は②-a を実装前へ延伸、第4部は②-b→②-a の昇格を示す）。
   各部の結はこの3要求／テーゼを同一の形で呼び戻す。次部（各制約の設計判断）へ橋渡しする。

## 第2部 節構成（確定）— 制約の設計を「課す/採らない」で二分

**背骨テーゼ**: PFDSL は2方向で制約を選ぶ。(A) グラフが満たすべき性質を機械検査可能な形で**課す**。
(B) 静的解析性と最小性を壊す機能は**採らない**。両者を貫く判定基準は
「DAG の静的解析性（ランク計算・到達性・正準順序）を壊さないか」「機械が検査できるか」
「pfdsl 単体に価値があるか」。各 ADR はこの基準を別々の誘惑に適用した記録。

導入で第1部の要求（②PFD 意味論を機械が検査する）を受け、上記の判定基準を提示する。

**A群 — 課す制約（言語層でグラフを検証可能にする）**

| 節 | ADR | 設計問題 → 決定 | 裏づけ・例 |
|---|---|---|---|
| A1 成果物の有形性 | 0001 | 「理解」「合意」のような不定形を出力に許すか → 非対称規則（出力は検証可能なモノ／入力は不定形可） | 完了判定・status・引き継ぎが成立する条件。「応急処置」→「暫定対応記録」。理解の外化は清水自身の XDDP の spec-out と同方向 |
| A2 単一生成元 | V001 | 同一成果物を複数プロセスが生成してよいか → 禁止 | 生成元の一意性が diff・改版・責任の基準点。Draw Tool も「書けないように」していた。0002/0003 の制約条件 |
| A3 終端健全性（言語層）| 0018 | 出力ノードが宙に浮く → 消費者なし終端 `check --audit`／孤立プロセス V020 で検出 | 形骸化の構造的検出（②-a の回収）。後続門番（手段成果物の後続登録 ADR-0018(b)）は運用ルールゆえ②-b として第4部へ送る |
| A4 プロセス粒度 | 0004 | どこまで分割するか → 4基準（時間的凝集禁止／新依存なし分割禁止／相互依存は1プロセス／所有権境界） | 偽依存と隠れ循環の排除。マイクロサービス同型。階層×プリセットが並列→上流方針→共同起草と収束 |

**B群 — 採らない選択（最小性と静的解析性を守る。継続して採らない不在を含む）**

| 節 | ADR | 欲しくなる場面 → 決定 | なぜ採らないか |
|---|---|---|---|
| B1 update 意味論 | 0003 | DB・本番・サービング中モデル → 不採用、スナップショット＋ビュー | update は循環を生み DAG 性・単一生成元・diff 安定性を壊す |
| B2 条件分岐 | 0005 | 合否・承認・閾値 → 不採用、成果物定義で解消 | 分岐は静的解析性を壊し状態機械と重複。承認記録・結果報告書＋`>>?` へ変換 |
| B3 subroutine | 0019 | 似た Process の再利用 → 却下、横断ラベル `tags` で代替 | 1:1 leveling から逸脱、図の現実地図性を損ない、境界チェックを複雑化（0004 と衝突） |

**橋渡し — 改版の表現（0002）**: B の「update を採らない」と A の「単一生成元」を両立させると、
改版はどう書くか。3形態（別 artifact／`>>?` 収束／定常サイクル）。A と B の制約から演繹される
帰結として、**B 群の後・第2部結の直前**に橋渡し節として置く（確定）。

**清水軸（keep/tighten/drop/add）と PFDSL の扱い軸は直交する2軸**: 「清水 PFD に対して何をしたか」
（keep 継承／tighten 精緻化／drop 意図的削除／add 追加）と「PFDSL が今どう扱うか」（A 課す／B 採らない／コーダ）は
別の問い。継承（keep）を機械検査として課すこと（単一生成元＝A）も、継承した不在を継続して採らないこと
（条件分岐＝B）もある。A群/B群に keep 項目が含まれるのは矛盾でなく、二軸が直交するため。

| 項目 | ADR | 清水軸 | PFDSL の扱い | 注 |
|---|---|---|---|---|
| 成果物の有形性 | 0001 | keep（二部構造・XDDP spec-out と同方向） | A 課す | 継承を機械検査として課す |
| 単一生成元 | V001 | keep | A 課す | 同上 |
| DAG 性／循環不在 | — | keep | A 課す | 同上 |
| 条件分岐の不在 | 0005 | keep（清水も持たない）＋add（変換パターンを規定） | B 採らない（継続して採らない） | 「採らない」は drop でなく継承の継続＋規約化 |
| プロセス粒度4基準 | 0004 | tighten（「見積もり可能な最小単位」を精緻化） | A 課す | 唯一の純 tighten |
| update 意味論 | 0003 | **drop（清水軸で唯一の意味論変更）** | B 採らない | 清水にあった update を落とした |
| subroutine | 0019 | keep を守る却下（1:1 leveling 保護）＋ `tags` は add | B 採らない | 却下の動機は継承の保護 |
| 終端門番（後続側）| 0018 | add（清水にない運用層） | 第4部 ②-b | 言語制約でなく運用ルール |
| 最小コア＋開く拡張 | 0021 | add | コーダ | B群本体と分離 |

**継承と判断（書き分けの注意・軸は vs 清水 PFD）**: このフレームの軸は「清水吉男の PFD に対して何を変えたか」。
**清水 PFD から継承**したもの — 二部構造・単一生成元・`>>?` フィードバック・条件分岐の不在 — は我々の発明として
書かない。清水 PFD の意味論に対して PFDSL が意図的に**変更（drop）**したのは update の不採用ただ一つ。他はすべて
継承（keep）か、清水 PFD に存在しない概念の**追加（add）**——テキスト DSL 化・tooling・`tags`・終端門番の運用層・
メタデータ拡張など。なお「追加」した PFDSL 固有概念がその後 PFDSL 内部で版を重ねて意味論を変える（例: `tags` を
Artifact 専用→両種別へ広げた ADR-0019）ことはあるが、それは**PFDSL 内部の版進化**であって「清水からの意味論変更」
とは別軸。両者を混同して「PFDSL は自分の意味論を一度も変えていない」と読まないこと。

**コーダ — 最小コアと開かれた拡張（0021）**: B1〜B3 が「意味論を採らない」のに対し、これは種類の違う
不採用。メタデータ拡張自体は開く（未宣言タグ・フィールドは許容）が、意味づけが完全に外部ツール依存で
グラフ意味論／可視化／`check` のどれにも寄与しないフィールド（例: `schedule:`）は core に彫り込まない。
「open（拡張は自由）と minimal（コアは最小）の両立」を短く添えるに留め、B 群本体とは分けて扱う。

**蝶番 — ルールは実践で限界を露呈した（ADR-0006 の物語）**: ここまでが「ルール（設計判断）」の話。
だが品質ガイド整備後の実践ラウンド2で、*ガイドに明記済みのミスが再発した*——修正開発プロセスに
`base_code` 入力が無い（誤りはガイドの例文そのものにあった）、消費されない成果物の放置が2例。根拠は少数
事例（計3件）なので一般法則でなく**示唆**として扱うが、教訓は明瞭——「列挙的・網羅的な確認を要するミスは
ルール提示だけでは防げない」（ルールの質でなく認知の限界）。そこで品質担保を**ルール（判断）＋ lint（機械検査）の
二層**に分けた。さらにこの実践→レビューのループは、ガイドを締めると同時に「どのミスがルールで消え／どれが
残るか」を2ラウンドの比較で測り、**lint バックログと優先度を方向づけた**。方法論が自分自身に適用されて検証
された具体例——これが第3部（言語の自己適用）・第4部（運用の自己適用）が“ツールと dogfooding の話”になる理由。

**結**: 課す/採らないは同じ基準の表裏。各 ADR はその基準を別々の誘惑に適用した記録。
ルールの限界（ADR-0006）が tooling と自己適用を要請する——第3部（言語の自己適用）へ橋渡し。

採否メモ: ADR-0008（存在様式でグループを切る）は `group` 用法の指針で本部から外す。
ADR-0011（能力成果物の世代還流 `>>?`）は第3部の結で回収。ADR-0006 は上記の蝶番で扱う。
終端門番の後続側（ADR-0018(b)）は運用ルールゆえ②-b として第4部へ送り、A群は言語層で課す制約に純化した。

## 第3部 節構成（確定）— 言語そのものの dogfooding（pfdsl）

仮タイトル: **「PFDSL を PFDSL の開発で使う」**（後で詰める）

**背骨テーゼ**: PFDSL を PFDSL 自身の開発で使ったら、図が3つの装置として働いた——
設計判断器（着手順・受け入れ・スキーマ分岐を即決させる）、テスト生成器（ADR の具体例
ラウンドがそのまま fixture になる）、レビュー装置（暗黙依存・相互依存を反証可能にする）。
第1部の3要求（①明示・②PFD 意味論の機械検査・③テキスト性）が揃った**構造的帰結**として説明できる。
各効能は `docs/pfd_payoff_log.md` の実エピソードで裏づける（反実仮想テスト＝「依存グラフがなければこの判断は
違っていたか」を満たすものだけを引く）。**ただし誠実に前置きする**: これは自リポ1個・短期間の自己観察（N=1）で、
判定者は著者本人。3装置は3要求に対応させた**事後整理**であり、どの箱にも入らない素材もある。

導入で第1部の要求を受け、「言語そのもの（pfdsl）の自己適用」に範囲を限定する——運用サイクル・
ハーネス（pfd-cycle / pfd-ops / pfd-retro）と AI 協働は第4部に送る、と最初に宣言する。

**C群 — 自己適用で観測された3つの効能**

| 節 | 効能 → 装置 | 実エピソード（payoff log）| 反実仮想（図がなければ）|
|---|---|---|---|
| C1 設計判断器 | 図が設計分岐を即決させた | 着手可能集合の機械導出（入力が全 done のプロセス列挙＝並列着手リストを9本即答）／新提案の受け入れコスト低減（1チェーン追加で並列性・接点・合流点が確定）／location 単数か辞書かを PFD 原則だけで3分割即決 | 文章ロードマップなら毎回頭の中で依存と並列性をやり直す |
| C2 テスト生成器 | 設計検証ラウンドがそのまま仕様のテストスイートになる（②-a を実装前へ延伸） | ADR-0020 の boundary-validation-log の具体例が 11 テストケースにそのまま変換（subflow 境界整合 §15.11） | 仕様テキストだけでは「非単射は集合等価ミスマッチで検出」等の非自明ケースを事前網羅できなかった |
| C3 レビュー装置 | 暗黙依存・相互依存が図でレビュー可能になる | 暗黙依存のレビュー可能化（「並列起草できる」直感が図で反証され、並列→上流方針→共同起草と収束し ADR-0004 基準3を発見）／相互依存の先回りモデル化（roadmap が #5/#6 を1プロセス3出力で畳んでいたため後続着手者が「並列主張」の罠を踏まずに正解構造を受け取る）／W002 拡張が自リポの `spec_v007`（criteria 欠落）を即時警告 | 直感の依存主張がレビュー対象にならず、起草が衝突して初めて相互依存に気付く |

**摩擦も書く（手前味噌回避）**: payoff log は定義上「効いた局面」しか溜まらない（生存者バイアス装置）。
本部には鏡像として**効かなかった/邪魔だった局面**を1節置く——図の作成・維持コスト、過剰制約で書きたい構造が
書けなかった場面、`check`/lint の偽陽性など。負の事例が1つも無いなら、その不在自体を疑う材料として正直に書く。

**結 — 効いた局面の共通構造**: 自己適用が効いたのは、第1部の3要求が揃ったことの構造的帰結として記述できる。
成果物が明示されているから完了が判定でき（C1 の status 導出）、PFD の整合条件が機械検査できるから設計の穴が
テストとして外化でき（②-a の延伸＝C2）、テキストだから依存主張が diff・レビューに載る（C3）。
ここで **能力成果物の世代還流（ADR-0011）** に触れる——言語・ツール（pfdsl 自身・ガイド・lint）が自分の生産
チェーンの上流プロセスへ前世代スナップショットとして `>>?` で還流する。これは自己適用が図に現れる一例であって、
「自己適用そのもの」と等号で結ぶ比喩はしない。第4部（運用ハーネスの dogfooding）へ橋渡しする。

採否メモ: 「着手可能集合の機械導出」「相互依存の先回りモデル化」は roadmap 運用に見えるが、
ここでは**言語機能（status・1プロセス複数出力・`>>?`）が判断を生んだ**側面に焦点を絞って扱い、
運用サイクル（pfd-cycle のフロー自体）と AI 協働は第4部に回す。境界が滲む素材は「言語の効能か、
運用の効能か」で振り分ける。

## 第4部 節構成（確定）— 運用ハーネスの dogfooding（pfd-cycle / pfd-ops / pfd-retro）

仮タイトル: **「暗黙文脈なしに運用を回す」**（後で詰める）

**背骨テーゼ**: 第1〜3部で PFDSL は「言語」として成立した。第4部は、その言語の上に**運用ハーネス**
（pfd-cycle / pfd-ops / pfd-retro）を載せると何が起きるかを扱う。運用層をテキスト化し機械検査可能にすると、
運用が**特定個人の暗黙文脈・気付きに依存しなくなる**——(a) 文脈を持たない参加者でも着手集合を導出して回せ、
(b) 失敗が「構造的欠落（あるべき edge・ノードの不在）」として検出でき、(c) 滞留を個別対応でなくルール/lint へ
移して再発を構造で止められる。文脈ゼロの AI エージェントは、この「暗黙文脈からの独立」を最も鋭く示す
テストケースとして D1 に一例だけ置く（主張そのものでなく検証手段）。AI↔AI の意思疎通など、エピソードで
裏づけられない一般化はしない。

**provenance 注記**: この運用層（pfd-cycle / pfd-ops / pfd-retro / companion・references の
層構成）は**すべて我々の追加（add）**であり、清水 PFD にも第1〜3部の言語仕様にも存在しない。
「言語の効能」（第3部）と「言語の上に載せた運用の効能」（本部）を本文で混同しない。

導入で第3部までの「言語」を受け、「言語の上の**運用層**」に範囲を移すと宣言する。

**D群 — 運用ハーネスの dogfooding で観測されたこと（自リポ1個の自己観察）**

| 節 | 主張 → 何が起きたか | 実エピソード（payoff log）| 反実仮想／効いた構造 |
|---|---|---|---|
| D1 文脈ゼロの参加（AI はテストケース）| 成果物・依存・状態がテキストに外化されているから、暗黙文脈なしに参加者が着手可能集合を導出して回せる。文脈ゼロの AI でそれを最も鋭く検証した | cold subagent で pfd-ops を dogfood → 設計盲点2件（issue close timing・チェーン未定義による孤児 process）を露出 → 学習ループ3経路に振り分け → 改訂後に同手順を cold agent で再実行したら両欠陥が消滅（dogfood→検出→改訂→再検証の閉ループ）| 人間の暗黙文脈に依存する運用なら、文脈ゼロの参加者は着手判断すらできない |
| D2 失敗様式が「構造的欠落」として検出できる | 「うっかり（書き忘れ）」を graph 上の edge/ノードの不在へ読み替えると、同型の複数件を1回の監査で束ねて検出できる | pfd-retro の3層監査：「忘れ物＝edge 欠落」で 3PR 連続の impl_flow 追記漏れを同型一括検出／列挙ドリフト検出／C層が運用ルールの暗黙スコープ（「open PR」が flow-sync 限定という文脈を文言が持たない欠落）を構造的不在として検出 | 文章ドキュメントなら「更新忘れ」の個別指摘で終わり、同型件を束ねられない |
| D3 ハーネスの自己改善（②-b→②-a の昇格）| 監査・指摘で露出した滞留を、個別対応で閉じず「ルール／lint／構造」へ移管して再発を構造で止める＝運用門番(②-b)で見つけた欠落を言語層(②-a)へ昇格させる | ユーザーの監査質問2連発が実滞留を検出 → その監査パターン自体を pfd-retro スキルに蒸留／ADR-0006 のlint要件経路が完結し孤児 process が V020 で機械検出可能に（ルールで防げない→ツールへ送る閉ループの実証）| 気付きに依存したままなら、検出者が代わると同じ滞留が再発する |
| D4 L3 カスタマイズ層（補強・短く）| **方法論が他リポへ運べる理由**を短く示す——層構成のツアーにはしない。汎用プロトコルとリポ固有バインディング（companion `.md`／references）を分離し配布可能にする | pfd-ops の汎用層／固有層分離、`skill sync` 配布、GitHub-Issues backend を references で差し替え | 運用知識がリポ固有にハードコードされると方法論を他プロジェクトへ移せない（※「運べる構造がある」までで、他リポでの効果実証ではない点を明示） |

**摩擦も書く**: 第3部と同様、運用層でも「効かなかった/重かった」局面（過剰モデル化・図メンテのコスト・
偽陽性）を1つ正直に置く。N=1（自リポ1個・観測は2026-06 の短期間）という限界も本部の導入で開示する。

**結 — 運用を気付きから独立させる（連載全体の結）**: テキスト性＋制約の機械検査＋失敗様式の構造化が
揃うと、運用は「誰かの気付き」に依存しなくなる。PFDSL は人間の作図言語にとどまらず、文脈を持たない参加者
（新任の人間でも、文脈ゼロの AI でも）が同じ記述を読んで運用に加わり、失敗が構造として現れる基盤になる。
第1部の「検証可能・レビュー可能・AI 共有可能なプロセス記述」というテーゼを運用の実地で回収して連載を閉じる。

**外部接続（脚注程度・主題化しない）**: 「手順を外化すれば脆い暗黙知が明示的な実行パスになる」という観察は
近年の AI エージェント研究にもある（Externalization in LLM Agents, arXiv 2604.08224, 2026）。本連載はその理論を
支柱に据えるのでなく、理論が予測する現象を自リポで1例観測したに留める（一般化しない）。pfd-cycle/ops/retro は
この種の運用層が近年 "harness engineering" と呼ばれることもある、という程度に脚注で触れる。流行語に密結合させ
ない（第1〜3部の主張は流行語なしで立つ）。

**構成判断（確定）**: 1記事に保つ。判断基準は「焦点が変わるテーマがあれば割る」。D1〜D3 は
背骨テーゼの3主張（文脈ゼロ参加／失敗の構造化／自己改善）が一直線に「運用を気付きから独立させる」結へ
向かう一貫したテーゼで、焦点は変わらない。D4（L3 カスタマイズ層）は「上記を他リポへ持ち出せる
構造」という補強であり、独立記事にするほどの焦点転換ではないため結の手前に短く置く。

採否メモ: `/code-review` による PR スコープ外バグ検出（payoff log 2026-06-15 の2件）は
「AI レビューの効能」だが PFD ハーネスそのものの dogfooding ではないため、本部では中核に据えず、
必要なら D2 の補強事例として軽く触れるに留める。

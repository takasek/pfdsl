# 構造の見落としは人が拾えない（PFDSL 連載 第2部）

## ガイドを増やしても見落としは再発する

品質ガイドを整備すると、ミスの一部は消える。
しかし別の種類のミスは残り続ける。
理由は、ガイドの多くが「漏れなく確認する」という作業をそのまま人に要求するからだ。
成果物が数十件あるとき、そのどれが要求を満たしているかを毎回目視で突き合わせるのは注意力に依存する作業であり、疲労や慣れによって精度が落ちる。

放置も同じ構図で起きる。
あるプロジェクトの成果物一覧に、誰にも参照されていない資料が紛れ込んでいるとする。
日常のレビューではまず気づかれない。
気づくのは、誰かが「使われていない成果物はないか」と問い直したときだけである。

この構図は品質ガイドの書き方の問題ではない。
列挙的な確認を人に任せている限り、同じ種類の見落としが繰り返される。

## PFD の意味論を機械で検査する

機械に確認を任せるには、確認できる形で書かれている必要がある。
PFDSLでは、出力である成果物だけを**有形な成果物**に限定する。
入力側はレビュアーの知識のように不定形なものを許すが、出力はファイルや文書のように保管し検証できるものでなければならない。
「理解した」は成果物にならないため、理解した内容をまとめた資料に置き換えて書く。
「応急処置をした」も同様に、何をどう当てたかを書いた暫定対応記録に置き換える。
出力を有形なものに限ることで完了の判定やstatusの管理が成立し、その先で機械検査の対象になる。
入力側まで有形さを求めると、経験や勘のように外化しづらいものまで無理に成果物へ変換することになり、図はかえって実態から離れる。
入力と出力を非対称に扱うのはそのためだ。
清水吉男がXDDPで導入したスペックアウトも、理解を資料として外化するという同じ発想に立つ。

こうして外化された図に対して、PFDSLの検査が見ているのは、テキストで書かれたものなら何でもCIに載せられるという一般論ではない。
PFDという図法の意味論に固有の壊れ方だけを対象にする。
具体的には次の4点である。

- **単一生成元**：同じ成果物を複数のプロセスが生成していないか
- **循環の禁止**：成果物とプロセスの依存グラフに循環がないか（循環があると着手順序を導出できない）
- **孤立プロセスの排除**：どのエッジにも参加しない、宣言だけのプロセスが残っていないか
- **完了基準の必須化**：生成される成果物に完了基準が書かれているか

孤立プロセスの検査は、たとえば次のように弾く。

```
orphan.pfdsl:1:1: error [V020]: Process 'publish' is declared but has no edges (orphaned process)
```

完了基準の検査は、次のように警告する。

```
nocrit.pfdsl:3:3: warning [W002]: Artifact 'spec' has no 'criteria' field
```

品質ガイドと機械検査は、役割が分かれた二つの層である。
品質ガイドが担うのは設計判断である。
成果物をどの粒度で切るか、どう名付けるか、理解や応急処置をどう資料に外化するかは人が決めるしかない。
列挙的な確認、つまり全件を漏れなく突き合わせる作業は機械に任せる。

この分業がどう効くかを、具体的な場面で見る。
完了基準の警告を、完了済みの成果物だけでなく作業中の成果物にも広げたとしよう。
ルールを適用した瞬間、自分の進行管理の図で、作業中なのに完了基準が書かれていない項目が引っかかる。
ルールを作った本人の図から真っ先に見つかるのは珍しくない。
人はルールを覚えたつもりでも、数十ノードの図を毎回全件照合してはいないからだ。
このとき、その項目に依存する別の成果物がすでに完了扱いになっている、という矛盾が同時に見つかることもある。
未完成の入力に依存する成果物を完了と呼んでよいかは、警告の対象を広げる前の検査からは見えていなかった問いである。

別の場面を考える。
経緯を知らないまま実装を引き継いだ担当者が、依存のチェーンを一部削除したとする。
削除の後に宣言だけが残ったプロセスは、人手のレビューでは見落とされやすい。
エッジを持たないノードは図の見た目をほとんど変えないからだ。
孤立プロセスの検査はこの型の欠落を機械的に検出する。

検査の対象を広げていくと、指摘の分布も変わっていく。
構造上の欠陥は機械化された範囲でまず検出されるようになり、残るのは検査の届かない散文の層、たとえば配布手順の説明文や、参照が連なっていく記述である。
これは経験則ではなく、機械検査が拾えるのは構造化された依存関係だけであり、散文の整合性そのものは検査の対象外だから起きる帰結である。

## 検査可能性のために入れない機能

図が文脈を持たない読み手にも静的に読み切れることが、ここまでの検査の前提になっている。
この前提を壊す機能は、便利であっても採用しない。

最初に採らないのはupdateである。
元のPFDには、既存の成果物を書き換えるupdateがある。
PFDSLはこれを持たない。
データベースや本番環境、学習済みモデルのように書き換わっていく資源は、ある時点のスナップショット、たとえば日次のダンプやリリース版、ある時点のモデルの重みとして書く。
更新後の自分を自分の入力にすると循環が生まれ、依存グラフから着手順序を導出できなくなる。

改版そのものは3つの形に分けて書く。
3つのうち後の2つの形では、下流の成果物から上流のプロセスへ戻る**フィードバックの辺**（`>>?`）を使う。
フィードバックの辺は着手順序の計算には影響しない。

1つ目は**別成果物形態**である。
版が承認や配布のベースラインになるなら、`spec_v1`と`spec_v2`のように別の成果物として書く。

```pfdsl
spec_v1 >> revise -> spec_v2
```

2つ目は**収束ループ形態**である。
同じ工程の中で収束するまで繰り返す作業は、フィードバックの辺で書く。

```pfdsl
draft >> write -> manuscript
manuscript >> review -> comment
comment >>? write
```

3つ目は**定常サイクル形態**である。
版を数え上げられない無限の繰り返し、たとえばモデルの再学習も同じ記法で書き、性質はコメントで補う。

```pfdsl
# 終わりのない再学習サイクル。data の版は列挙しない
data >> train -> model
model >> evaluate -> eval_metrics
eval_metrics >>? train
```

3つの形は併用しない。
版が何を意味するかを問えば、どれを使うべきかが決まる。

二つ目に採らないのは条件分岐である。
合格したら次へ、承認されたら次へ、と書きたくなる場面は多い。
PFDSLではこの分岐を、成果物の定義そのものが誤っているしるしと捉える。
分岐が必要になるのは、判定の結果そのものが成果物として書かれていないからだ。
判定結果を成果物にすれば、後続のプロセスはそれを入力として受け取るだけでよく、分岐は要らなくなる。
テストに合格したら次へ、ではなく、テストの結果報告書を常に出力し、不具合票という空でもありうる成果物をフィードバックの辺で戻す。

```pfdsl
spec >> implement -> code
code >> test -> [test_report, bug_report]
bug_report >>? implement
```

承認されたら次へ、も同様に、承認記録という成果物を作り、後続のプロセスはそれを入力として受け取る形に置き換える。
清水のPFDも元々この種の分岐を持たない。

三つ目に採らないのはサブルーチンである。
似たプロセスを共通化して呼び出す仕組みを導入すると、着手可能な集合を求めるために呼び出し先まで展開して読む必要が生まれ、図を静的に読み切れなくなる。
共通性を示したいときは、横断的なラベルで束ねるにとどめる。

## 書くこと自体が構造を暴く

機械検査が走る前の段階でも、図に書くという作業そのものが構造の誤りを表面化させることがある。

複数の仕様案を並行して起草できる、という想定を立て、それぞれ独立したプロセスとして図に描いたとしよう。
実際には両案が同じ上流の方針に依存しているなら、独立した起草という主張自体がレビューで問題になる。
指摘を受けて、上流の方針を先に共有してから合意のもとで起草する、という構成に直すことになる。
ここから、相互に依存する作業は分割せず一つのプロセスにまとめ、共有すべき決定を独立した成果物として外化する、という判断基準が得られる。

階層をもつ図で、親の図と子の図の境界がどう対応するかを定める場面を考える。
実装より前に、境界の対応を具体例でたどる検証を行うと、そのままテストケースとして使える組み合わせが十件以上見つかることがある。
親側にある二つの異なる成果物を、誤って子側の同じ成果物に対応させてしまうケースを考える。
この誤りがどの検査で弾かれるかは、仕様の文章を読むだけでは判断しにくい。
図の上で対応を実際にたどることで、初めて答えが出る。

## 払うコスト

それでも、成果物の粒度を見誤ることはある。

入力の集合が同じプロセスは統合する、という粒度の規則があるとする。
この規則に従って、知見を外部の資料にまとめる作業を、一つの対話プロセスに統合したとしよう。
統合した時点では入力の集合は同じに見え、規則の適用として筋が通っている。

ところが、その作業を丸ごと別の担当者、たとえば経緯を知らないAIに委譲すると、存在しない参照IDを作り出してしまうことがある。
参照すべき図が実は委譲先の入力に含まれていなかったことが、この失敗によって初めて分かる。
統合した先には、隠れた独立のプロセスがあったことになる。

図があらかじめ正しい分割を示していたわけではない。
委譲の失敗によって、入力の不足が明らかになった。
粒度の判断は、実際に運用してみて初めて更新される。

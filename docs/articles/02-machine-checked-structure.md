# 構造の見落としは人が拾えない — PFDSL 連載 第2部

## 痛み — ガイドとレビューを整備してもミスは再発する

図の質を保つために、著者のリポジトリでは品質ガイドを整備してきた。
効果は測れないわけではなかった。
実践ラウンドを重ねて前後を比較すると、ガイドを足したことで消えたミスと、足しても残ったミスをはっきり分けて数えられた。
つまりガイドは効いてはいる。
だが残るミスは残る。
ルールを一本増やしても、それを毎回思い出して照合する作業そのものは人に残るからだ。

放置のされ方も同じ形をしていた。
誰にも消費されない成果物がリポジトリに紛れ込んでも、日常のレビューでは気づかれない。
実際に見つかったのは、ユーザーから「見落としはないか」と問い直されたときで、しかもそれが2件も溜まっていた。
問われるまで誰も気づいていなかった、という点が重要だ。
この問い直しのパターン自体は後で運用の手順に組み込まれ、以後は誰かに問われるのを待たずに一定の間隔で効くようにはなった。
それでも、問い直しの仕組みが働くたびに新たな滞留が見つかるという構図自体は変わっていない。

これはガイドの書き方が悪いという話ではない。
列挙的に確認しなければならない項目、たとえば「この成果物は誰かに読まれているか」「この宣言は実際に使われているか」といった問いは、人の注意力に向いた仕事ではない。
向いていない仕事を向いていない人に割り当て続ける限り、ガイドをいくら書き足しても見落としは形を変えて戻ってくる。
測定できたのは「ルールで消えたミス」と「ルールを増やしても残るミス」の内訳であって、後者の存在そのものは消せなかった。
ここから導かれるのは、ルールの質を上げる努力とは別の軸、つまり照合作業そのものを人の手から外す軸が要る、ということだった。

## PFD 固有の意味論を機械検査する

機械に確認を任せるには、確認できる形で書かれている必要がある。
PFDSLでは出力である成果物だけを有形なものに限定する。
入力にはレビュアーの知識のような不定形なものを許すが、出力はファイルやドキュメントのように保管し検証できるものでなければならない。
「理解した」は成果物にならないので、理解した内容をまとめた資料に外化する。
「応急処置をした」も同様に、何をどう当てたかを書いた暫定対応記録に外化する。
出力が有形だからこそ完了の判定が成立し、statusの管理や引き継ぎが成り立ち、その先で機械検査の対象になれる。
逆に、入力側まで有形さを求めてしまうと、レビュアーの経験や勘のように外化しづらいものまで無理に成果物へ変換する羽目になり、図はかえって現実から遊離する。
入力と出力で非対称な規則にしているのはそのためだ。
理解を資料として外に出すという発想自体は、清水吉男がXDDPで唱えたスペックアウトと同じ方向を向いている。

外化された図に対して、PFDSLの検査は何を見るのか。
テキストで書かれたものなら何でもCIに載せられる、という一般論ではない。
PFDという図法の意味論に固有の壊れ方だけを、狙って検査する。
具体的には次の4点である。

1. 同じ成果物を複数のプロセスが生成していないか（単一生成元）
2. グラフが循環していないか（成果物とプロセスの依存グラフは循環のない有向グラフでなければ、着手順序を導出できない）
3. どのエッジにも参加しない宣言だけのプロセスが残っていないか（孤立プロセス）
4. 生成される成果物に完了基準が書かれているか

実際に検査を走らせると、たとえば宣言だけ残ったプロセスはこう弾かれる。

```
orphan.pfdsl:1:1: error [V020]: Process 'publish' is declared but has no edges (orphaned process)
```

完了基準を書き忘れた成果物はこう警告される。

```
nocrit.pfdsl:3:3: warning [W002]: Artifact 'spec' has no 'criteria' field
```

この二つの層の分業は明確だ。
ガイド、つまり品質ガイドが担うのは設計判断だけである。
成果物をどの粒度で切るか、何と名付けるか、理解や応急処置をどう外化するかは、人が決めるしかない。
列挙的な確認、つまり「漏れなく全部を突き合わせたか」という仕事は、全部機械へ送る。

この分業が実際にどう効いたか、3件挙げる。

1件目。
完了基準の警告を、完了済みの成果物だけでなく作業中や未着手の成果物にも広げる変更を実装した。
その日のうちに、この警告が著者自身の進行管理の図を直撃した。
作業中なのに完了基準が書かれていない項目が見つかり、さらに未完成の仕様に依存しているのに完了扱いになっている項目という矛盾も同時に見つかった。
拡張前の検査は完了済みの成果物しか見ていなかったので、どちらの問題も見えていなかった。

2件目。
文脈ゼロのAIエージェントに実タスクを任せる運用の中で、ある見落とし方が見つかった。
依存のチェーンを削除したあとに、宣言だけのプロセスが図に残り、検査をそのまま素通りしていたのだ。
これはガイドの書き方では防げない型の見落としだったので、機械検査の要件に変換して実装した。
以後、この見落としは機械が検出するようになった。
見落としを見つけ、要件に変換し、機械化するという流れが1周した形になる。

3件目。
文脈を持たない外部のレビュアー、といっても中身はAIだが、そのレビュアーにリポジトリの構造そのものを監査させたことがある。
返ってきた指摘は14件あったが、そのすべてが機械検査の届かない散文の層、たとえば配布手順を説明する文章や、参照が連なっていく箇所に集中していた。
機械化が済んでいる領域からは新規の欠陥が出なかった。
分業の境目が指摘の分布として可視化され、次にどこを機械化すべきかまで特定できた。
機械が守っている範囲と、まだ人の注意力に頼っている範囲の境界線が、レビュー結果という形で初めて目に見えたことになる。

## 検査可能性を守るために入れない機能

ここまでの検査が成り立つのは、図が文脈ゼロの読者、人であれ機械であれ、静的に読み切れる形をしているからだ。
だから機能を足すときは、その読み切りやすさを壊さないかを先に問う。
便利に見えても、壊すなら入れない。
実際に候補に挙がった機能は、どれも単体では書きたくなる自然な要求だった。
それでも採らなかったのは、便利さと引き換えに図を動的にしか読めなくしてしまうからだ。

まず、既存の成果物を書き換えるupdateは持たない。
本家のPFDにはこれがあるが、採らない。
データベースや本番環境、学習済みモデルのような書き換わっていく資源は、ある時点のスナップショットとして書く。
日次のダンプ、リリースされた版、ある時点のモデルの重み、といった具合だ。
「更新したあとの自分」を自分の入力にしてしまうと循環が生まれ、静的に読み切れるという前提が壊れる。

改版そのものは3つの形で書き分ける。
このうち後の2形態では、下流の成果物から上流のプロセスへ戻るフィードバックの辺、`>>?` を使う。
このフィードバックの辺は着手順序の計算には影響しない、あくまで補助的な依存を表す記法だ。

1つ目は別成果物形態である。
版が承認や配布、契約のベースラインになるなら、`spec_v1` と `spec_v2` のように別の成果物として書く。

```pfdsl
spec_v1 >> revise -> spec_v2
```

2つ目は収束ループ形態である。
同じフェーズの中で繰り返し収束していく作業は、フィードバックの辺で書く。

```pfdsl
draft >> write -> manuscript
manuscript >> review -> comment
comment >>? write
```

3つ目は定常サイクル形態である。
モデルの再学習のように、版を数え上げられない無限の繰り返しも同じ`>>?`で書くが、性質はコメントで補う。

```pfdsl
# 終わりのない再学習サイクル。data の版は列挙しない
data >> train -> model
model >> evaluate -> eval_metrics
eval_metrics >>? train
```

この3形態は併用しない。
「この版が意味するものは何か」を問えば、どれを使うべきかは自然に決まる。

次に、条件分岐も持たない。
合格したら次へ、承認されたら次へ、と書きたくなる場面は多いが、その分岐は成果物の定義そのものが誤っているシグナルだと解釈する。
「テストに合格したら次へ」ではなく、テストの結果報告書を常に出力する形にし、不具合票という空でもありうる成果物をフィードバックの辺で戻す。

```pfdsl
spec >> implement -> code
code >> test -> [test_report, bug_report]
bug_report >>? implement
```

「承認されたら次へ」も同様に、承認記録という成果物を作り、後続のプロセスはそれを入力として受け取る形に置き換える。
清水のPFDも元々こうした分岐を持たない図法である。

最後に、サブルーチンも持たない。
似たプロセスをくくり出して共通化し、呼び出す仕組みは導入しない。
呼び出しの仕組みを入れると、図を読むだけで着手可能な集合を導出するには、呼び出し先まで展開して読む手間が発生し、静的な読み切りやすさが後退する。
共通性を示したいときは、横断的なラベルで束ねるところまでに留める。

## 図そのものが検査装置になる

機械検査の手前でも、図に書くという行為そのものが構造を暴くことがある。
実例を3件挙げる。

1件目。
複数の仕様案件を並列で起草できる、という直感が著者にはあった。
だが図に書いた途端、その主張自体がレビューの対象になった。
指摘を受けて、並列に起草する案から、上流の方針だけ先に共有する案を経て、最終的に共同で起草する案へと3段階で修正された。
相互に依存する作業は分割せず1つのプロセスにまとめ、共有すべき決定を独立した成果物として外化する、という判断基準はこの過程で見つかった。

2件目。
図の入れ子、つまり親の図と子の図の境界がどう対応するかという仕様を書いたとき、実装より先に境界の具体例を図の上でトレースする検証ラウンドを回した。
このトレースがそのまま11個のテストケースになった。
中でも、親側にある2つの異なる成果物を子側の同じ成果物に対応させてしまう誤った境界対応がある。
この誤りがどの検査で弾かれるかは、仕様の文章を読むだけでは非自明だった。
図の上で実際に対応をたどってみて、初めて答えが出た。
仕様のテキストだけでは、こうした非自明なケースを事前に網羅できなかった。

3件目。
相互に依存すると分かっていた2つの案件があった。
著者は最初からこれを1つのプロセスが複数の成果物を出す形で図に起こしておいた。
後からこの案件に着手した作業者、これもAIだが、図を読むだけで「分割せず共同で設計し、共有すべき決定を別の成果物として外化する」という正しい構造を受け取れた。
もしこれを独立した2つの項目として並べていたら、相互依存に気づくのは実際に起草がぶつかってからだったはずだ。
図に書いておくという先回りが、後から来る読み手の判断ミスをレビュー前に消していたことになる。

3件に共通するのは、レビューや実装が始まるより前に、図を書くという行為自体が判断の材料を机の上に並べてしまうことだ。
機械検査はその材料が矛盾していないかを後から確かめるが、材料そのものを並べるのは人が図を書く手だ。

## 払うコスト

それでも粒度の読み違えは残る。

著者は粒度についてのルール、入力の集合が同一のプロセスは統合する、というルールに従っていた。
その規則どおりに、知見を外に出すという作業を1つの対話プロセスへ統合したことがある。
統合した時点では入力の集合は確かに同一に見えており、ルールの適用としては筋が通っていた。
後日その作業を一括してAIに委譲したところ、存在しない参照IDを捏造するという失敗が起きた。
委譲した結果を突き合わせて初めて、参照先が実在しないという食い違いに気づいた形になる。
この失敗を見て初めて、「参照先の図」という入力が実は欠けていたこと、つまり統合した先に隠れた独立プロセスがあったことが分かった。
結局、この作業は分割し直すことになった。

図が事前に正解を出したわけではない。
委譲の失敗が、入力の不足を露出させた。
粒度の判断は、観測されて初めて更新される。

# 今、何に着手できるか — PFDSL 連載 第1部

プロジェクトの進め方 — 誰が何を作り、何に依存して完了するか — が暗黙知のままだと、人もAIも毎回文脈の口移しでしか参加できない。
プロセスを成果物依存グラフとしてテキストに外化し機械検査に載せれば、文脈ゼロの参加者が図を読むだけで作業を選び、回し、監査できる。
PFDSL はそのための言語と運用である。

連載は4部構成で、各部が1つの痛みを扱う。
第1部「今、何に着手できるか」。
第2部「図は腐る」。
第3部「構造の見落としは人が拾えない」。
第4部「AI・新規参加者への文脈の口移し」。

## 痛み

書き捨てで終わらないプロジェクトには、ほぼ必ず依存関係が生まれる。
ある資料は別の資料の合意を前提にしている。
ある実装はある仕様が固まらないと始められない。
これは開発だけの話ではない。
企画も、デザインも、法務のレビューも、資料作成も、成果物同士が互いを前提にし合う点では同じ構造を持つ。
たとえばイベントの告知文は会場の確保が確定してからでないと日程を書けないし、契約書のひな形は先方との条件合意が済むまで最終版にできない。
どの業種でも、ある成果物が別の成果物の完了を待っているという関係自体は普遍的に存在する。

問題は、その依存関係の把握がたいてい個人の頭の中にしかないことだ。
担当者が変わると、まず「今どこまで進んでいて、次に何ができるか」を聞き直すところから始まる。
本人ですら、しばらく手を離していた案件に戻ると同じ問いを自分に向け直す羽目になる。
着手判断も、優先順位づけも、新しい提案が既存の作業とどう衝突するかの見積もりも、頭の中で毎回ゼロから組み立て直す作業になる。
一度考えて出した結論は、次に同じ問いが来たときには残っていない。
考えた過程そのものが記録されていないからだ。

たとえば、複数の案件を並行して抱えているとき、「今どれが空いていてどれが待ちなのか」を答えるには、それぞれの前提がどこまで満たされているかを一件ずつ思い出す必要がある。
思い出せなければ、担当者に聞いて回るか、資料を読み返すしかない。
新しい提案が来たときも同じだ。
それが今動いている作業と競合するのか、並行して進められるのか、判断するには既存の依存関係を頭の中に展開し直さなければならない。
その展開作業は毎回一からやることになり、しかも展開した結果は次には残らない。

チームが大きくなるほど、この再構築コストは全員分に積み上がる。
誰か一人が「今は何が空いているか」を知っていても、それは共有されない限り本人の頭の外には存在しない。

## 成果物で考える

この痛みへの対処は、作業（プロセス）ではなく成果物を主語にして進行を書くことから始まる。
「設計をしている」ではなく「仕様書ができた」を単位にすると、何を生んで完了したかが検証可能になる。
これは清水吉男が派生開発手法XDDPのために考案したPFD（Process Flow Diagram）の核であり、PFDは成果物とそれを生成するプロセスの依存関係を明示的に書く図法である。

PFDSLはこの図法をテキストで書くための最小限の記法を持つ。

- `A >> P -> B` — A はプロセス P の入力、B は P の出力
- `[a, b] >> P` — 集合記法
- frontmatter で成果物に status（done / wip / todo など）を付けられる

要件と制約から設計を経て仕様書ができ、それをもとに実装してコードができ、コードをレビューしてレビュー結果ができる、という流れは次のように書ける。

```pfdsl
[requirement, constraint] >> design -> spec
[spec, codebase] >> implement -> code
code >> review -> review_report
```

矢印の向きが依存の向きそのものであり、図を上から読むだけで誰が何を待っているかが分かる。
この図には作業の進め方や担当者は一切書かれていない。
書かれているのは成果物と、それを生む依存関係だけだ。
それで十分なのは、次に何ができるかを決めるのに必要なのが「誰が今忙しいか」ではなく「何が揃っているか」だからである。

## 着手可能集合の機械導出

成果物にstatusを付けると、この図から次に着手できる作業が導出できるようになる。
規則は単純で、入力となる成果物が全てdoneのプロセスが、今着手できる作業である。

先ほどの図に進捗を足してみる。

```pfdsl
---
artifact:
  requirement:
    status: done
  constraint:
    status: done
  spec:
    status: todo
  codebase:
    status: done
  code:
    status: todo
---
[requirement, constraint] >> design -> spec
[spec, codebase] >> implement -> code
code >> review -> review_report
```

designの入力はrequirementとconstraintで、どちらもdoneなのでdesignには今着手できる。
implementの入力はspecとcodebaseだが、specがまだtodoなのでimplementはまだ着手できない。
reviewの入力はcodeで、これもtodoのままなので同様にまだ着手できない。

この列挙は機械的な規則の適用でしかない。
人が目で追っても、スクリプトが走査しても、文脈を持たないAIエージェントが読んでも、同じ結論に到達する。
判断の材料が個人の記憶ではなく図とstatusという外部データになっているからだ。
誰がやっても同じ答えになるということは、答えを出すために背景の共有が要らないということでもある。
図とstatusさえ渡せば、その場に居合わせていなかった人にも同じ着手可能集合が見える。

以降の実例はすべて、著者が自分のリポジトリを数週間運用した自己観察で、判定は著者による。
効果測定ではなく「この条件でこう効いた」の記録である。

ある日、オープンなissueが8件たまっていて、優先順位を付ける必要が生じた。
issueとその依存関係をPFDとして書き起こしたところ、入力が全てdoneのプロセスの列挙、つまり並列で着手できる作業のリストがstatusから機械的に導出できた。
9本の作業を並列で着手できると即答できた。
issueを一つずつ読み比べて依存を頭の中で組み立て直す必要はなかった。

別の日には、仕様に新しいメタデータを追加する提案が上がり、それを受け入れるかどうかの判断が必要になった。
既存の依存グラフに1本のチェーンを追加しただけで、いくつかのことが確定した。
その提案は仕様の他の起草作業と並列に進められること。
既存方針との接点はパス解決の規則1点だけであること。
最終的な合流点は仕様を1つにまとめる統合プロセスであること。
これらは文章のロードマップだけを頼りにしていたら、提案が来るたびに頭の中でやり直していたはずの判断だった。
図に1本線を足すだけでその判断が済んだのは、既存の依存関係がすでにテキストとして存在していたからにほかならない。

## 払うコスト

ここまでの効きには前提となるコストがある。
前倒しのモデリングだ。
フラットなToDoリストであれば、思いついた順に項目を並べて後から調整すればよい。
成果物依存グラフはそうはいかない。
最終成果物から遡って、何が何に依存するかを最初にある程度書き切る必要がある。
その手間は、ToDoリストを書く手間よりも確実に大きい。
書き切る前の段階では、項目を1つ追加するたびに他の項目との依存を確認し直すことになり、着手までの時間がその分だけ延びる。
この初期コストを払わずに済ませる方法は今のところない。
